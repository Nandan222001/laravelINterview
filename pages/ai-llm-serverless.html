Event Processing & Integrations (1-30)

Question 1
Explain how AWS Lambda processes events from different event sources (S3, DynamoDB, SQS, API Gateway).
Lambda processes events through different integration patterns:
  • S3: Push-based, async invocation when objects are created/deleted
  • DynamoDB Streams: Poll-based, Lambda polls the stream for changes
  • SQS: Poll-based, Lambda polls queue and processes in batches
  • API Gateway: Synchronous invocation, waits for response
Each source has different retry behaviors and error handling mechanisms.

What is the Lambda event object structure for API Gateway proxy integration?
The event object contains: httpMethod, path, pathParameters, queryStringParameters, headers, body (JSON string), requestContext (with request ID, account, and authorizer data), and isBase64Encoded flag.
Question 3
Write a Lambda function that processes S3 object creation events and generates thumbnails.
Python - S3 Thumbnail Generator lambda_handler.py 
import json
import boto3
import os
from PIL import Image
from io import BytesIO

s3_client = boto3.client('s3')

def lambda_handler(event, context):
    for record in event['Records']:
        bucket = record['s3']['bucket']['name']
        key = record['s3']['object']['key']
        
        if 'thumbnails/' in key:
            continue
        
        try:
            # Download image from S3
            response = s3_client.get_object(Bucket=bucket, Key=key)
            image_data = response['Body'].read()
            
            # Generate thumbnail
            img = Image.open(BytesIO(image_data))
            img.thumbnail((200, 200))
            
            # Save thumbnail to buffer
            buffer = BytesIO()
            img.save(buffer, format=img.format)
            buffer.seek(0)
            
            # Upload thumbnail to S3
            thumbnail_key = f"thumbnails/{key}"
            s3_client.put_object(
                Bucket=bucket,
                Key=thumbnail_key,
                Body=buffer,
                ContentType=response['ContentType']
            )
            
            print(f"Generated thumbnail: {thumbnail_key}")
            
        except Exception as e:
            print(f"Error processing {key}: {str(e)}")
            raise
    
    return {
        'statusCode': 200,
        'body': json.dumps('Thumbnails generated successfully')
    }

Cold Start Optimization (51-80)

Questions 51-80
Cold Start Mitigation Strategies and Performance Optimization

Cold Start Times by Runtime:

  • Node.js: 150-300ms (fastest for scripting)
  • Python: 200-400ms (good balance)
  • Go: 100-200ms (compiled, fast startup)
  • Java: 1-3 seconds (JVM initialization overhead)
  • .NET: 500-800ms (improved with .NET 6+)

Optimization Techniques:

  • Provisioned Concurrency: Pre-warmed instances, eliminates cold starts. Cost: ~$15/month per instance. Use for latency-sensitive APIs
  • Package Size Reduction: Use webpack/esbuild for tree-shaking. Target <10MB zipped. Remove dev dependencies
  • ARM64 Graviton2: 20% better price-performance, 15-20% faster cold starts vs x86
  • VPC Configuration: VPC adds 1-10s for ENI attachment. Use Hyperplane ENI for <100ms. Avoid VPC if not needed
  • Lambda SnapStart (Java): 10x faster cold starts using CRaC snapshots. Reduces Java cold start from 3s to 300ms
  • Lazy Loading: Import heavy modules inside handler functions, not at global scope
  • Memory Allocation: More memory = more CPU. Test 512MB-3GB range. Sweet spot often 1024-2048MB
  • Connection Pooling: Create DB connections globally (outside handler). Reused across warm invocations
  • Initialization Code: Place expensive operations (DB connections, config loading) in global scope
  • Dependency Optimization: Use lightweight alternatives (axios→node-fetch, moment→dayjs)
  • Container Reuse: Lambda reuses containers for ~5-15 minutes of inactivity
  • Warm-up Strategies: CloudWatch Events to ping function every 5 minutes (costs ~$0.20/month)

        'statusCode': 200,
        'body': json.dumps({'message': 'Received'})

    \"\"\"Verify HMAC SHA256 signature\"\"\"

        body.encode('utf-8'),
        hashlib.sha256

Attention Mechanism (151-190)

Question 151
Explain the core concept of attention mechanism in neural networks.

Attention allows models to focus on relevant parts of input when processing each element. Instead of compressing all information into a fixed-size vector (bottleneck in RNNs), attention computes weighted sums where weights indicate relevance.

Key Benefits:

  • Variable Focus: Different parts of input get different importance
  • Long-Range Dependencies: Can attend to distant tokens directly (no sequential processing)
  • Parallelization: All attention scores computed simultaneously
  • Interpretability: Attention weights show what model focuses on
Question 152
What is the mathematical formula for scaled dot-product attention?
Attention(Q, K, V) = softmax(QK^T / √d_k) V Where: - Q (Query): What we're looking for [n × d_k] - K (Key): What's available to match [m × d_k] - V (Value): Actual information [m × d_v] - d_k: Dimension of keys (scaling factor) - √d_k: Prevents dot products from growing too large Steps: 1. Compute similarity: QK^T (matrix multiply) 2. Scale: divide by √d_k 3. Normalize: softmax (sum to 1) 4. Weighted sum: multiply by V

Why Scale by √d_k? Large dot products push softmax into regions with small gradients (vanishing gradient). Scaling keeps dot products in reasonable range.

Question 153
Write Python code to implement basic attention mechanism from scratch using NumPy.
Python - Attention from Scratch attention.py 
import numpy as np

def scaled_dot_product_attention(Q, K, V, mask=None):
    \"\"\"
    Scaled Dot-Product Attention
    
    Args:
        Q: Queries (batch, seq_len_q, d_k)
        K: Keys (batch, seq_len_k, d_k)
        V: Values (batch, seq_len_v, d_v)
        mask: Optional mask (batch, seq_len_q, seq_len_k)
    \"\"\"
    d_k = Q.shape[-1]
    
    # Step 1: Compute attention scores (Q * K^T)
    scores = np.matmul(Q, K.transpose(0, 2, 1))
    
    # Step 2: Scale by sqrt(d_k)
    scores = scores / np.sqrt(d_k)
    
    # Step 3: Apply mask if provided (for padding or causal)
    if mask is not None:
        scores = np.where(mask == 0, -1e9, scores)
    
    # Step 4: Apply softmax
    # Subtract max for numerical stability
    scores_max = np.max(scores, axis=-1, keepdims=True)
    exp_scores = np.exp(scores - scores_max)
    attention_weights = exp_scores / np.sum(exp_scores, axis=-1, keepdims=True)
    
    # Step 5: Weighted sum of values
    output = np.matmul(attention_weights, V)
    
    return output, attention_weights

# Example usage
batch_size, seq_len, d_k = 2, 4, 8
Q = np.random.randn(batch_size, seq_len, d_k)
K = np.random.randn(batch_size, seq_len, d_k)
V = np.random.randn(batch_size, seq_len, d_k)

output, weights = scaled_dot_product_attention(Q, K, V)
print(f\"Output shape: {output.shape}\")  # (2, 4, 8)
print(f\"Attention weights shape: {weights.shape}\")  # (2, 4, 4)
print(f\"Weights sum (should be 1): {weights[0,0].sum()}\")  # ~1.0

Multi-Head Attention (191-220)

Question 169
Write code to implement multi-head attention from scratch.
PyTorch - Multi-Head Attention multihead_attention.py 
import torch
import torch.nn as nn
import math

class MultiHeadAttention(nn.Module):
    def __init__(self, d_model=512, num_heads=8, dropout=0.1):
        super().__init__()
        
        assert d_model % num_heads == 0, \"d_model must be divisible by num_heads\"
        
        self.d_model = d_model
        self.num_heads = num_heads
        self.d_k = d_model // num_heads  # Dimension per head
        
        # Linear projections for Q, K, V
        self.W_q = nn.Linear(d_model, d_model)
        self.W_k = nn.Linear(d_model, d_model)
        self.W_v = nn.Linear(d_model, d_model)
        
        # Output projection
        self.W_o = nn.Linear(d_model, d_model)
        
        self.dropout = nn.Dropout(dropout)
        
    def split_heads(self, x, batch_size):
        \"\"\"Split into multiple heads\"\"\"
        x = x.view(batch_size, -1, self.num_heads, self.d_k)
        return x.transpose(1, 2)  # (batch, heads, seq_len, d_k)
    
    def forward(self, query, key, value, mask=None):
        batch_size = query.size(0)
        
        # 1. Linear projections
        Q = self.W_q(query)  # (batch, seq_len, d_model)
        K = self.W_k(key)
        V = self.W_v(value)
        
        # 2. Split into multiple heads
        Q = self.split_heads(Q, batch_size)  # (batch, heads, seq_len, d_k)
        K = self.split_heads(K, batch_size)
        V = self.split_heads(V, batch_size)
        
        # 3. Scaled dot-product attention
        scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k)
        
        if mask is not None:
            scores = scores.masked_fill(mask == 0, -1e9)
        
        attn_weights = torch.softmax(scores, dim=-1)
        attn_weights = self.dropout(attn_weights)
        
        attn_output = torch.matmul(attn_weights, V)
        
        # 4. Concatenate heads
        attn_output = attn_output.transpose(1, 2).contiguous()
        attn_output = attn_output.view(batch_size, -1, self.d_model)
        
        # 5. Final linear projection
        output = self.W_o(attn_output)
        
        return output, attn_weights

# Example usage
mha = MultiHeadAttention(d_model=512, num_heads=8)
x = torch.randn(32, 10, 512)  # (batch=32, seq_len=10, d_model=512)
output, weights = mha(x, x, x)
print(f\"Output: {output.shape}\")  # torch.Size([32, 10, 512])

Positional Encoding (221-250)

Questions 221-250
Positional Encoding Strategies

Why Needed: Transformers process all tokens in parallel (unlike RNNs). Without positional information, "cat sits on mat" = "mat on sits cat".

Python - Sinusoidal Positional Encoding positional_encoding.py 
import torch
import torch.nn as nn
import math

class PositionalEncoding(nn.Module):
    def __init__(self, d_model, max_len=5000, dropout=0.1):
        super().__init__()
        self.dropout = nn.Dropout(p=dropout)
        
        # Create positional encoding matrix
        pe = torch.zeros(max_len, d_model)
        position = torch.arange(0, max_len).unsqueeze(1).float()
        div_term = torch.exp(torch.arange(0, d_model, 2).float() * 
                            -(math.log(10000.0) / d_model))
        
        # Apply sin to even indices
        pe[:, 0::2] = torch.sin(position * div_term)
        # Apply cos to odd indices  
        pe[:, 1::2] = torch.cos(position * div_term)
        
        pe = pe.unsqueeze(0)  # Add batch dimension
        self.register_buffer('pe', pe)
        
    def forward(self, x):
        x = x + self.pe[:, :x.size(1)]
        return self.dropout(x)

Encoding Types:

  • Sinusoidal (Original Transformer): PE(pos,2i)=sin(pos/10000^(2i/d)), PE(pos,2i+1)=cos(...). No training needed, generalizes to longer sequences
  • Learned Embeddings (BERT): nn.Embedding(max_len, d_model). Learned during training, fixed max length
  • Relative (T5): Encodes relative distances between tokens, not absolute positions
  • ALiBi: Adds bias to attention scores based on distance. Linear penalty for distance
  • RoPE (LLaMA): Rotary embeddings applied to Q and K. Excellent length extrapolation

Feed-Forward Networks & Layer Norm (251-280)

Questions 251-280
Position-wise Feed-Forward Networks and Normalization

Feed-Forward Network (FFN):

FFN(x) = max(0, xW₁ + b₁)W₂ + b₂ Architecture: Input (d_model=512) ↓ Linear (512 → 2048) [W₁, b₁] ↓ Activation (ReLU/GELU/SwiGLU) ↓ Dropout (0.1) ↓ Linear (2048 → 512) [W₂, b₂] ↓ Output (d_model=512) Expansion ratio: typically 4x GPT: 4x (512 → 2048) BERT: 4x (768 → 3072) LLaMA: ~2.7x with SwiGLU

Activation Functions:

  • ReLU: max(0, x). Simple, fast. Used in original Transformer
  • GELU: x·Φ(x) where Φ is CDF of normal distribution. Smooth, better gradients. Used in BERT, GPT
  • SwiGLU: Swish(xW)⊗xV. Gated activation. Used in LLaMA, PaLM. Best performance but more parameters

Layer Normalization:

LayerNorm(x) = γ · (x - μ) / σ + β Where: μ = mean across d_model dimension σ = std across d_model dimension γ, β = learned affine parameters Pre-LN (Modern): Post-LN (Original): x → LN → Attn → + x → Attn → + → LN x → LN → FFN → + x → FFN → + → LN Pre-LN: Better training stability, no warm-up needed Post-LN: Slightly better final performance, needs warm-up

Attention Mechanism (151-190)

Question 151
Explain the core concept of attention mechanism in neural networks.
Attention allows the model to focus on relevant parts of the input when processing each element. Instead of compressing all information into a fixed-size vector, attention computes weighted sums of input representations, where weights indicate relevance. This solves the bottleneck problem in seq2seq models.
Question 152
What is the mathematical formula for scaled dot-product attention?
Attention(Q, K, V) = softmax(QK^T / √d_k)V
Where Q=queries, K=keys, V=values, d_k=dimension of keys.
The scaling by √d_k prevents dot products from growing too large, which would push softmax into regions with small gradients.
Question 153
Write Python code to implement basic attention mechanism from scratch using NumPy.
Python - Attention from Scratch attention.py 
import numpy as np

def scaled_dot_product_attention(Q, K, V, mask=None):
    """
    Scaled Dot-Product Attention
    
    Args:
        Q: Queries (batch, seq_len, d_k)
        K: Keys (batch, seq_len, d_k)
        V: Values (batch, seq_len, d_v)
        mask: Optional mask
    """
    d_k = Q.shape[-1]
    
    # Compute attention scores: Q * K^T / sqrt(d_k)
    scores = np.matmul(Q, K.transpose(0, 2, 1)) / np.sqrt(d_k)
    
    # Apply mask if provided
    if mask is not None:
        scores = np.where(mask == 0, -1e9, scores)
    
    # Apply softmax
    attention_weights = np.exp(scores - np.max(scores, axis=-1, keepdims=True))
    attention_weights = attention_weights / np.sum(attention_weights, axis=-1, keepdims=True)
    
    # Weighted sum of values
    output = np.matmul(attention_weights, V)
    
    return output, attention_weights

K = np.random.randn(batch_size, seq_len, d_k)

print(f"Attention weights shape: {weights.shape}")
Question 169
Write code to implement multi-head attention from scratch.
Python - Multi-Head Attention multihead_attention.py 
import torch
import torch.nn as nn
import math

class MultiHeadAttention(nn.Module):
    def __init__(self, d_model=512, num_heads=8, dropout=0.1):
        super().__init__()
        
        assert d_model % num_heads == 0
        
        self.d_model = d_model
        self.num_heads = num_heads
        self.d_k = d_model // num_heads
        
        # Linear projections
        self.W_q = nn.Linear(d_model, d_model)
        self.W_k = nn.Linear(d_model, d_model)
        self.W_v = nn.Linear(d_model, d_model)
        self.W_o = nn.Linear(d_model, d_model)
        
        self.dropout = nn.Dropout(dropout)
        
    def split_heads(self, x, batch_size):
        x = x.view(batch_size, -1, self.num_heads, self.d_k)
        return x.transpose(1, 2)
    
    def forward(self, query, key, value, mask=None):
        batch_size = query.size(0)
        
        # Linear projections
        Q = self.W_q(query)
        K = self.W_k(key)
        V = self.W_v(value)
        
        # Split into multiple heads
        Q = self.split_heads(Q, batch_size)
        K = self.split_heads(K, batch_size)
        V = self.split_heads(V, batch_size)
        
        # Scaled dot-product attention
        scores = torch.matmul(Q, K.transpose(-2, -1)) / math.sqrt(self.d_k)
        
        if mask is not None:
            scores = scores.masked_fill(mask == 0, -1e9)
        
        attn_weights = torch.softmax(scores, dim=-1)
        attn_weights = self.dropout(attn_weights)
        
        attn_output = torch.matmul(attn_weights, V)
        
        # Concatenate heads
        attn_output = attn_output.transpose(1, 2).contiguous()
        attn_output = attn_output.view(batch_size, -1, self.d_model)
        
        # Final linear projection
        output = self.W_o(attn_output)
        
        return output, attn_weights

Write code to implement a RAG system with document chunking.
Python - RAG with Smart Chunking rag_chunking.py 
import openai
import pinecone
from typing import List, Dict

class RAGPipeline:
    def __init__(self, index_name: str):
        self.index_name = index_name
        self.chunk_size = 1000
        self.chunk_overlap = 200
        
        # Initialize Pinecone
        if index_name not in pinecone.list_indexes():
            pinecone.create_index(
                name=index_name,
                dimension=1536,
                metric='cosine'
            )
        
        self.index = pinecone.Index(index_name)
    
    def chunk_text(self, text: str) -> List[str]:
        """Split text into overlapping chunks"""
        chunks = []
        start = 0
        
        while start < len(text):
            end = start + self.chunk_size
            chunk = text[start:end]
            
            # Try to break at sentence boundary
            if end < len(text):
                last_period = chunk.rfind('.')
                if last_period > self.chunk_size * 0.5:
                    chunk = chunk[:last_period + 1]
                    end = start + last_period + 1
            
            chunks.append(chunk)
            start = end - self.chunk_overlap
        
        return chunks
    
    def ingest_documents(self, documents: List[Dict]):
        """Ingest documents with chunking and embedding"""
        all_chunks = []
        
        for doc in documents:
            chunks = self.chunk_text(doc['text'])
            
            for i, chunk in enumerate(chunks):
                all_chunks.append({
                    'id': f"{doc['id']}_chunk_{i}",
                    'text': chunk,
                    'metadata': {
                        'source': doc['id'],
                        'chunk_index': i,
                        **doc.get('metadata', {})
                    }
                })
        
        # Generate embeddings in batches
        self._embed_and_upsert(all_chunks)
    
    def _embed_and_upsert(self, chunks: List[Dict], batch_size: int = 100):
        for i in range(0, len(chunks), batch_size):
            batch = chunks[i:i + batch_size]
            texts = [c['text'] for c in batch]
            
            # Generate embeddings
            response = openai.Embedding.create(
                input=texts,
                model="text-embedding-ada-002"
            )
            
            # Prepare vectors
            vectors = []
            for chunk, emb_data in zip(batch, response['data']):
                vectors.append({
                    'id': chunk['id'],
                    'values': emb_data['embedding'],
                    'metadata': {
                        'text': chunk['text'],
                        **chunk['metadata']
                    }
                })
            
            self.index.upsert(vectors=vectors)
    
    def query(self, question: str, top_k: int = 5) -> str:
        """Query with RAG"""
        # Generate query embedding
        query_emb = openai.Embedding.create(
            input=[question],
            model="text-embedding-ada-002"
        )['data'][0]['embedding']
        
        # Search
        results = self.index.query(
            vector=query_emb,
            top_k=top_k,
            include_metadata=True
        )
        
        # Build context
        context = "\n\n".join([
            match['metadata']['text'] 
            for match in results['matches']
        ])
        
        # Generate answer
        response = openai.ChatCompletion.create(
            model="gpt-4",
            messages=[
                {"role": "system", "content": "Answer based on context."},
                {"role": "user", "content": f"Context:\n{context}\n\nQuestion: {question}"}
            ]
        )
        
        return response['choices'][0]['message']['content']

Domain-specific vs general: general good baseline, domain fine-tuned better for niche. Fine-tuning: contrastive learning on domain pairs. Batch implementation: async requests, rate limiting. Cosine similarity: dot(a,b)/(norm(a)×norm(b)), range [-1,1], 1=identical. Normalization: L2 normalize before storage for faster dot product. Symmetric vs asymmetric: symmetric for doc-doc, asymmetric for query-doc. Multi-lingual: multilingual-e5, LaBSE, mUSE. Semantic deduplication: cluster by similarity >0.95. Quality evaluation: MTEB benchmarks (retrieval, classification, clustering). Embedding drift: retrain periodically, A/B test. Model versioning: track model version with vectors, migrate carefully. Caching: Redis/DynamoDB with embedding hash as key. Incremental updates: only embed new/changed docs. Chunk overlap impact: 10-20% overlap improves boundary cases. Metadata role: filter before vector search for precision. Dense Passage Retrieval (DPR): dual-encoder for query & passage. Query expansion: generate multiple phrasings, embed all. Embedding mismatch: query reformulation, hybrid search. Hard negative mining: train on challenging negatives. Latency: OpenAI ~100-500ms, local ~10-50ms. Document clustering: k-means on embeddings. Dimensionality reduction: PCA/UMAP to 128-256 dims (slight quality loss). Storage: separate table with doc_id foreign key or JSONB in same table. Emerging models: OpenAI v3 (customizable dimensions), Voyage AI, BGE-large.

Banking Document Search (586-600): Design for regulatory docs, loan policies, compliance. Ingestion pipeline: PDF/Word→parse (PyPDF2, python-docx, Unstructured)→chunk→embed→index. PDF/Word/Excel handling: Unstructured library, LlamaParse for complex layouts. Document parsing: extract text, preserve structure (headers, tables). Tables & structured data: extract separately, create structured embeddings or use specialized models. Preprocessing: remove headers/footers, normalize whitespace, handle OCR errors. Metadata: document_type, date, version, department, compliance_category. Source attribution: store chunk_id→document mapping, return with answers. Questions 641-670 - Fine-tuning Strategies
Fine-tuning vs Prompt Engineering vs RAG

Fine-tuning: Train model on custom data. Use cases: consistent behavior, style matching, domain terminology. Methods: Full fine-tuning (update all weights, expensive), LoRA (Low-Rank Adaptation, update small matrices, 0.1% parameters), QLoRA (4-bit quantized LoRA, fits on single GPU), Prefix-tuning, Adapter layers.

LoRA: Rank r=8-64, alpha=16-32. Formula: W' = W + BA where B,A are low-rank matrices. Benefits: 100x less memory, fast iteration, merge/unmerge easily. QLoRA: 4-bit quantization + LoRA, train 65B model on 48GB GPU.

When to use: Fine-tuning: <1000 examples, consistent style needed, lower latency required. Prompt Engineering: <100 examples, rapid iteration, flexibility. RAG: need current info, source attribution, no training data.

Best practices: Start with prompting, add RAG if need knowledge, fine-tune if need consistency. Dataset quality > quantity. 100-1000 high-quality examples sufficient for LoRA. Validation set essential. Learning rate: 1e-4 to 5e-5. Epochs: 3-5 typically. Monitor for overfitting.

Questions 701-730 - LangChain Framework
LangChain for LLM Application Development

Core Components: Models (LLMs, Chat Models, Embeddings), Prompts (PromptTemplate, FewShotPromptTemplate), Chains (LLMChain, SequentialChain, RouterChain), Agents (ReAct, self-ask, plan-and-execute), Memory (ConversationBufferMemory, ConversationSummaryMemory), Callbacks (logging, monitoring), Document Loaders (PDF, CSV, Web, APIs), Text Splitters (RecursiveCharacterTextSplitter, TokenTextSplitter), Retrievers (VectorStoreRetriever), Output Parsers (JSON, XML, PydanticOutputParser).

LCEL (LangChain Expression Language): Composable chains using | operator. `chain = prompt | model | output_parser`. Benefits: streaming, async, batching, retries built-in.

Agents: LLM decides which tools to use. Tools: Search (SerpAPI, DuckDuckGo), Calculators, APIs, Python REPL. Agent types: zero-shot-react (decides based on descriptions), conversational-react (maintains chat history), plan-and-execute (plans then executes).

Memory: ConversationBufferMemory (stores all messages), ConversationSummaryMemory (summarizes old messages), ConversationBufferWindowMemory (keeps last k messages). Integration with chains via memory parameter.

Production tips: Use LangSmith for debugging, caching to reduce costs, async for concurrency, custom callbacks for monitoring, error handling with retries, token counting before API calls, streaming for better UX.

Questions 731-750 - OpenAI API
OpenAI API Integration

Models: GPT-4 (8K/32K/128K context, $0.03/1K input, $0.06/1K output), GPT-4 Turbo (128K, $0.01/$0.03), GPT-4o (128K, $0.005/$0.015, faster), GPT-3.5 Turbo (16K, $0.0005/$0.0015, cheapest).

Parameters: temperature (0-2, 0=deterministic, 1=balanced, 2=creative), top_p (nucleus sampling, 0.1=conservative, 1.0=all tokens), frequency_penalty (-2 to 2, reduce repetition), presence_penalty (-2 to 2, encourage new topics), max_tokens (limit output), n (number of completions), stop (stop sequences).

Function calling: Define functions in JSON schema, model decides when to call, parse arguments from response, execute function, send result back to model. Use for tool use, API calls, structured data extraction.

JSON mode: `response_format={"type": "json_object"}`. Guarantees valid JSON output. Useful for structured extraction.

Rate limiting: RPM (requests per minute), TPM (tokens per minute), RPD (requests per day). Tier-based: Free (3 RPM), Tier 1 (500 RPM), Tier 2 (5000 RPM). Handle 429 errors with exponential backoff.

Streaming: `stream=True`. Receive tokens as generated. Better UX for long responses. Parse SSE (Server-Sent Events).

Cost tracking: Log prompt_tokens + completion_tokens per request. Calculate cost = (input_tokens×input_price + output_tokens×output_price) / 1000. Monitor with callbacks.

Vision (GPT-4V, GPT-4o): Pass images as URLs or base64. Max resolution: 2048×2048. Pricing: ~$0.01 per image. Use for OCR, image analysis, visual Q&A.

Questions 751-850 - Token Management & Advanced LLM Topics
Claude API, Token Limits, Embeddings, Semantic Search, LLM System Design

Claude API: Models: Opus (best quality, 200K context, $15/$75 per 1M tokens), Sonnet (balanced, 200K, $3/$15), Haiku (fast, 200K, $0.25/$1.25). Features: longer context (200K vs GPT-4's 128K), extended thinking, constitutional AI (safer outputs), vision (PDFs, images), tool use. System prompts: placed outside conversation history, stronger influence.

Token Management: Context windows: GPT-4 (8K/128K), Claude (200K), Gemini 1.5 (1M tokens). Token counting: tiktoken library for OpenAI, anthropic tokenizer for Claude. Strategies: sliding window (keep recent + relevant), summarization (compress old messages), chunk documents (fit in context), prioritize by relevance. Cost optimization: shorter prompts, caching (Claude prompt caching saves 90%), smaller models where sufficient.

System Design: Components: API Gateway (rate limiting, auth), Cache (Redis for responses), Queue (SQS for async processing), Vector DB (Pinecone for RAG), Monitoring (CloudWatch, LangSmith). Patterns: Circuit breaker (fail fast when API down), fallback (GPT-4→GPT-3.5→error message), retry with backoff, request deduplication, cost budgeting per user. Scaling: horizontal scaling of workers, load balancing across providers, caching at multiple levels (response cache, embedding cache, context cache).

Security: Prompt injection prevention (input validation, separate user input from system prompts), PII detection & redaction (regex + NER models), content moderation (OpenAI Moderation API), rate limiting (per user/IP), audit logging (who, what, when, cost), encryption (TLS for transit, KMS for rest).

Performance: Model quantization (4-bit/8-bit, 2-4x faster, slight quality loss), batching (process multiple requests together), KV cache (cache attention keys/values for generation), speculative decoding (small model predicts, large model verifies), prompt compression (reduce tokens while preserving meaning), streaming (show tokens as generated).

Questions 941-1000 - Advanced RAG & Future Trends
Advanced RAG Techniques, Multimodal LLMs, Production Best Practices

Advanced RAG: HyDE (Hypothetical Document Embeddings: generate hypothetical answer, embed it, retrieve similar docs), Self-RAG (model decides when to retrieve), Corrective RAG (critique retrieved docs, re-retrieve if needed), Query expansion (rephrase query multiple ways), Re-ranking (two-stage: fast retrieval + slow rerank with cross-encoder), Parent-document retrieval (retrieve small chunks, return larger parent context), Agentic RAG (agent decides retrieval strategy), Multi-hop (iterative retrieval across documents), Fusion retrieval (combine multiple retrieval methods with learned weights).

Evaluation: Retrieval metrics: precision@k, recall@k, MRR (mean reciprocal rank), NDCG (normalized discounted cumulative gain). Generation metrics: faithfulness (answer grounded in context), answer relevance (addresses question), context relevance (retrieved docs relevant). Multimodal LLMs: GPT-4V (vision + text), Gemini (text + image + audio + video), Claude 3 (text + images + PDFs). Use cases: document analysis (scanned forms, invoices), chart/graph understanding, image-based Q&A, visual search. Best practices: high-quality images (2048×2048 max), clear prompts, provide context, handle multimodal tokens in cost calculation.

Production Best Practices: Monitoring: latency (p50/p95/p99), cost per request, error rates, user satisfaction. A/B testing: model versions, prompt variations, retrieval params. Versioning: prompt templates, models, embedding models. Gradual rollout: canary deployments (5%→25%→100%), feature flags. Cost control: budgets per user, circuit breakers, cheaper models for simple queries. Data flywheel: log queries, collect feedback, fine-tune, improve retrieval. Future Trends: Longer contexts (10M tokens+), cheaper inference ($/token decreasing 10x yearly), multimodal models (unified text/image/audio/video), on-device LLMs (Llama 3.2 1B/3B for mobile), agentic workflows (complex multi-step reasoning), RAG+fine-tuning hybrid, constitutional AI (safer, more aligned), specialized models (code, math, scientific reasoning).
