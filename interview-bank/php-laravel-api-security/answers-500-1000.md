# PHP, Laravel & Advanced API Security - Answers (501-1000)

## Section 11: API Development & REST (Questions 501-600)

501. REST (Representational State Transfer): architectural style for building APIs. Core principles: resources identified by URIs, standard HTTP methods (GET, POST, PUT, DELETE), stateless, client-server separation, caching. Benefits: scalability, simplicity, standard conventions.

502. RESTful resource design: use nouns for resources: `/posts`, `/users`, `/comments`. Avoid verbs: don't use `/getPost`, `/createUser`. URLs represent resources, HTTP methods represent actions.

503. HTTP methods: GET (retrieve, safe, idempotent), POST (create, not idempotent), PUT (replace, idempotent), PATCH (partial update, idempotent), DELETE (remove, idempotent), HEAD (like GET but no body), OPTIONS (describe options).

504. Status codes: 200 OK, 201 Created, 204 No Content, 301 Moved Permanently, 304 Not Modified, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 500 Server Error.

505. API response format: consistent JSON structure: `{ "success": true/false, "data": {...}, "error": {...} }`. Or HAL format with links. Choose format early, document it. Never mix formats.

506. Resource representation: JSON typically, but support XML if needed. Include only necessary fields, use field selection if client needs subset. Document response structure in API docs.

507. Pagination: limit results to prevent huge responses. Implement: page/limit (offset-based), cursor (efficient for large datasets), link headers. Example: `GET /posts?page=2&limit=10`.

508. Sorting: allow sort by multiple fields: `GET /posts?sort=-created_at,+title` (- descending, + ascending). Validate field names, prevent SQL injection, index columns being sorted.

509. Filtering: allow filtering by fields: `GET /posts?status=published&author_id=1`. Use query parameters, validate filter values. Complex filters use Elastic Search or dedicated query language.

510. Search API: full-text search across resources. Example: `GET /search?q=laravel&type=posts`. Use search engines (Elastic, Meilisearch, Algolia) for performance. Implement fuzzy matching, typo tolerance.

511. API versioning: use URL path (`/v1/posts`, `/v2/posts`) or header (`Accept: application/vnd.myapi.v1+json`). Path more discoverable, header cleaner. Maintain backward compatibility for 1-2 versions.

512. Content negotiation: response format determined by `Accept` header or URL extension (`.json`, `.xml`). Implement: `Accept: application/json` returns JSON, `Accept: application/xml` returns XML.

513. API documentation: OpenAPI/Swagger defines API contract. Tools: Swagger UI, Postman, ReDoc. Auto-generate from attributes or write manually. Document endpoints, parameters, responses, errors.

514. Hypermedia (HATEOAS): responses include links to related resources. Example: `{ "id": 1, "name": "Post", "_links": { "self": "/posts/1", "comments": "/posts/1/comments" } }`. Improves discoverability.

515. API rate limiting: throttle requests per user/IP to prevent abuse. `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` headers inform client. Return 429 Too Many Requests when exceeded.

516. API rate limiting implementation: use Redis counter: increment on each request, reset daily/hourly. Example: `$key = "rate-limit:user:$userId"; $count = Redis::incr($key); Redis::expire($key, 3600);`

517. API pagination with cursors: more efficient than offset-based for large datasets. Cursor is opaque token representing position. Client sends cursor in next request, server responds with next page + cursor.

518. API error responses: consistent error format. Example: `{ "error": { "code": "VALIDATION_FAILED", "message": "Email is required", "details": [...] } }`. Include error codes for programmatic handling.

519. API request logging: log method, path, status, response time, user. Example: `POST /api/users 201 125ms user:5`. Useful for debugging, performance analysis, security audits.

520. API request/response examples: include in documentation with real examples. Tools: Swagger/OpenAPI auto-generates examples. Helps API consumers understand format.

521. API client generation: auto-generate client libraries from OpenAPI spec. OpenAPI Generator creates SDK in TypeScript, Python, Java, etc. Reduces manual client implementation effort.

522. API webhooks: server sends notifications to client's endpoint when events occur. Example: on user registration, POST to `https://client.example.com/webhooks/user-created`. Requires signature verification.

523. Webhook delivery: ensure delivery by implementing retries with exponential backoff. Store failed deliveries, manual resend capability. Timeout ~5 seconds, fail fast if endpoint down.

524. Idempotency: ensure repeated requests have same effect as single request. Use idempotency key: client sends `Idempotency-Key` header, server returns same response if resubmitted. Prevents duplicate operations.

525. API testing tools: Postman (GUI), curl (CLI), RestClient (IDE), ThunderClient (VS Code), REST Client extension. Document test cases in collection.

526. API mocking: mock API server for client development. Tools: Prism (OpenAPI-based), Mockoon, json-server. Allows client development before API ready.

527. API response caching: use HTTP caching headers. `Cache-Control: public, max-age=3600` - cache 1 hour. ETags for conditional requests. `Vary: Accept-Language` for language-specific caching.

528. API staleness: allow stale responses for performance. `Cache-Control: max-age=60, stale-while-revalidate=300` - serve stale if revalidation fails. Useful for non-critical data.

529. Graph QL vs REST: GraphQL allows clients to request specific fields, REST returns full resource. GraphQL better for complex queries, REST simpler for simple lookups. Choose based on use case.

530. Bulk operations: `/posts/batch` endpoint for bulk create/update/delete. Reduces number of requests, atomic transactions. Return array of results/errors per item.

531. Nested resources: `/users/1/posts/2` represents post 2 of user 1. Verify user ownership before returning post. Use path parameters: `Route::get('/users/{user}/posts/{post}')`.

532. API gateway: single entry point for multiple backend services. Handles: authentication, rate limiting, routing, logging, monitoring. Tools: Kong, AWS API Gateway, Azure API Management.

533. Microservices API: multiple services exposing APIs, gateway routes requests. Each service owns data, exposes API. Communication via HTTP REST or message queues. Improves scalability, complexity.

534. API monitoring: track performance metrics (response time, throughput, errors). Tools: Datadog, New Relic, APM tools. Alerts on thresholds (>500ms, error rate >5%).

535. API best practices: versioning, documentation, error handling, security, rate limiting, caching, monitoring, testing. Consistency across endpoints, pagination, filtering, sorting.

536. API deprecation: announce deprecation policy upfront. Example: "deprecated endpoints removed after 12 months notice". Version old endpoints until deprecation date, force upgrade before date.

537. API backward compatibility: don't remove fields, only add. Don't change field types or meanings. Use nullable fields for new features. Validate field presence server-side, allow unknown fields.

538. API security headers: same security headers as web. CSP, HSTS, X-Frame-Options, X-Content-Type-Options. Additionally: Strict-Transport-Security for HTTPS enforcement.

539. API authentication: use OAuth2 for delegated access, API keys for service-to-service, JWTs for stateless auth. Tokens in Authorization header or query params. Refresh tokens for long-lived access.

540. CORS for APIs: `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`. Preflight for complex requests. Don't use `*` in production, specify exact origins.

541. OpenAPI 3.0 structure: metadata (title, version), servers, paths, components (schemas, responses, parameters). Auto-generate with Laravel packages (Scramble, OpenAPI generators).

542. Swagger/OpenAPI decorators: annotate controllers/methods with OpenAPI attributes. Example: `#[OA\Get(path: '/posts', ...)]` - documents endpoint in OpenAPI spec automatically.

543. API request validation: validate input before processing. Laravel validation: `$request->validate(['email' => 'required|email'])`. Return 422 with validation errors if invalid.

544. API response wrapping: wrap single resource: `{ "data": {...} }`, collection: `{ "data": [...], "meta": { "total": 100 } }`. Consistent structure improves client parsing.

545. API debugging: include `X-Request-Id` header for tracing across logs. Use header for API consumers to reference issues. Helps troubleshoot complex requests.

546. API performance optimization: pagination (don't return millions of rows), eager loading (prevent N+1), caching (Redis for frequently accessed data), indexing (database query optimization).

547. API scalability: horizontal scaling with load balancers, caching layer (Redis), database optimization (replication, sharding). Monitor metrics, scale based on load.

548. API versioning strategy: path-based (cleaner URLs), header-based (cleaner code), query parameter (least common). Support 1-2 previous versions, deprecate old versions with notice period.

549. API SDK development: if API large/complex, develop SDK for clients. Wrap HTTP calls, handle authentication, provide typed interfaces. OpenAPI can auto-generate SDKs.

550. API contract testing: verify API contracts match clients' expectations. Pact framework for contract testing. Prevents breaking changes, improves confidence in API changes.

## Section 12: Payment Processing & Stripe Integration (Questions 551-650)

551. Stripe integration: payment processing platform. Key: API key (public for frontend, secret for backend), webhook endpoints, payment methods. Benefits: PCI compliance, fraud detection, payment recovery.

552. Stripe API setup: install `composer require stripe/stripe-php`, set `STRIPE_SECRET_KEY` in `.env`. Create Stripe instance: `$stripe = new StripeClient(env('STRIPE_SECRET_KEY'));`

553. Creating payment intents: `$intent = $stripe->paymentIntents->create(['amount' => 5000, 'currency' => 'usd']);` - creates intent for $50. Client receives intent secret, completes payment in frontend.

554. Payment method: customer specifies card/bank account. Stripe securely handles details. Create: `$paymentMethod = $stripe->paymentMethods->create(['type' => 'card', 'card' => [...]]);`

555. Charging customer: `$stripe->charges->create(['amount' => 5000, 'currency' => 'usd', 'source' => $paymentMethodId, 'customer' => $customerId]);` - charges customer directly.

556. Creating customers: `$customer = $stripe->customers->create(['email' => 'user@example.com', 'name' => 'John']);` - represents recurring customer. Store `$customer->id` in database.

557. Saving payment methods: `$stripe->paymentMethods->attach($paymentMethodId, ['customer' => $customerId]);` - attaches payment method to customer for future use.

558. Subscriptions: recurring charges on schedule. Create: `$subscription = $stripe->subscriptions->create(['customer' => $customerId, 'items' => [['price' => $priceId]]]);`

559. Subscription prices: pricing defined in Stripe dashboard or API. Include: amount, currency, interval (monthly, yearly), recurring count. Price ID used in subscriptions.

560. Webhook handling: Stripe sends events (charge.succeeded, charge.failed, customer.subscription.created) to webhook endpoint. Verify signature: `$event = Event::constructFrom($payload);`

561. Webhook security: verify webhook signature using shared secret. `Webhook::constructEvent($body, $sig, $secret)` - throws if invalid. Prevents replay attacks.

562. Handling payment failures: charge may fail (insufficient funds, card declined, 3D Secure required). Webhook sends `charge.failed` event. Retry payment, notify user, attempt recovery.

563. Payment recovery: retry failed payments automatically or prompt user to update payment method. Stripe includes retry logic, PaymentIntent has built-in retry strategies.

564. Handling 3D Secure: if charge requires 3D Secure (authentication), PaymentIntent status becomes `requires_action`, includes authentication URL. Client authenticates, payment completes.

565. SCA (Strong Customer Authentication) / PSD2: EU regulation requiring authentication for payments. Stripe handles automatically. Charge may return `requires_action`, client authenticates via Stripe's hosted page.

566. Idempotency with Stripe: use Idempotency-Key header to prevent duplicate charges. `$stripe->paymentIntents->create([...], ['idempotency_key' => $uuid]);` - repeats safe.

567. Handling card errors: `CardError` (card declined, invalid expiry), `RateLimitError` (Stripe API limit exceeded), `InvalidRequestError` (bad request), `AuthenticationError` (invalid key). Handle each appropriately.

568. Customer portal: hosted page for customers to manage subscriptions. Stripe-hosted, branded, password-protected. Endpoint: `$session = $stripe->billingPortal->sessions->create(['customer' => $customerId]);`

569. Dunning management: recover failed subscriptions via retry logic. Stripe automatically retries failed charges. Configure retry schedule in dashboard.

570. Invoice management: `$invoice = $stripe->invoices->create(['customer' => $customerId]);` - manually create invoice. Or auto-created from subscriptions. Customers receive email, can pay via portal.

571. Refunds: `$stripe->refunds->create(['charge' => $chargeId]);` - full refund. `['charge' => $chargeId, 'amount' => 2500]` - partial refund. Processes within 5-10 business days.

572. Disputes (chargebacks): customer disputes charge with credit card company. Stripe notifies via webhook `charge.dispute.created`. Provides evidence submission deadline (usually 10 days).

573. Fighting disputes: provide evidence (delivery confirmation, digital receipt, customer communication) through Stripe dashboard. Upload documents, explain situation. Stripe submits to card network.

574. Testing Stripe: use Stripe test keys (publishable_test, secret_test). Test cards: 4242 4242 4242 4242 (success), 4000 0000 0000 0002 (decline), various others for specific scenarios.

575. Stripe test scenarios: test failures, retries, webhook delivery. Use test event JSON to simulate webhooks locally. Webhook testing tools: ngrok for local webhook testing.

576. Stripe tax calculation: `$stripe->tax->transactions->createFromCalculation(['calculation_id' => $calcId])` - applies taxes calculated by Stripe Tax. Supports 70+ jurisdictions.

577. Stripe tax compliance: handles sales tax, VAT, GST for subscription billing. Dashboard shows tax configuration by location. Use for compliance, reporting.

578. Stripe reporting: export transaction data from dashboard. API access to all charges, subscriptions, refunds. Integrate with accounting software (QuickBooks, Xero).

579. Stripe fraud detection: Stripe analyzes patterns for fraud. Suspicious charges flagged (charge.fraud_detected event). Configure rules in dashboard.

580. Stripe radar: built-in fraud detection using machine learning. Customize rules per business. Block/alert on high-risk charges. Improves over time with data.

581. Stripe charges vs PaymentIntents: Charges are simple one-time (legacy), PaymentIntent support advanced scenarios (3D Secure, SCA, payment recovery). Use PaymentIntent for new implementations.

582. PaymentElement: Stripe's hosted element for payments. Handles all payment method logic. Frontend: `<stripe-payment-element></stripe-payment-element>`, Stripe automatically handles UI.

583. Stripe Connect: marketplace/platform payments. Separate account creation, payment routing. Stripe handles split payments, settlement to sellers. Good for multi-vendor platforms.

584. Stripe ACH transfers: US bank transfers cheaper than cards. Setup: `$paymentMethod = $stripe->paymentMethods->create(['type' => 'us_bank_account', ...]);` - collect bank details.

585. Stripe international payments: support 135+ currencies, 100+ payment methods (alipay, klarna, etc). Automatically handles payment method routing by location.

586. PCI compliance: with Stripe, you don't handle card data (Stripe's responsibility). Card data never touches your servers. Simplifies PCI compliance, reduces liability.

587. Laravel Cashier: simplifies Stripe integration. `$user->charge(2000, 'default_payment_method')` charges user. `$user->subscriptionBuilder($planId)->create()` creates subscription. Reduces boilerplate.

588. Stripe dashboard: configure: products, prices, tax rates, webhook endpoints, API keys. Monitor: recent charges, disputes, fraud, revenue, subscription metrics.

589. Stripe webhooks production: generate webhook signing secret from dashboard. Add endpoint URL. Stripe retries failed deliveries for 72 hours. Verify all webhooks before processing.

590. Stripe balance: money held by Stripe, settled to bank account. View in dashboard: available (cleared), pending (processing), reserved (held for disputes). Settlement schedule (daily, weekly).

591. Marketplace payments: Split payments using Stripe Connect. Create connected account for seller: `$account = $stripe->accounts->create(['type' => 'express', ...]);` - generates onboarding link.

592. Application fees: take percentage fee from each transaction. With Connect: `'application_fee_amount' => 500` on charge - takes 5% fee for marketplace.

593. Subscription changes: customer upgrades/downgrades plan. `$stripe->subscriptions->update($subscriptionId, ['items' => [['id' => $itemId, 'price' => $newPriceId]]]);` - applies pro-rata adjustments.

594. Subscription cancellation: `$stripe->subscriptions->cancel($subscriptionId);` - ends immediately. Or schedule: `['cancel_at' => $timestamp]` - ends at timestamp.

595. Subscription pause: `$stripe->subscriptions->update($subscriptionId, ['pause_collection' => ['behavior' => 'pause']])` - pauses billing, continues later. Resume: remove pause.

596. Billing cycle anchors: control when renewal occurs. Set: `['billing_cycle_anchor' => $timestamp]` - aligns all subscriptions to same day. Useful for monthly billing on specific date.

597. Proration: when changing mid-cycle, Stripe adjusts charges. Example: upgrade mid-month, credit for days remaining on old plan, charge for new. Controlled via `proration_behavior`.

598. Stripe compliance: GDPR (right to delete customer), PCI DSS (never store card data), export controls. Check dashboard for compliance status, export data when needed.

599. Stripe pricing models: flat-rate (fixed fee per transaction), percentage-based (2.2% + $0.30), subscription ($25/month for features). Choose based on volume, transaction size.

600. Stripe documentation: extensive API docs, SDKs for all languages, code examples. Dashboard provides API reference. Build live API example tool for testing.

## Section 13: Advanced OOP & Design Patterns (Questions 601-700)

601. SOLID principles: Single Responsibility (one class, one reason to change), Open/Closed (open for extension, closed for modification), Liskov Substitution (subtypes substitutable), Interface Segregation (specific interfaces), Dependency Inversion (depend on abstractions).

602. Single Responsibility: class should have single purpose. Example: `UserValidator` validates users, `UserRepository` fetches users, `UserNotifier` sends notifications. Separation enables testing, reuse.

603. Open/Closed Principle: open for extension (add new features), closed for modification (don't change existing code). Use inheritance/composition: `class PdfExporter extends Exporter` - new exporter without modifying old.

604. Liskov Substitution: subclass can replace parent without breaking code. Example: `function save(User $user)` accepts both `User` and subclass `Admin`. Subclass can't require additional parameters, can't weaken contracts.

605. Interface Segregation: client-specific interfaces over general-purpose. Instead of `Worker` (work, eat, sleep), use `Workable`, `Eatable`, `Sleepable`. Client implements only needed interfaces.

606. Dependency Inversion: depend on abstractions (interfaces), not implementations. Constructor injection: `function __construct(Logger $logger)` should type-hint interface: `LoggerInterface`. Allows swapping implementations.

607. Design Patterns: reusable solutions for common problems. Creational (constructing objects), Structural (organizing objects), Behavioral (communication between objects). Learn patterns to communicate solutions.

608. Singleton pattern: single instance of class, shared globally. Implementation: private constructor, static method returning instance. Laravel service container implements singletons. Use rarely (testability issues).

609. Factory pattern: create objects without specifying classes. `VehicleFactory::create('car')` returns Car instance. Abstracts object creation, simplifies client code.

610. Repository pattern: abstraction for data access. `UserRepository::find($id)` fetches user, implementation hides database details. Simplifies testing (swap with mock repository).

611. Observer pattern: notify multiple objects of events. Publisher (notifies), Subscriber (listens). Laravel events implement this. `Event::listen('UserCreated', callback)` - callback fires on event.

612. Strategy pattern: multiple algorithms, client selects one. `$validator = new EmailValidator()` or `PhoneValidator()` - both implement `Validator` interface. Interchangeable at runtime.

613. Decorator pattern: add behavior to object dynamically. Example: `new CachedUserRepository(new DatabaseUserRepository())` - caches on top of database repository.

614. Adapter pattern: make incompatible interfaces compatible. Example: `PaymentGatewayAdapter` wraps Stripe API, provides consistent interface. Allows swapping payment providers.

615. Builder pattern: construct complex objects step-by-step. `QueryBuilder::where(...)->orderBy(...)->limit(10)->get()` - fluent interface builds query. Improves readability over constructor with many params.

616. Template Method pattern: define algorithm structure in base class, subclasses override steps. Example: `PaymentProcessor` has `validate()`, `charge()`, `confirm()` - subclasses override charge() for different providers.

617. Chain of Responsibility: pass request through handlers until one handles. Middleware pipeline implements this - each middleware can handle or pass to next.

618. State pattern: object behavior changes based on state. `PaymentState::Pending`, `Processing`, `Completed` - each state defines allowed transitions. Example: order workflow.

619. Composite pattern: tree structure of objects. `File` and `Folder` both extend `Node`, folder contains files/folders. Recursive structure, treat individual/composite uniformly.

620. Proxy pattern: placeholder for another object. Lazy loading, access control, logging. Example: `UserProxy` loads user lazily from database when accessed.

621. Facade pattern: simplified interface to complex subsystem. `Stripe` facade provides easy methods instead of multiple API calls. `Cache::put()` wraps cache driver complexity.

622. Mediator pattern: central object coordinates between others. `ChatRoom` mediates between users - users send messages to chat room, which broadcasts to others.

623. Iterator pattern: sequentially access collection elements without exposing structure. `foreach` uses Iterator interface internally. `Collection::each()` provides iteration.

624. Prototype pattern: create copies of object. `clone $object` creates shallow copy. Override `__clone()` for deep copy. Useful for expensive-to-create objects.

625. Flyweight pattern: share common data between objects to reduce memory. Example: font objects in text editor - instead of storing font per character, reference shared font object.

626. Command pattern: encapsulate request as object. `RefundCommand`, `ChargeCommand` - can be queued, logged, undone. Useful for undo/redo, command logging.

627. Value Object pattern: immutable object representing value (money, color, date). Example: `class Money { public function __construct(int $amount, string $currency) { $this->amount = $amount; } }` - never modified.

628. Data Transfer Object (DTO): simple class holding data for transfer between layers. Example: `CreateUserRequest { name, email, password }` - doesn't contain logic, just holds data.

629. Aggregate pattern: DDD (Domain-Driven Design) concept. Aggregate is root entity + related entities/values. Example: Order aggregate contains OrderLineItem objects, manages them together.

630. Entity vs Value Object: Entity has identity (User with ID), Value Object doesn't (Money doesn't have ID, just amount/currency). Entities mutable, Values immutable.

631. Dependency Injection Container: auto-resolves dependencies. Laravel's service container: `app()->make(UserService::class)` - automatically injects UserRepository into UserService.

632. Service Locator anti-pattern: object fetches own dependencies from container. Less testable than dependency injection. Laravel discourages service locator, prefers dependency injection.

633. Inheritance vs Composition: favor composition over inheritance. Composition more flexible: `Engine` composed in `Car` vs `Car extends Vehicle`. Composition avoids rigid hierarchies.

634. Trait mixins: share code across unrelated classes. `use SoftDeletes` in multiple models. Better than inheritance for cross-cutting concerns.

635. Interface segregation in Laravel: specific interfaces for specific behaviors. `Authenticatable`, `Authorizable`, `Cacheable` - classes implement needed interfaces.

636. Anti-patterns: God Object (too much responsibility), Copy-paste code (reuse instead), Hardcoding (use config/DI), Tightly coupled code, Unclear naming. Learn to identify and refactor.

637. Refactoring: improve code without changing behavior. Techniques: Extract Method (reduce function length), Extract Class (separate concerns), Move Method (better location), Rename (clearer intent).

638. Testing-driven design: tests force better design. Tight coupling hard to test, drives loose coupling. SOLID principles emerge naturally when writing testable code.

639. Hexagonal Architecture: core domain at center, external systems at edges (database, APIs, UI). Ports (interfaces) separate domain from systems. Tests replace external systems with fakes.

640. Onion Architecture: core (domain), application, infrastructure layers. Dependencies point inward. Domain knows nothing about infrastructure. Testable, flexible.

641. CQRS (Command Query Responsibility Segregation): separate read and write models. Commands modify state, Queries read state. Scales reads independently, eventual consistency for reports.

642. Event Sourcing: store events (state changes) instead of state. Rebuild state by replaying events. Benefits: audit trail, time-travel debugging, event replay. Complexity: eventually consistent.

643. API client pattern: dedicated class encapsulating API interactions. Example: `StripeClient` wraps Stripe API. Easier testing, consistent error handling, central place for API changes.

644. Wrapper/Wrapper pattern: wrap third-party library for easy swapping. Example: `PaymentGatewayWrapper` wraps Stripe/Paypal, provides consistent interface. Reduces coupling.

645. Delegation pattern: object delegates work to another. Example: `Repository::save()` delegates to `Database` and `Cache`. Separates concerns, reuses logic.

646. Null Object pattern: return special null-like object instead of null. Prevents null-check everywhere. Example: `Optional::ofNullable($value)->map(fn($v) => ...)` - no null checks needed.

647. Parameter Object pattern: group related parameters into object. Instead of `function($name, $email, $phone)`, pass `User` object. Reduces parameter count, improves readability.

648. Tell, Don't Ask principle: tell objects what to do, don't ask state and do it yourself. `$user->promote()` better than `if ($user->role == 'user') { $user->role = 'admin'; }`.

649. Favor composition over inheritance: inheritance creates rigid hierarchies, composition flexible. Compose behavior instead of inheriting. Easier to test, change, understand.

650. DRY (Don't Repeat Yourself): eliminate duplication, create abstractions. Extract common code to methods, classes, traits. Duplication indicates missing abstraction, harder to maintain.

## Section 14: Message Queues & Background Jobs (Questions 651-750)

651. Laravel queues: decouple time-consuming tasks. Push jobs to queue, workers process asynchronously. Benefits: faster response times, background processing, distribution across servers.

652. Queue drivers: database (simple, slow), Redis (fast, reliable), SQS (AWS, scalable), Beanstalk (AWS, reserved), Sync (testing, immediate execution).

653. Creating jobs: `php artisan make:job SendEmail` creates `App\Jobs\SendEmail`. Implement `handle()` method with job logic. `dispatch()` queues job for processing.

654. Job dispatch: `SendEmail::dispatch($user)` queues job. `dispatch(new SendEmail($user))` alternative syntax. Both add job to queue, return immediately.

655. Delayed jobs: `SendEmail::dispatch($user)->delay(Carbon::now()->addSeconds(30))` - job executes after delay. Useful for scheduled emails, retry attempts.

656. Job queues: assign jobs to specific queues: `$this->onQueue('emails')` in constructor. Run workers on specific queue: `php artisan queue:work --queue=emails`. Prioritize critical queues.

657. Queue priority: `$this->queue('critical')` runs first. Multiple queues: critical, normal, low. Workers process in order, skip empty queues. Good for SLA management.

658. Failed jobs: job throws exception, Laravel stores in failed job table. Review in dashboard: `php artisan queue:failed-list`. Retry: `php artisan queue:retry {id}` or `all` for all.

659. Job retries: configure retries in job: `public $tries = 3;` - job retried 3 times before giving up. Use exponential backoff: `public $backoff = [60, 120, 300]` - wait 1min, 2min, 5min.

660. Max job time: set timeout: `public $timeout = 60` - job killed if takes >60s. Prevents hanging jobs. System must gracefully handle partial state.

661. Queue workers: `php artisan queue:work` processes jobs continuously. Workers listen on queue, fetch jobs, execute `handle()` method, mark complete. Consume CPU, memory.

662. Daemon workers: process runs continuously in background. Production setup: multiple workers on each server, managed by Supervisor (monitors, restarts). Preferable to cron jobs.

663. Supervisor configuration: monitors workers, auto-restarts if crashes. Config: program name, command, directory, env, restart policy. Example: monitor 3 queue workers.

664. Queue monitoring: `php artisan queue:failed-list` shows failed jobs. `php artisan queue:work --verbose` shows debug info. Tools: Horizon (dashboard) shows queue metrics.

665. Database queue: `'QUEUE_CONNECTION=database'` in `.env`, run `php artisan queue:table` to create table. Simple but slow; queue stored in database, workers poll. Use for development/small apps.

666. Redis queue: `'QUEUE_CONNECTION=redis'` - fast, reliable. Redis handles queue storage, workers subscribe to channels. Preferred for production.

667. SQS queue: `'QUEUE_CONNECTION=sqs'` - AWS service, managed, scales automatically. No server management, pay per message. Good for serverless, high volume.

668. Queue event monitoring: `Queue::looping()`, `Queue::exceptionOccurred()` callbacks. Log when workers start, jobs fail, etc. Useful for debugging.

669. Job middleware: apply middleware to jobs. `$this->middleware(new RateLimitMiddleware())` in job. Middleware can skip job, retry, or transform data before execution.

670. Batch processing: group related jobs, process together. `Bus::batch([new Job1(), new Job2(), ...])->then(fn() => ...)->finally(fn() => ...)->dispatch()` - callback after all complete.

671. Job event hooks: fired during job lifecycle. `ProcessingConnection`, `ProcessingQueue`, `Processing` (starting), `Processed` (completed), `Failed` (exception). Use for logging, monitoring.

672. Listener jobs: listen for events, queue jobs. Example: `UserCreated` event triggers `SendWelcomeEmail` job. Decouple event handling from job dispatch.

673. Queued event listeners: `public $queue = 'emails'` in listener - event listener queued instead of synchronous. Faster initial response, eventual consistency.

674. Synchronous queue: `'QUEUE_CONNECTION=sync'` - execute immediately, no actual queueing. Useful for testing (no separate worker needed), development.

675. Queue with Tinker: `dispatch(new Job())` in tinker shell. `Queue::size()` gets queue length. Test jobs without workers running.

676. Timeout handling: job exceeds timeout, killed by worker. Partial state: database transaction rolled back if configured, or manual cleanup needed. Implement graceful shutdown.

677. Job payloads: serialize job data for transmission. Custom properties: `public function __construct(public User $user, public int $timeout) {}`. PHP serializes, deserializes on execution.

678. Serialization concerns: avoid serializing database connections, file handles. Only serialize simple data. Use IDs instead of full objects.

679. Job state: no state between executions. Job re-executed if retried, must be idempotent. Use database to track state if needed.

680. Job tagging: categorize jobs with tags: `public $tags = ['emails', 'users']` in job. Use for metrics, filtering, conditional processing.

681. Queue flush: `php artisan queue:flush` clears all pending jobs. Useful when deploying breaking changes. Lost jobs - consider queue drain (process before deleting).

682. Rate limiting jobs: prevent too many from executing simultaneously. Use Redis semaphore: `Redis::throttle('key')->allow(5)->every(60)->block(0)->then(fn() => ...)`.

683. Queue performance tuning: use dedicated queue servers, tune worker count (1 per CPU core), monitor memory/CPU, batch similar jobs, optimize job handler logic.

684. Job retry with backoff: `public $backoff = [30, 60, 120]` - retry after 30s, 60s, 120s. Gives failing service time to recover. Use for external APIs.

685. Job retry strategy: retry on specific exceptions only. `public $maxExceptions = 3` - fail completely after 3 exceptions. Or catch, decide per exception.

686. Job async processing: long-running job (report generation, image processing). Job runs in background, client doesn't wait. Notify completion via email/notification.

687. Job progress tracking: store progress in cache/database during execution. Frontend polls for progress. Long jobs (~minutes) should show progress, not hang.

688. Queue naming: use descriptive names like `emails`, `payments`, `reports`. Workers can specialize - high-priority workers handle payment jobs, lower-priority for reports.

689. Dead Letter Queue: failed jobs after retries stored for analysis. Review, fix root cause, retry. Prevents infinite retry loops, isolates problematic jobs.

690. Queue transactions: job should be atomic. If partial failure, use transactions to rollback or implement idempotency (safe to re-execute).

691. Queue notifications: notify user when background job complete. Store job ID, user subscribes to completion. Email/webhook/push notification when done.

692. Queue testing: `Queue::fake()` prevents actual dispatch. `Queue::assertPushed(JobClass::class)` verifies job queued. `Queue::assertNotPushed()` verifies not queued.

693. Mocking job dependencies: inject dependencies into job constructor, mock in tests. `$job = new MyJob($mockService)` - test with mocked service.

694. Job failure notification: override `failed()` method in job. Called when job finally fails after retries. Example: send error email to admin.

695. Chained jobs: `Bus::chain([Job1::class, Job2::class, ...])->dispatch()` - execute jobs sequentially. Job2 starts only after Job1 completes. Useful for dependent tasks.

696. Job batches: similar to chaining but parallel. `Bus::batch([...])->then()->dispatch()` - execute all, then callback. Better for independent parallel jobs.

697. Queue dashboard (Horizon): real-time monitoring of queue metrics. Shows throughput, failures, pending jobs. Useful for debugging queue issues.

698. Queue pruning: old jobs fill failed job table. `queue:prune` command removes failed jobs older than N days. Schedule daily: `$schedule->command('queue:prune')->daily();`

699. Queue reservation: job reserved (locked) when worker starts, released if worker dies/timeout. Prevents duplicate processing by multiple workers. Timeout duration configurable.

700. Queue connection settings: timeout, retry delay, max failures. Configure per connection in `config/queue.php`. Different settings for different environments (dev faster retries, prod slower).

## Section 15: Advanced Database Concepts (Questions 701-800)

701. Database transactions: atomic operations - either all succeed or all fail. Syntax: `DB::transaction(fn() => { save(), delete(), etc. })`. Prevents partial updates on failure.

702. ACID properties: Atomicity (all or nothing), Consistency (valid state), Isolation (concurrent transactions don't interfere), Durability (committed persists). Fundamental database guarantees.

703. Isolation levels: Read Uncommitted (dirty reads), Read Committed (default, safe), Repeatable Read (consistent reads), Serializable (transactions sequentially). Trade-off: safety vs concurrency.

704. Dirty reads: transaction reads uncommitted data from other transaction. Prevented in Read Committed+. Causes: low isolation level, reading intermediate state.

705. Phantom reads: query returns different rows due to concurrent insert/delete. Prevented in Repeatable Read+. Example: query returns different count due to concurrent insert.

706. Lost updates: two transactions overwrite each other's changes. Prevention: row-level locking, versioning (optimistic locking), serializable isolation. Handle with application logic.

707. Optimistic locking: use version column. Read version, update if version unchanged, increment version. If fails (version changed), retry transaction. Prevents lost updates without locks.

708. Pessimistic locking: lock rows during transaction. `SELECT ... FOR UPDATE` locks rows until commit. Prevents concurrent modification but can deadlock. Use for critical sections.

709. Deadlocks: two transactions wait for each other's locks. Solution: always acquire locks in same order, use timeouts, minimize lock duration. Detect with `SELECT ... FOR UPDATE NOWAIT`.

710. Distributed transactions: transactions spanning multiple databases. Use two-phase commit (2PC) but risky. Prefer saga pattern: local transactions coordinated by compensating transactions.

711. Database indexing: fast lookups via B-tree structure. Without index: full table scan (O(n)), with index: logarithmic (O(log n)). Trade-off: slower writes (index updates), faster reads.

712. Index selectivity: fraction of rows matching condition. High selectivity (few rows) - index helps. Low selectivity (many rows) - table scan faster. Example: gender (2 values) poor selectivity.

713. Composite indexes: multiple columns. `CREATE INDEX idx ON users(status, created_at)` - helps queries filtering by both. Order matters: most selective first. Leftmost prefix queryable.

714. Index statistics: database maintains estimates for query planning. Outdated stats cause bad plans. Update with `ANALYZE TABLE` (MySQL), `ANALYZE STATISTICS` (PostgreSQL). Regular maintenance.

715. Query execution plans: explain how database executes query. `EXPLAIN SELECT ...` shows plan. Look for: full table scan (bad), index scan (good), hash join, nested loop, etc.

716. Covering index: index includes all columns needed, no table lookup. Example: index on (id, name, email) for query selecting all three. Faster than index + table lookup.

717. Partial indexes: index subset of rows. `CREATE INDEX idx ON users(email) WHERE status = 'active'` - only indexes active users. Reduces index size, speeds lookups.

718. Unique indexes: enforce column uniqueness. `CREATE UNIQUE INDEX` prevents duplicates. PRIMARY KEY is unique index on primary key column. Can combine with partial index.

719. Foreign keys: enforce referential integrity. `FOREIGN KEY (user_id) REFERENCES users(id)` - prevents orphaned rows. ON DELETE CASCADE deletes children on parent delete.

720. Cascade delete: dangerous if not careful. Child rows deleted when parent deleted. Example: user deleted, all posts deleted. Can cause data loss. Use carefully, document.

721. Database views: virtual tables from queries. `CREATE VIEW active_users AS SELECT * FROM users WHERE status = 'active'`. Simplify complex queries, provide abstraction layer.

722. Materialized views: views with persisted data. Not live (doesn't query base tables), must refresh. Faster reads but stale data. Use for reports, analytics.

723. Query optimization: identify slow queries in logs, explain plan, add indexes, rewrite query. Tools: MySQL Slow Query Log, PostgreSQL query analyzer, APM tools (New Relic).

724. N+1 problem: fetching parent then children one-by-one in loop. Parent query returns 100 rows, 100 child queries (N+1). Solution: eager load relationships.

725. Eager loading in Laravel: `User::with('posts')->get()` - single query fetches users, single query fetches all posts. Prevents N+1, critical for performance.

726. Lazy evaluation: postpone expensive operations. Laravel queries lazy until executed: `User::where(...)` builds query but doesn't run. Only runs on `->get()`, foreach, etc.

727. Query caching: database caches query results. Parser cache (query plans), result cache (results). Invalidated on writes. Use with caution - can mask slow queries.

728. Database sharding: partition data across servers. Each server holds subset of data. Improves scalability, but complex: ID generation, cross-shard queries, consistency.

729. Sharding strategies: range-based (id ranges), hash-based (hash ID to shard), geographic (by location). Choose based on access patterns. Rebalancing required if shards become unequal.

730. Replication: copies of database for failover, load balancing. Master-Slave (master writes, slaves read), Master-Master (both write, conflicts). Replication lag: eventual consistency.

731. Database failover: switch to replica if master fails. Automatic (health checks), manual (admin intervention). Requires unique constraints to prevent duplicate IDs, careful recovery.

732. Connection pooling: reuse database connections. Creates pool of open connections, reuses for requests. Reduces connection overhead. Configured in database adapter.

733. Connection limits: databases have max connection limits. Connection leaks (not releasing) exhaust pool. Monitor with `SHOW PROCESSLIST` (MySQL). Close connections explicitly if needed.

734. Database migrations: version control for schema changes. `php artisan migrate` runs new migrations. Reversible with rollback. Document schema evolution, enable team collaboration.

735. Migration best practices: small migrations (one change), reversible with `down()`, no data loss (add column with default, delete in separate migration).

736. Schema builder: fluent syntax for migrations. `Schema::create('users', fn($table) => $table->id()->string('name'))` - creates table with id, name. Supported by MySQL, PostgreSQL.

737. Database seeding: populate test/demo data. `php artisan seed` runs seeders. Use factories for realistic data. Separate seeders by environment (demo, test).

738. Bulk operations: insert multiple rows at once. `DB::table('users')->insert([...])` - single query inserts all. Faster than individual inserts. Useful for imports.

739. Batch inserts: split large insert into smaller batches to avoid memory issues. Insert 1000 at a time instead of 1M. `collect($items)->chunk(1000)->each(fn($chunk) => insert($chunk))`.

740. Soft deletes: mark deleted instead of removing. `deleted_at` timestamp. Query excludes soft-deleted by default. `->withTrashed()` includes them. Good for audit trails, recovery.

741. Paranoid mode: query always excludes soft-deleted. Implemented via global scope. Prevents accidental access to deleted records.

742. Database logging: log all queries for debugging. `DB::enableQueryLog()` then `DB::getQueryLog()` shows executed queries with bindings. Useful for N+1 detection.

743. Query monitoring: track slow queries in production. Set threshold (>500ms), log/alert. Use APM tools or database slow query log. Profile regularly.

744. Explain analyze: `EXPLAIN ANALYZE` actually executes query, shows real execution stats (not estimates). More accurate than EXPLAIN but slower. Use for debugging.

745. Index hints: suggest which index to use. `USE INDEX (idx_name)` or `IGNORE INDEX`. Should be unnecessary if stats/optimizer good. Use only if query planner chooses badly.

746. Query rewriting: rewrite complex query for better performance. Use JOINs instead of subqueries, push down filters (WHERE before JOIN), select only needed columns.

747. Common Table Expressions (CTEs): named subqueries. `WITH cte AS (SELECT ...) SELECT * FROM cte` - improves readability, reusable. PostgreSQL, newer MySQL versions.

748. Window functions: compute values relative to rows. `ROW_NUMBER() OVER (ORDER BY created_at)` - row number per user. Leaderboards, running totals. SQL standard.

749. EXPLAIN output interpretation: type (index, seq scan), rows (estimated), extra (using where, using index, etc). Full index scan better than seq scan. "Using index" means no table lookup.

750. Database monitoring: monitor connections, slow queries, replication lag, disk space, CPU. Tools: Percona Monitoring, pt-query-digest, database-native tools.

## Section 16: DevOps, Deployment & Production (Questions 751-850)

751. Deployment pipeline: code -> test -> build -> deploy -> monitor. CI/CD automates steps. Tools: GitHub Actions, GitLab CI, Jenkins. Enables fast, reliable releases.

752. Staging environment: production-like environment for testing before live. Same config, database (anonymized), servers. Test migrations, performance, integrations. Catch issues early.

753. Blue-green deployment: two identical environments. Deploy to inactive (green), test, switch traffic to green. Instant rollback if issues. Eliminates downtime.

754. Canary releases: deploy to small percentage of users, monitor, gradually increase. Catch issues affecting subset before full rollout. Slow rollout (hours to days).

755. Rolling deployment: gradually replace servers with new version. 10% servers updated, monitor, then 25%, etc. Maintains capacity, but increased complexity.

756. Database migrations in production: risky operation. Backup first, test in staging, schedule maintenance window if risky. Zero-downtime: expand schema (add column), deploy code, remove old column.

757. Zero-downtime deployment: code supports both old and new schema/API. Deploy code first (backwards compatible), then migrate schema, then remove old support. Takes time but safer.

758. Version control: GitHub, GitLab, Bitbucket. Branches for features, main for release. PR reviews, CI checks before merge. Enables collaboration, history, rollback.

759. Git workflow: main (stable), develop (integration), feature branches (work-in-progress). Merge develop to main for release. Hotfix branches for urgent fixes.

760. Semantic versioning: MAJOR.MINOR.PATCH (1.2.3). Major: breaking changes, Minor: new features, Patch: bug fixes. Helps users understand compatibility.

761. Release notes: document changes, breaking changes, migration steps, known issues. Important for users to plan upgrades. Generated from commit messages/PR titles.

762. Rollback strategy: ability to quickly revert to previous version. Keep previous releases, database migrations must be reversible. Document rollback procedure.

763. Health checks: endpoints returning system status. `/health` returns 200 OK if healthy, 500 if degraded. Used by load balancers, monitoring. Check: database connectivity, external APIs, disk space.

764. Error tracking: catch exceptions, report to service (Sentry, Rollbar). Automatically notified of errors, collect stack traces, affected users. Enables proactive debugging.

765. Application monitoring: track metrics (response time, throughput, error rate), logs (application events), traces (request flow). Tools: New Relic, Datadog, Prometheus + Grafana.

766. Infrastructure as Code: define servers, networks, databases in code. Terraform, CloudFormation. Enables reproducible, version-controlled infrastructure. Disaster recovery becomes re-applying code.

767. Container deployment: Docker for packaging app. Kubernetes (K8s) for orchestration, scaling, networking. Simplifies deployment, enables microservices, improves resource utilization.

768. Load balancing: distribute traffic across servers. Health checks remove failed servers. Algorithms: round-robin, least connections, IP hash. Improves availability, scalability.

769. Auto-scaling: automatically add/remove servers based on load. Metrics: CPU, memory, request count. Increases capacity on high load, decreases on low load. Cost-effective.

770. SSL/TLS certificates: HTTPS encryption. Let's Encrypt provides free certificates (90-day expiry, auto-renewal). Configure in web server (nginx, Apache). Mandatory for production.

771. HTTPS enforcement: redirect HTTP to HTTPS in middleware or web server. Ensures all traffic encrypted. Check HSTS header for browser enforcement. Never serve sensitive data over HTTP.

772. Database backups: automated daily/hourly backups. Storage: separate location (different server, cloud). Encryption for security. Test restores regularly. Recovery Time Objective (RTO) and Recovery Point Objective (RPO) defined.

773. Backup retention: keep backups for configurable period (7-30 days). Old backups deleted to save space. Immutable backups for compliance (WORM - Write Once, Read Many).

774. Disaster recovery plan: document steps for major outages. Communication, incident response, recovery procedures. Regular drills to test. RTO/RPO targets (e.g., 1 hour RTO, 15 min RPO).

775. Secrets management: API keys, database passwords in secure vault (HashiCorp Vault, AWS Secrets Manager). Never in code, config files, logs. Rotation mechanism.

776. Environment configuration: use environment variables for config. Secrets, database URLs, API endpoints configured per environment. `.env` file for development, secrets manager for production.

777. Application logs: log important events (requests, errors, business events). Levels: DEBUG, INFO, WARNING, ERROR. Structured logging (JSON) for parsing, querying. Retention: aggregate old logs.

778. Log aggregation: centralize logs from multiple servers. Tools: ELK stack (Elasticsearch, Logstash, Kibana), Splunk, Datadog. Enables searching, alerting, pattern detection across servers.

779. Alerting: notifications when metrics exceed thresholds. Email, SMS, Slack alerts. Critical: immediate notification, warning: less urgent. Avoid alert fatigue (too many alerts).

780. On-call rotation: team members on-call for urgent issues. Escalation policy if primary doesn't respond. Tools: PagerDuty, Opsgenie. Ensures issues get attention.

781. Status page: public page showing service status. Updates on incidents, estimated time to resolution. Reduces support requests. Tools: StatusPage.io, Cachet, Instatus.

782. Change management: control changes to production. Change log, approval process, deployment schedule. Prevent risky changes during peak hours. Rollback plan documented.

783. Access control: limit who can deploy, access logs, manage secrets. SSH keys for server access, separate keys per environment. Audit log of access, changes.

784. Security patching: apply security updates regularly. Monitor CVEs (Common Vulnerabilities), update dependencies. Automated: `composer audit`, Dependabot. Security updates may not wait for normal release cycle.

785. Performance tuning: baseline current performance, identify bottlenecks (database queries, slow endpoints, memory), optimize, measure improvement. Continuous process.

786. CDN (Content Delivery Network): cache static assets on edge servers globally. Assets served from nearest server, faster for users, reduces server load.

787. Caching strategy: layer caching - browser (HTTP headers), CDN (geography), server (Redis). Each layer reduces load on layer below. Cache busting with version hashes.

788. Service mesh: manages traffic between services. Handles: load balancing, retries, circuit breaking, monitoring. Tools: Istio, Linkerd. Advanced networking for microservices.

789. API Gateway: single entry point for microservices. Handles: routing, rate limiting, authentication, logging. Simplifies client interactions, centralizes policies.

790. Message brokers: decouple services with async messaging. RabbitMQ, Kafka, Redis Streams. Services publish events, others subscribe. Improves resilience, enables scalability.

791. Event-driven architecture: services communicate via events. Order service publishes OrderCreated event, notification service subscribes. Loose coupling, independently scalable.

792. Circuit breaker: prevent cascading failures. If service down (failures >threshold), circuit "opens", fail fast without calling service. Auto-recovers when service back up.

793. Bulkheads: isolate resources. Thread pools per service, separate queues. Prevents one service from consuming all resources, affecting others.

794. Graceful degradation: service partially unavailable, critical features still work. Example: cache used if database down. Better UX than complete outage.

795. Chaos engineering: intentionally inject failures to test resilience. Chaos Monkey randomly kills servers, network partitions, etc. Reveals weaknesses before production incidents.

796. Feature flags: deploy code but disable in production. Gradually enable for users. A/B testing, rollback without redeployment. Tools: LaunchDarkly, Split.io.

797. Observability: understand system behavior. Three pillars: logs (events), metrics (measurements), traces (requests). Enables debugging complex issues, monitoring.

798. Distributed tracing: track requests across services. Trace ID propagated through calls. Shows latency per service, identifies slow services. Tools: Jaeger, Zipkin.

799. Profiling: measure where code spends time. CPU profiling (CPU hotspots), memory profiling (memory allocation), I/O profiling (network/disk access). Identifies optimization opportunities.

800. Incident response: when issues occur, quick response minimizes impact. War room (sync communication), root cause analysis, fix, post-mortem (prevent recurrence). Blameless culture.

## Section 18: Monitoring, Logging & Debugging (Questions 851-950)

859. Performance monitoring: track metrics that matter. Response time (per endpoint), throughput (requests/second), error rate. Alert on degradation.

868. Performance profiling: measure where time/resources spent. Blackfire, Xdebug profiler, PHP-SPX. Identifies bottlenecks. CPU, memory, I/O profiling available.

870. CPU profiling: find CPU hotspots. Which functions use most CPU. Optimize frequently-called functions. Use sampling profiler (low overhead) vs instrumentation (accurate but slow).

871. Network profiling: measure HTTP request/response times. Breakdown: DNS, connection, SSL, time to first byte (TTFB), download time. Browser developer tools show breakdown.

872. Benchmarking: measure code performance. `microtime()` for simple, Blackfire for detailed. Compare: different algorithms, with/without cache. Trends over releases.

873. Load testing: simulate concurrent users, measure performance under load. Tools: ApacheBench (ab), wrk, JMeter. Test against staging, not production!

874. Stress testing: increase load until failure. Identify breaking point, resource exhaustion. Useful for capacity planning.

875. Latency profiling: measure end-to-end latency (user perspective). Network latency (client to server), server processing time, rendering time. Important for user experience.

876. Request tracing: track request through multiple services. Distributed tracing IDs. Shows latency per service. Tools: Jaeger, Zipkin, Datadog.

877. Observability tools: comprehensive monitoring. New Relic, Datadog, APM tools. Combine metrics, logs, traces. Enable debugging complex systems.

878. Custom metrics: track business metrics (signups/day, revenue, conversion rate). Not just infrastructure metrics. Use APM or custom solutions (StatsD, Prometheus).

879. Alert thresholds: set realistic thresholds to avoid false alarms. Alert on meaningful degradation, not normal variation. Escalation: warning at lower threshold, critical at higher.

880. Alert fatigue: too many alerts loses effectiveness. Focus on actionable alerts. Tune thresholds, combine related alerts. Better fewer important alerts than many noise.

881. Dashboard design: visualize important metrics. Real-time updates, historical trends, drill-down capabilities. Different dashboards for different audiences (ops, business).

882. Alerting via multiple channels: Slack for team, email for important, PagerDuty for urgent. Route based on severity. Include context (metric value, trend) in alert.

883. Incident management: respond to incidents systematically. Severity levels (SEV1 critical, SEV2 major, SEV3 minor). War room, communication, root cause, follow-up.

884. Post-mortems: after incidents, analyze what happened. Timeline, root cause, contributing factors, action items. Blameless: focus on systems, not people.

885. On-call expectations: primary on-call responds first, escalates if needed. Clear escalation paths, predefined business hours. Compensation for on-call time.

886. Runbooks: documented procedures for common incidents. Troubleshooting steps, escalation contacts, rollback procedures. Available when needed, regularly updated.

887. Status pages: communicate outages to customers. Expected duration, impact, mitigation steps. Reduces support emails. Real-time updates build trust.

888. Monitoring checklist: disk space, memory, CPU, database connections, failed jobs, payment failures, external API availability. Application-specific metrics.

889. Alert routing: severity-based routing. Errors → Slack, critical → PagerDuty → SMS, warnings → email. Prevent on-call for non-urgent alerts.

890. Metric retention: keep metrics for historical analysis. 1-hour granularity for recent, 1-day for old. Storage limited; balance detail vs cost.

891. Anomaly detection: algorithms detect unusual patterns. Temperature (deviation from baseline), statistical methods, ML models. Reduces false positives vs fixed thresholds.

892. Baseline metrics: establish normal behavior for comparison. Peak hours vs off-peak, weekday vs weekend. Anomalies detected relative to baseline.

893. Capacity planning: forecast future capacity needs. Growth trends, peak load, resource costs. Plan infrastructure upgrades. Avoid surprises, improve budget accuracy.

894. Cost monitoring: track infrastructure costs. By service, by region, resource utilization. Identify waste, optimize allocation. Chargeback per team if multi-tenant.

895. Telemetry data retention: keep logs/metrics for compliance, debugging. Different retention per type: errors (1yr), normal logs (30 days). Regulatory requirements vary.

896. Privacy in logging: don't log PII (passwords, credit cards, emails). Mask or redact sensitive fields. GDPR compliance: avoid storing unnecessary data.

897. Security monitoring: log security events (login attempts, permission changes, data access). Alert on suspicious patterns (failed logins, unusual access).

898. Audit logging: immutable logs of sensitive operations. Separate from application logs. Required for compliance (HIPAA, SOC 2, PCI). Tamper-proof.

899. Compliance monitoring: track security controls. Automated checks for misconfigurations (database exposed, no encryption). Tools: Wiz, CloudSploit, Prowler.

900. Operational metrics: track operations (deployment frequency, lead time for changes, mean time to recovery, change failure rate). DevOps Four Golden Signals enable continuous improvement.

## Section 19: Advanced Security & Hardening (Questions 901-1000)

901. OWASP Top 10 update: regularly updated list of critical vulnerabilities. Current focus: injection, broken auth, sensitive data exposure, XXE, access control, misconfiguration, XSS, insecure deserialization, vulnerable dependencies, insufficient logging.

902. Dependency vulnerability scanning: `composer audit` checks for known CVEs. Tools: Snyk, Dependabot automatically check and create PRs. Regular scanning prevents supply chain attacks.

903. Software Bill of Materials (SBOM): list all dependencies with versions. CycloneDX, SPDX formats. Regulatory requirement emerging. Enables tracking vulnerabilities across components.

904. Supply chain security: verify package integrity. Use package manager checksums (composer.lock). Code signing verifies source. Protect against compromised dependencies.

905. Secrets in dependencies: scan dependencies for hardcoded secrets. Tools: TruffleHog, detect-secrets. Secrets in code/config are serious vulnerabilities.

906. Docker image scanning: scan container images for vulnerabilities. Registry scanning (Docker Hub, ECR), Trivy, Grype. Prevents vulnerable containers in production.

907. Kubernetes security: network policies restrict traffic, RBAC controls access, pod security policies enforce security requirements. Secure by default, explicitly allow access.

908. WAF (Web Application Firewall): filters HTTP traffic, blocks attacks. Rules for SQL injection, XSS, DDoS. Cloud-native (AWS WAF, Cloudflare) or on-premises (ModSecurity).

909. Rate limiting and throttling: prevent brute force, abuse, DoS. Per-IP, per-user, per-API-key rates. Exponential backoff for retries. Whitelists for trusted sources.

910. IP whitelisting: restrict access to known IPs. VPN endpoints, office IPs, deployment servers. More restrictive than rate limiting, better for sensitive endpoints.

911. Geofencing: block access from suspicious locations. If user normally in US, login from China triggers alert/challenge. Fraud detection using location patterns.

912. Device fingerprinting: identify devices by characteristics (user-agent, IP, timezone). Detects suspicious devices accessing account. May require additional verification.

913. Passwordless authentication: remove password burden. Options: magic links (email), biometrics (fingerprint), hardware keys (security keys). More secure, better UX.

914. Hardware security keys: physical devices for auth (YubiKey). FIDO2 standard. Cannot be phished, very secure. Cost/user experience trade-off.

915. Biometric authentication: fingerprint, face recognition. Device handles biometric, device-specific. Good security + UX. Regulatory concerns (privacy, accuracy bias).

916. Auth0/Okta: managed identity providers. Handle authentication, MFA, social logins. Reduces implementation burden, leverages expertise. Cost per user.

917. OAuth2.0 implementation: delegated access for third-party apps. User authorizes app, gets token, app uses token to access resources. Standard protocol, multiple providers.

918. SAML: enterprise single sign-on. Provider authenticates user, application trusts provider. Common in corporate environments. Complex protocol vs OAuth2.

919. OpenID Connect: OAuth2 + identity layer. Combines auth and authorization. Modern alternative to SAML. Simpler, more flexible than SAML.

920. Zero Trust Security: assume breach, verify every access. Never trust user location or device. Explicit auth for all access, monitoring, principle of least privilege. Modern security model.

921. Defense in depth: multiple layers of security. Firewall, WAF, application validation, database encryption, data access controls. No single point of failure.

922. Principle of least privilege: users/processes get minimum permissions needed. Service account only for required tables, customer sees only their data. Limits damage from compromise.

923. Role-based access control (RBAC): users assigned roles, roles have permissions. Admin, editor, viewer roles. Simpler than attribute-based access control (ABAC).

924. Attribute-based access control (ABAC): fine-grained permissions based on attributes. User.department, resource.sensitivity, action.type. Complex but flexible.

925. API security best practices: validate all input, authenticate all endpoints, rate limit, log accesses, encrypt sensitive data, use HTTPS, manage API keys carefully.

926. Token expiration: short-lived tokens (15 min) + refresh tokens (1 week). Access token for requests, refresh token to get new access token. Balances convenience and security.

927. Refresh token rotation: issue new refresh token with each use, invalidate old one. Detects token compromise (old token reused). Prevents infinite access.

928. Token revocation: invalidate tokens before expiry if needed. Store revoked token IDs in blacklist (Redis), check before accepting. Prevents compromised tokens.

929. CORS security: specify allowed origins, methods, headers. `Access-Control-Allow-Origin: https://trusted.example.com` - only trusts specific origin. Never use `*` in production.

930. CORS credentials: `credentials: 'include'` in requests sends cookies. CORS must explicitly allow: `Access-Control-Allow-Credentials: true`. Prevents credential theft.

931. Subresource Integrity (SRI): verify script/stylesheet hash matches expected. `<script src="..." integrity="sha384-hash"></script>`. Detects CDN compromise.

932. Certificate pinning: pin expected TLS certificate in app. Prevents MITM even if CA compromised. Mobile apps commonly pin certificates.

933. Salting passwords: random salt prepended to password before hashing. Prevents rainbow tables, same password produces different hash. bcrypt handles salting automatically.

934. Password hashing algorithm: bcrypt best for passwords (slow, resistant to GPU attacks). Argon2 even better. Never use MD5/SHA1 for passwords. Cost factor controls speed.

935. Peppering passwords: secret pepper added to all passwords. Different from salt (salt is unique per user). Not commonly done, adds complexity without much benefit.

936. Password reset security: tokens with short expiry (15 min), one-time use, unique per request. Email reset link includes token. Verify token before allowing reset.

937. Email verification: verify user owns email before allowing login. Send verification email with link/code. Prevents account takeover, catches typos.

938. Account lockout: lock account after N failed login attempts. Temporary lock (5-30 min) or permanent until admin unlock. Prevents brute force but enables DOS.

939. Behavioral analysis: detect anomalies in user behavior. Unusual login time, location, device. Challenge or block suspicious activity. ML models improve accuracy.

940. Data privacy: identify PII, encrypt, limit access, retention policies. GDPR (EU), CCPA (California), others regulate. Compliance required for legal operations.

941. Data deletion: right to be forgotten (GDPR). Delete all personal data on request. Challenging: backups, logging, data copies in third-party systems.

942. Encryption key management: secure key storage, rotation, access control. HSM (Hardware Security Module) for critical keys. KMS (Key Management Service) for cloud.

943. Encryption at rest: database encryption, file encryption, backup encryption. Protects against physical theft, unauthorized access. Managed keys or BYOK (Bring Your Own Key).

944. Encryption in transit: TLS 1.2+ only, disable older protocols (SSL, TLS 1.0, 1.1). PFS (Perfect Forward Secrecy) resists future key compromise. Check ciphersuites.

945. Secrets rotation: periodically change API keys, passwords, certificates. Automated rotation for credentials. Manual for high-impact secrets. Enables detecting compromise earlier.

946. Audit logs: immutable logs of sensitive operations. What happened, who did it, when, from where. Reviewed for suspicious patterns. Regulatory compliance requires.

947. Incident response playbooks: documented procedures for different incident types. Data breach, ransomware, DDoS, insider threat. Speeds response, reduces improvisation.

948. Penetration testing: hire security professionals to attack systems. Find vulnerabilities before attackers. Regular testing (annually), after major changes.

949. Security awareness training: educate team on threats, secure practices. Phishing simulations, password hygiene, social engineering. Human security important as technical.

950. Vulnerability disclosure program: responsible way for researchers to report vulnerabilities. Coordination helps fix before public disclosure. HackerOne, Bugcrowd manage programs.

951. Secure coding training: teach team secure practices. Input validation, output encoding, access control, crypto. Regular training keeps knowledge current.

952. Code review security focus: reviewers check for security issues (injection, auth flaws, hard-coded secrets). Security SIEM (tool) assists, catches common issues.

953. Static analysis security testing (SAST): automated code analysis finds vulnerabilities. Tools: SonarQube, Checkmarx, Snyk. Finds common patterns but false positives possible.

954. Dynamic analysis security testing (DAST): test running application for vulnerabilities. Simulates attacks: SQL injection, XSS, CSRF. Tools: Burp, OWASP ZAP.

955. Interactive application security testing (IAST): hybrid of SAST and DAST. Instruments running application, detects vulnerabilities at runtime. More accurate than SAST/DAST alone.

956. Security scorecard: measure security posture. Vulnerability count, coverage percentage, incidents. Trends show improvement/degradation. Motivates investment.

957. Bug bounty programs: reward researchers for finding vulnerabilities. Incentivizes responsible disclosure, finds issues before public. Cost depends on scope/severity.

958. Threat modeling: identify potential threats, plan defenses. STRIDE methodology (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege).

959. Attack trees: visualize attack paths. Root: attacker goal. Leaves: individual attacks. Shows where defenses needed. Helps prioritize security investments.

960. Security by design: build security into systems from start, not afterwards. Threat modeling early, security requirements, secure architecture. Less costly than retrofitting.

961. Security testing in SDLC: integrate security testing into development pipeline. SAST in code review, DAST in testing, manual penetration testing. Continuous security.

962. Third-party risk management: vendors access data/systems. Assess vendor security: SOC 2 certification, breach history, incident response, contracts. Ongoing monitoring.

963. Acceptable use policies: define allowed system use. Prevent unauthorized access, data theft, illegal activity. Communicate to all users, enforce consequences.

964. Incident classification: severity levels (SEV1 critical, SEV2 major, SEV3 minor, SEV4 info). Response time and resources allocated per level. Common definition enables understanding.

965. MTTR (Mean Time To Repair): average time to fix incidents. Metric for operational excellence. Improvements: better monitoring (faster detection), runbooks (faster resolution), automation.

966. MTTD (Mean Time To Detect): average time to detect incident. Better: proactive monitoring, anomaly detection, security monitoring. Faster detection = less damage.

967. RTO/RPO: Recovery Time Objective (how fast to recover), Recovery Point Objective (how much data tolerable to lose). Defined per system criticality. Drives backup/replication strategy.

968. Business continuity planning: prepare for disasters. Backup systems, alternate locations, communication plans. Regular drills validate plans, identify gaps.

969. Disaster recovery testing: simulate disasters, practice recovery. Document results, improve procedures. Discover issues in controlled environment before real disaster.

970. Insurance for cyber incidents: cyber liability insurance covers incident costs (investigation, notification, recovery, lawsuits). Deductible and coverage limits vary.

971. Vendor security requirements: contracts require security standards. SLA for security patches, incident response. Right to audit, access logs. Negotiate protections.

972. Data classification: categorize data by sensitivity. Public, internal, confidential, restricted. Determines controls: public accessible, restricted encrypted/restricted access.

973. Data residency requirements: data stored in specific location (country). GDPR (EU data in EU), healthcare (regulated locations). Impacts cloud provider choice, costs.

974. Compliance frameworks: standards for security controls. ISO 27001 (information security), SOC 2 (service controls), HIPAA (healthcare), PCI DSS (payments). Audit required for certification.

975. Continuous compliance: maintain controls between audits. Automated checks for misconfigurations. Regular testing. Evidence collection for audits.

976. Privacy by design: build privacy into systems. Data minimization (collect only necessary), encryption, access controls, retention policies. GDPR requirement, best practice.

977. Consent management: track user consent for data processing. Cookie consent, email preferences, privacy settings. Legally required in many jurisdictions.

978. Right to access: users can request copy of their data. API endpoint returns user data in exportable format (JSON, CSV). GDPR requirement, commonly implemented.

979. Right to deletion: users can request all personal data deleted. Challenge: backups, related data, legitimate retention needs. Processes to handle safely.

980. Data breach notification: legally required to notify users of breaches (usually 30-72 hours). Public disclosure, regulatory agencies, affected users. Document causes, improvements.

981. Regulatory updates: security/privacy regulations constantly change. Subscribe to regulatory updates, legal consultation. Change processes to stay compliant.

982. Cross-border data transfers: moving data internationally restricted in some regions. EU to US transfers restricted after GDPR. Data localization requirements in some countries.

983. Zero-knowledge architecture: service provider can't access user data. End-to-end encryption, keys held by user. Example: ProtonMail encrypted email. Privacy-focused.

984. Anonymous data: data stripped of identifiers. More lenient regulation, but re-identification possible. Must document anonymization method.

985. Pseudonymization: replace identifiers with pseudonyms. Reversible (bad), irreversible (good). Reduces risk, eases regulations.

986. Security awareness metrics: track phishing click-through rates, training completion, incident reports. Trends indicate awareness level. Target: high reporting, low click-through.

987. Security culture: team values security, takes responsibility, reports issues. Leadership support essential. Reward proactive security contributions.

988. Security escalation procedures: clear procedures for reporting security issues. Do not publish before responsible disclosure period (usually 90 days). Protection for security researchers.

989. Vulnerability management: track identified vulnerabilities, prioritize by severity/exploitability, fix, verify. Process prevents accumulation of unpatched issues.

990. Metrics-driven security: measure security metrics, set targets, track progress. Vulnerability count, mean time to patch, breach incidents, etc. Data drives decision-making.

991. Red teaming: independent team simulates attackers. Real-world attacks, physical and cyber. Reveals gaps that technical controls alone miss.

992. Blue teaming: defensive team (blue) vs red team (red). Exercises build team's response capabilities. Coordination with red team improves defenses.

993. Purple teaming: collaborative red and blue. Continuous feedback loop, lessons improve both attack and defense. Better than separate exercises.

994. Security tool stack: SIEM (Splunk, ELK) aggregates logs. EDR (endpoint detection) finds compromises. SOAR (security orchestration) automates response. Choose tools that integrate.

995. Automation for security: automate response to common incidents. Isolate infected machines, block malicious IPs, disable accounts. Faster response reduces damage.

996. Security roadmap: multi-year plan for security improvements. Prioritize based on risk, dependencies, budget. Share with leadership, get buy-in.

997. Executive reporting: communicate security status to leadership. Risk dashboard, trends, budget needs. Non-technical language, business impact.

998. External compliance audit: third-party auditor verifies compliance. SOC 2 Type II shows controls effective over time. Quarterly or annual frequency.

999. Vendor risk scorecard: rate vendors on security. Code review quality, incident response, compliance certifications, track record. Updates quarterly based on new info.

1000. Security evolution: threats constantly evolve, defenses must adapt. Stay current with threat landscape, emerging vulnerabilities, new techniques. Continuous learning culture essential for security.
