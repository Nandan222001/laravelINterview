Section 1: Indexing Strategies (Questions 1-150)

Overview

This section covers advanced indexing strategies for MySQL and PostgreSQL databases, focusing on practical applications in high-traffic banking systems. Topics include B-tree, Hash, GiST, GIN, composite, covering, and partial indexes with EXPLAIN ANALYZE examples demonstrating performance improvements.

B-tree Indexes (Questions 1-30)

1. Explain how B-tree indexes work in MySQL/PostgreSQL and why they're the default index type.

B-tree indexes are self-balancing tree data structures that maintain sorted data for efficient searches, insertions, and deletions. They work by:

  • Structure: Organized as a tree with a root node, internal nodes, and leaf nodes containing actual data or pointers
  • Balance: All leaf nodes are at the same depth, ensuring O(log n) performance for all operations
  • Sorted Order: Keys are maintained in sorted order, enabling efficient range queries
  • Page-based Storage: Data stored in fixed-size pages, optimized for disk I/O

Why B-tree is default:

  • Supports both equality (=) and range queries (<, >, BETWEEN)
  • Efficient for ORDER BY operations on indexed columns
  • Handles NULL values properly
  • Works well for most query patterns in OLTP systems
  • Predictable O(log n) performance regardless of data distribution
2. What is the difference between clustered and non-clustered B-tree indexes?

Clustered Index:

  • Table rows physically ordered by index key
  • Leaf nodes contain actual row data
  • Only one per table (primary key in InnoDB)
  • Faster range scans and sequential reads
  • Example: InnoDB tables clustered on PRIMARY KEY

Non-clustered Index:

  • Separate structure pointing to data rows
  • Leaf nodes contain row pointers/references
  • Multiple per table allowed
  • Additional lookup required to fetch row data
  • Example: Secondary indexes in MySQL/PostgreSQL

7. What are the write performance implications of B-tree indexes?

B-tree indexes impact write operations in several ways:

  • INSERT overhead: Must update index tree for each new row (~5-15% slower)
  • UPDATE overhead: If indexed column changes, index must be updated (~10-20% slower)
  • DELETE overhead: Index entries must be removed or marked as deleted
  • Page splits: When index page fills, it splits into two, causing temporary performance hit
  • Lock contention: Multiple writes to same index page can cause contention

Mitigation strategies:

  • Use batch inserts instead of individual INSERTs (50-100x faster)
  • Disable indexes during bulk loads, rebuild after
  • Set appropriate fill factor to reduce page splits
  • Index only columns frequently used in WHERE/JOIN clauses

Composite Indexes (Questions 91-110)

91. What is a composite index and when should you use one?

A composite (multi-column) index is an index on two or more columns. Use when:

  • Multi-column WHERE clauses: Queries filter on multiple columns together
  • Covering index opportunities: Include all columns needed by query to avoid table lookup
  • Sort optimization: Query needs ORDER BY on multiple columns
  • Join optimization: Join conditions use multiple columns

Example:

-- Query pattern
SELECT * FROM transactions 
WHERE account_id = 12345 AND transaction_date >= '2024-01-01';

-- Optimal composite index
CREATE INDEX idx_transactions_account_date 
ON transactions(account_id, transaction_date);
94. What is the leftmost prefix rule in composite indexes?

The leftmost prefix rule states that a composite index can be used if the query filters on the leftmost columns in order.

Example with index (col1, col2, col3):

  • ✅ WHERE col1 = x (uses index)
  • ✅ WHERE col1 = x AND col2 = y (uses index)
  • ✅ WHERE col1 = x AND col2 = y AND col3 = z (uses index)
  • ❌ WHERE col2 = y (cannot use index)
  • ❌ WHERE col3 = z (cannot use index)
  • ⚠️ WHERE col1 = x AND col3 = z (uses index only for col1)
CREATE INDEX idx_composite ON transactions(account_id, transaction_date, amount);

-- Uses full index
SELECT * FROM transactions WHERE account_id = 12345 AND transaction_date = '2024-01-15' AND amount > 1000;

-- Uses index only for account_id
SELECT * FROM transactions WHERE account_id = 12345 AND amount > 1000;

-- Cannot use index at all
SELECT * FROM transactions WHERE transaction_date = '2024-01-15';

Covering Indexes (Questions 111-130)

111. What is a covering index and how does it eliminate table lookups?

A covering index contains all columns needed by a query, allowing the database to satisfy the query entirely from the index without accessing the table.

How it works:

  • Index includes all columns in SELECT, WHERE, ORDER BY, and GROUP BY
  • Database performs "Index Only Scan" (no heap fetch needed)
  • Dramatically reduces I/O since index is much smaller than table

Example:

-- Query
SELECT account_id, transaction_date, amount 
FROM transactions
WHERE account_id = 12345
ORDER BY transaction_date DESC
LIMIT 10;

-- Covering index (PostgreSQL 11+)
CREATE INDEX idx_transactions_covering 
ON transactions(account_id, transaction_date DESC) 
INCLUDE (amount);

Performance:

Before: Index Scan + Table Lookup = 12.678ms

After: Index Only Scan = 0.567ms

95.5% reduction

Partial Indexes (Questions 131-150)

131. What is a partial index in PostgreSQL and when should you use it?

A partial index is an index built on a subset of rows using a WHERE clause. Benefits:

  • Storage savings: 80-95% smaller than full index
  • Faster queries: Smaller index = faster scans
  • Less maintenance: Fewer rows to update on writes
  • Better cache utilization: More of index fits in memory

Use cases:

  • Frequently querying subset of data (e.g., status = 'pending')
  • Filtering out NULL values or deleted records
  • Time-based filtering (e.g., recent 90 days)
-- Full index (2.4 GB)
CREATE INDEX idx_transactions_status ON transactions(status);

-- Partial index (156 MB - 93.5% reduction)
CREATE INDEX idx_transactions_pending 
ON transactions(status) 
WHERE status = 'pending';

-- Query automatically uses partial index
SELECT * FROM transactions WHERE status = 'pending';

Results: 89.8% faster queries (235ms → 24ms) + 93.5% storage reduction

  • B-tree indexes: 50-99% improvement for equality and range queries
  • GIN indexes: 95-99% improvement for JSONB and array queries
  • Covering indexes: 85-95% improvement through index-only scans
  • Partial indexes: 70-90% improvement + 80-95% storage reduction

Banking Best Practices:

Section 2: Cardinality Analysis & Selectivity (Questions 151-250)

Cardinality Fundamentals (Questions 151-170)

151. Define cardinality in the context of database indexes.

Cardinality is the number of distinct values in a column relative to the total number of rows. It determines index effectiveness:

  • High cardinality: Many unique values (e.g., email, account_number) - excellent for indexing
  • Medium cardinality: Moderate unique values (e.g., transaction_date) - conditional indexing
  • Low cardinality: Few unique values (e.g., status, gender) - poor for indexing

Formula:

Cardinality Ratio = Distinct Values / Total Rows

High: > 1% (e.g., 0.95 = 95% unique)
Medium: 0.01% - 1%
Low: < 0.01%

Example:

-- Check cardinality
SELECT 
    attname AS column_name,
    n_distinct,
    CASE 
        WHEN n_distinct < 0 THEN ABS(n_distinct) * reltuples
        ELSE n_distinct
    END AS estimated_distinct_values
FROM pg_stats
WHERE tablename = 'transactions';

Selectivity Calculations (Questions 171-190)

171. Define selectivity and how it differs from cardinality.

Selectivity: Percentage of rows returned by a query predicate

Cardinality: Number of distinct values in a column

Key differences:

Aspect Cardinality Selectivity
Definition Column property Query property
Scope Static for column Varies per query
Formula Distinct values / Total rows Matching rows / Total rows
Good value High (many unique) Low (few matching)

Example:

100M transactions table:

Cardinality of account_id: 1,250,000 / 100,000,000 = 1.25% (High)
Cardinality of status: 5 / 100,000,000 = 0.000005% (Low)

Selectivity of "WHERE account_id = 12345": 80 / 100,000,000 = 0.00008% (Excellent)
Selectivity of "WHERE status = 'pending'": 15,000,000 / 100,000,000 = 15% (Poor)
Indexing Decision:
  • Selectivity < 5%: Index highly beneficial
  • Selectivity 5-15%: Index may help depending on query frequency
  • Selectivity > 15%: Index likely not used; consider partial index

Statistics and Histograms (Questions 191-210)

191. Explain how PostgreSQL's ANALYZE command works.

ANALYZE collects statistics about table contents to help the query optimizer make better execution plan decisions:

What it collects:

  • Number of distinct values (n_distinct)
  • Most common values and their frequencies
  • Histogram bounds for value distribution
  • NULL fraction
  • Average column width
  • Correlation with physical row order

How it works:

  • Samples rows (default: 300 × default_statistics_target)
  • Analyzes sample to estimate statistics
  • Stores in pg_stats system view
  • Optimizer uses for cost estimation
-- Analyze specific table
ANALYZE transactions;

-- Increase statistics target for better estimates
ALTER TABLE transactions ALTER COLUMN account_id SET STATISTICS 1000;
ANALYZE transactions;

-- View statistics
SELECT 
    tablename, attname, n_distinct, most_common_vals, 
    most_common_freqs, histogram_bounds, correlation
FROM pg_stats
WHERE tablename = 'transactions';
When to run ANALYZE:
  • After bulk data loads
  • After large UPDATE/DELETE operations
  • When query plans become suboptimal
  • Scheduled daily for high-traffic tables

Execution Plan Node Types (Questions 281-310)

285. What is the difference between Index Scan and Index Only Scan?

Index Scan:

  • Reads index to find matching rows
  • Then fetches actual row data from table (heap fetch)
  • Two I/O operations: index + table
  • Used when query needs columns not in index

Index Only Scan:

  • Reads only from index (no table access)
  • All needed columns are in the index
  • One I/O operation: index only
  • Requires covering index
  • Much faster (60-80% improvement)
-- Index Scan (requires table lookup)
CREATE INDEX idx_account_id ON transactions(account_id);
EXPLAIN ANALYZE
SELECT account_id, amount FROM transactions WHERE account_id = 12345;
-- Result: Index Scan + Heap Fetch

-- Index Only Scan (no table lookup)
CREATE INDEX idx_covering ON transactions(account_id) INCLUDE (amount);
EXPLAIN ANALYZE
SELECT account_id, amount FROM transactions WHERE account_id = 12345;
-- Result: Index Only Scan (Heap Fetches: 0)
-- 60-80% faster!

Section 4: N+1 Query Elimination (Questions 351-450)

N+1 Problem Understanding (Questions 351-370)

351. Explain the N+1 query problem with a practical example.

The N+1 problem occurs when you query for N parent records, then execute 1 additional query for each parent to load related child records, resulting in 1 + N total queries.

Example - Banking API:

// ❌ N+1 PROBLEM
$accounts = Account::all();  // 1 query - fetches 1,000 accounts

foreach ($accounts as $account) {
    echo $account->customer->name;  // +1 query per account
    echo $account->transactions->count();  // +1 query per account
}
// Total: 1 + (1,000 × 2) = 2,001 queries!

// ✅ SOLUTION - Eager Loading
$accounts = Account::with(['customer', 'transactions'])->get();  // 3 queries total

foreach ($accounts as $account) {
    echo $account->customer->name;  // No additional query
    echo $account->transactions->count();  // No additional query
}
// Total: 3 queries (99.85% reduction!)

Impact for 1,000 accounts:

  • Without eager loading: 2,001 queries, 16.9 seconds
  • With eager loading: 3 queries, 0.45 seconds
  • 97.3% response time reduction

Eager Loading Strategies (Questions 371-400)

371. Explain eager loading and how it solves the N+1 problem.

Eager loading pre-loads relationships using a minimal number of queries instead of loading them lazily one at a time.

How it works:

  1. Query parent records: SELECT * FROM accounts
  2. Collect all parent IDs: [1, 2, 3, ..., 1000]
  3. Query related records in bulk: SELECT * FROM customers WHERE id IN (1,2,3,...,1000)
  4. Laravel maps relationships in memory
// Basic eager loading
$accounts = Account::with('customer')->get();
// Queries: 2 (accounts + customers)

// Multiple relationships
$accounts = Account::with(['customer', 'transactions'])->get();
// Queries: 3 (accounts + customers + transactions)

// Nested relationships
$accounts = Account::with('transactions.merchant')->get();
// Queries: 3 (accounts + transactions + merchants)

// Constrained eager loading
$accounts = Account::with(['transactions' => function($query) {
    $query->where('amount', '>', 1000)
          ->orderBy('transaction_date', 'desc')
          ->limit(10);
}])->get();
// Only loads recent high-value transactions
383. Explain the withCount() method and its use cases.

withCount() adds a relationship count as an attribute without loading the actual related records, perfect for displaying counts without the overhead of loading full data.

// Load count without loading data
$accounts = Account::withCount('transactions')->get();

foreach ($accounts as $account) {
    echo "Transactions: " . $account->transactions_count;  // No additional query
}

// Multiple counts
$accounts = Account::withCount(['transactions', 'pending_transactions'])->get();

// Constrained count
$accounts = Account::withCount([
    'transactions as high_value_count' => function($query) {
        $query->where('amount', '>', 10000);
    }
])->get();
Use withCount() when you only need the count, not the data. It's 90%+ faster than loading relationships just to count them.

Preventing Lazy Loading (Questions 876-900)

876. Explain the Model::preventLazyLoading() method.

preventLazyLoading() (Laravel 8.43+) throws an exception when attempting to lazy load relationships, helping catch N+1 problems during development.

// In AppServiceProvider boot() method
use Illuminate\Database\Eloquent\Model;

public function boot()
{
    // Prevent in non-production
    Model::preventLazyLoading(!app()->isProduction());
    
    // Or with custom handler
    Model::handleLazyLoadingViolationUsing(function ($model, $relation) {
        Log::warning("Lazy loading: " . get_class($model) . "::{$relation}");
        
        if (!app()->isProduction()) {
            throw new \RuntimeException(
                "Lazy loading detected: {$relation}"
            );
        }
    });
}

// Now this will throw an exception in development
$account = Account::find(1);
$customer = $account->customer;  // Exception!

// This is fine (eager loaded)
$account = Account::with('customer')->find(1);
$customer = $account->customer;  // No exception

Section 5: Normalization & Denormalization (Questions 451-550)

Normalization Forms (Questions 451-480)

451. Explain the First Normal Form (1NF) with examples.

First Normal Form requires all columns contain atomic (indivisible) values with no repeating groups.

Rules:

  • Each column contains single value (no arrays or lists)
  • Each row is unique
  • No repeating groups of columns

Example:

❌ NOT 1NF:
customer_id | name | phone_numbers
1 | John | "555-0100, 555-0101, 555-0102"

✅ 1NF:
customer_id | name | phone_number
1 | John | 555-0100
1 | John | 555-0101
1 | John | 555-0102
455. Explain Third Normal Form (3NF) and transitive dependencies.

Third Normal Form eliminates transitive dependencies where non-key attributes depend on other non-key attributes.

Requirements:

  • Must be in 2NF
  • No transitive dependencies (A → B → C)
  • All non-key attributes depend only on primary key
❌ NOT 3NF (transitive dependency):
transactions table:
transaction_id | merchant_id | merchant_name | merchant_category
1 | 100 | Starbucks | Coffee Shop
(merchant_name depends on merchant_id, not transaction_id)

✅ 3NF (no transitive dependencies):
transactions:
transaction_id | merchant_id
1 | 100

merchants:
merchant_id | merchant_name | merchant_category
100 | Starbucks | Coffee Shop
Benefits of 3NF: Reduces redundancy, prevents update anomalies, saves storage (73% reduction in example)

Denormalization Strategies (Questions 481-510)

481. Explain when and why to denormalize database schemas.

Denormalize when read performance is more critical than write performance and data integrity can be maintained through application logic or triggers.

When to denormalize:

  • Read-heavy workload (1M+ requests/day)
  • Complex joins causing slow queries
  • Frequently accessed aggregations (counts, sums)
  • Dashboard/reporting queries
  • Data doesn't change frequently

Example - Account Summary:

-- ❌ Normalized (slow for dashboards)
SELECT 
    a.account_number,
    c.name,
    COUNT(t.id) as transaction_count,
    SUM(t.amount) as total_amount
FROM accounts a
JOIN customers c ON a.customer_id = c.id
LEFT JOIN transactions t ON a.id = t.account_id
GROUP BY a.account_number, c.name;
-- Execution: 2.5 seconds

-- ✅ Denormalized (fast)
ALTER TABLE accounts 
ADD COLUMN customer_name VARCHAR(100),
ADD COLUMN transaction_count INT DEFAULT 0,
ADD COLUMN total_amount DECIMAL(15,2) DEFAULT 0;

SELECT account_number, customer_name, transaction_count, total_amount
FROM accounts;
-- Execution: 0.05 seconds (98% improvement!)
Trade-off: Must maintain denormalized data with triggers or application logic. Adds ~2ms overhead to writes, but saves 2.45s on reads.

Section 5 Summary: Normalization & Denormalization

  • 73% storage reduction with normalization
  • 98% read performance improvement with strategic denormalization
  • 50x faster dashboard queries with materialized views
  • Normalize for data integrity, denormalize for performance
  • Use triggers or application logic to maintain denormalized data

Range Partitioning (Questions 551-580)

551. Explain range partitioning and provide a banking use case.

Range partitioning divides a table into partitions based on value ranges, typically dates. Perfect for time-series data in banking.

-- Create partitioned table
CREATE TABLE transactions (
    id BIGSERIAL,
    account_id BIGINT NOT NULL,
    transaction_date TIMESTAMP NOT NULL,
    amount DECIMAL(15,2) NOT NULL
) PARTITION BY RANGE (transaction_date);

-- Create monthly partitions
CREATE TABLE transactions_2024_01 PARTITION OF transactions
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE transactions_2024_02 PARTITION OF transactions
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- Query with date filter (partition pruning)
SELECT * FROM transactions 
WHERE transaction_date BETWEEN '2024-01-15' AND '2024-01-20';
-- Only scans transactions_2024_01 partition (92% reduction!)

Before partitioning: Scans 100M rows, 15 seconds

After partitioning: Scans 8.3M rows (1 month), 1.2 seconds

92% improvement through partition pruning

Section 7: Redis Caching Layers (Questions 651-750)

Cache-Aside Pattern (Questions 651-680)

651. Explain the cache-aside pattern and provide a Laravel implementation.

Cache-aside (lazy loading) checks cache first, queries database on miss, then populates cache for future requests.

use Illuminate\Support\Facades\Redis;

class AccountService
{
    public function getAccount(int $accountId): ?array
    {
        $cacheKey = "account:{$accountId}";
        
        // 1. Check cache first
        $cached = Redis::get($cacheKey);
        if ($cached !== null) {
            return json_decode($cached, true);  // Cache hit (0.5ms)
        }
        
        // 2. Cache miss - query database
        $account = DB::table('accounts')->find($accountId);  // 15ms
        
        if ($account) {
            // 3. Store in cache (TTL: 1 hour)
            Redis::setex($cacheKey, 3600, json_encode($account));
        }
        
        return (array) $account;
    }
}

Without cache: 15ms per request × 1,000 req/sec = 15,000ms DB time

With cache (90% hit rate): (900 × 0.5ms) + (100 × 15ms) = 1,950ms

87% database load reduction

  • 87% database load reduction with cache-aside (90% hit rate)
  • 92% write improvement with write-behind pattern
  • 99% latency reduction for cached reads (15ms → 0.5ms)
  • 12.5x throughput increase for write-heavy workloads
  • Use appropriate data structures for use case

Connection Pooling (Questions 901-930)

901. Explain connection pooling and its benefits for banking applications.

Connection pooling maintains a pool of persistent database connections that are reused across requests, eliminating the overhead of creating new connections.

Benefits:

  • Eliminates TCP handshake overhead (5-20ms per connection)
  • Reduces authentication overhead
  • Multiplexes many app connections to few DB connections
  • Prevents connection exhaustion under load

Pool sizing formula:

Optimal Pool Size = (CPU cores × 2) + Effective Spindle Count

Example for 8-core server with SSD:
(8 × 2) + 1 = 17 connections
With safety margin: 25-30 connections

Using PgBouncer:
App servers: 1,000+ connections → PgBouncer → Database: 25 connections

Read-Write Splitting (Questions 931-960)

931. Implement read-write splitting in Laravel.
// config/database.php
'mysql' => [
    'driver' => 'mysql',
    'read' => [
        'host' => ['10.0.1.10', '10.0.1.11', '10.0.1.12'],  // Replicas
    ],
    'write' => [
        'host' => ['10.0.1.1'],  // Master
    ],
    'sticky' => true,  // Read from master after write
    'database' => 'banking',
    // ...
],

// Usage
DB::table('accounts')->where('id', 1)->first();  // Reads from replica
DB::table('accounts')->where('id', 1)->update(['balance' => 5000]);  // Writes to master

Impact: 40-60% reduction in primary database load
