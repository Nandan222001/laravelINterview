2. Keys (Primary, Unique, Foreign, Composite)

Primary Key

Uniquely identifies each record in a table. Only one primary key per table.

-- Single column primary key
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR(100),
    name VARCHAR(100)
);

-- Alternative syntax
CREATE TABLE users (
    id INT AUTO_INCREMENT,
    email VARCHAR(100),
    name VARCHAR(100),
    PRIMARY KEY (id)
);

-- Add primary key to existing table
ALTER TABLE users ADD PRIMARY KEY (id);

Composite Primary Key

Primary key consisting of multiple columns combined.

-- Composite primary key example
CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT,
    price DECIMAL(10,2),
    PRIMARY KEY (order_id, product_id)
);

-- Student course enrollment
CREATE TABLE enrollments (
    student_id INT,
    course_id INT,
    semester VARCHAR(20),
    grade CHAR(2),
    PRIMARY KEY (student_id, course_id, semester)
);

Unique Key

Ensures all values in a column are different. Multiple unique keys allowed per table.

-- Single unique constraint
CREATE TABLE users (
    id INT PRIMARY KEY,
    email VARCHAR(100) UNIQUE,
    username VARCHAR(50) UNIQUE,
    name VARCHAR(100)
);

-- Named unique constraint
CREATE TABLE users (
    id INT PRIMARY KEY,
    email VARCHAR(100),
    username VARCHAR(50),
    CONSTRAINT uk_email UNIQUE (email),
    CONSTRAINT uk_username UNIQUE (username)
);

-- Composite unique constraint
CREATE TABLE products (
    id INT PRIMARY KEY,
    sku VARCHAR(50),
    warehouse_id INT,
    UNIQUE (sku, warehouse_id)
);

-- Add unique constraint to existing table
ALTER TABLE users ADD UNIQUE (email);

Foreign Key

Links two tables together, ensuring referential integrity.

-- Basic foreign key
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    total DECIMAL(10,2),
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- Foreign key with named constraint
CREATE TABLE posts (
    id INT PRIMARY KEY,
    user_id INT,
    title VARCHAR(200),
    content TEXT,
    CONSTRAINT fk_posts_user FOREIGN KEY (user_id) 
        REFERENCES users(id)
);

-- Foreign key with CASCADE actions
CREATE TABLE comments (
    id INT PRIMARY KEY,
    post_id INT,
    user_id INT,
    content TEXT,
    FOREIGN KEY (post_id) REFERENCES posts(id)
        ON DELETE CASCADE
        ON UPDATE CASCADE,
    FOREIGN KEY (user_id) REFERENCES users(id)
        ON DELETE SET NULL
        ON UPDATE CASCADE
);

-- Add foreign key to existing table
ALTER TABLE orders 
ADD CONSTRAINT fk_orders_user 
FOREIGN KEY (user_id) REFERENCES users(id);

Foreign Key Actions

Action Description Use Case
CASCADE Automatically delete/update child rows when parent is deleted/updated Order items when order is deleted
SET NULL Set foreign key to NULL when parent is deleted Comments when user is deleted (keep comment, remove author)
RESTRICT Prevent deletion/update if child rows exist Prevent category deletion if products exist
NO ACTION Similar to RESTRICT (default in MySQL) Standard referential integrity
SET DEFAULT Set foreign key to default value Rarely used in MySQL
⚠️ Important: Foreign keys in InnoDB require that referenced columns have an index. InnoDB automatically creates indexes on foreign key columns if they don't exist.

3. Database Normalization

Normalization is the process of organizing data to reduce redundancy and improve data integrity.

First Normal Form (1NF)

Rules:

  • Each column contains atomic (indivisible) values
  • Each column contains values of a single type
  • Each column has a unique name
  • Order in which data is stored doesn't matter
-- ❌ VIOLATES 1NF (multiple values in one column)
CREATE TABLE students_bad (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    phone_numbers VARCHAR(200)  -- '555-1234, 555-5678, 555-9012'
);

-- ✅ FOLLOWS 1NF (atomic values)
CREATE TABLE students (
    id INT PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE student_phones (
    id INT AUTO_INCREMENT PRIMARY KEY,
    student_id INT,
    phone_number VARCHAR(20),
    phone_type ENUM('mobile', 'home', 'work'),
    FOREIGN KEY (student_id) REFERENCES students(id)
);

Second Normal Form (2NF)

Rules:

  • Must be in 1NF
  • All non-key columns must depend on the entire primary key (no partial dependencies)
  • Only applies to tables with composite primary keys
-- ❌ VIOLATES 2NF (partial dependency)
CREATE TABLE order_items_bad (
    order_id INT,
    product_id INT,
    product_name VARCHAR(100),  -- Depends only on product_id
    product_price DECIMAL(10,2), -- Depends only on product_id
    quantity INT,
    PRIMARY KEY (order_id, product_id)
);

-- ✅ FOLLOWS 2NF (no partial dependencies)
CREATE TABLE products (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    price DECIMAL(10,2)
);

CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT,
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);

Third Normal Form (3NF)

Rules:

  • Must be in 2NF
  • No transitive dependencies (non-key columns depend only on primary key)
-- ❌ VIOLATES 3NF (transitive dependency)
CREATE TABLE employees_bad (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    department_id INT,
    department_name VARCHAR(100),    -- Depends on department_id
    department_location VARCHAR(100) -- Depends on department_id
);

-- ✅ FOLLOWS 3NF (no transitive dependencies)
CREATE TABLE departments (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    location VARCHAR(100)
);

CREATE TABLE employees (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    department_id INT,
    FOREIGN KEY (department_id) REFERENCES departments(id)
);
💡 When to Normalize vs Denormalize: Normalize for data integrity and reducing redundancy. Denormalize selectively for read-heavy applications where performance is critical.

4. SQL Joins

INNER JOIN

Returns records that have matching values in both tables.

-- Basic INNER JOIN
SELECT 
    orders.id,
    orders.order_date,
    users.name AS customer_name,
    users.email
FROM orders
INNER JOIN users ON orders.user_id = users.id;

-- Multiple INNER JOINs
SELECT 
    o.id AS order_id,
    u.name AS customer,
    p.name AS product,
    oi.quantity,
    oi.price
FROM orders o
INNER JOIN users u ON o.user_id = u.id
INNER JOIN order_items oi ON o.id = oi.order_id
INNER JOIN products p ON oi.product_id = p.id
WHERE o.order_date >= '2024-01-01';

LEFT JOIN (LEFT OUTER JOIN)

Returns all records from the left table and matched records from the right table.

-- Get all users and their orders (including users with no orders)
SELECT 
    u.id,
    u.name,
    u.email,
    COUNT(o.id) AS total_orders,
    COALESCE(SUM(o.total), 0) AS total_spent
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name, u.email;

-- Find users who never placed an order
SELECT u.id, u.name, u.email
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;

SELF JOIN

A table joined with itself to compare rows within the same table.

-- Employee hierarchy (employee and their manager)
CREATE TABLE employees (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    manager_id INT,
    FOREIGN KEY (manager_id) REFERENCES employees(id)
);

SELECT 
    e.name AS employee,
    m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
✅ Join Performance Tips:
  • Always index foreign key columns used in joins
  • Use INNER JOIN when possible (faster than OUTER joins)
  • Filter data early with WHERE clauses before joining
  • Use EXPLAIN to analyze join performance

5. Indexes & Types

What is an Index?

An index is a data structure that improves the speed of data retrieval operations on a database table at the cost of additional writes and storage space.

Primary Index (Clustered Index)

-- Primary key automatically creates a clustered index in InnoDB
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR(100),
    name VARCHAR(100)
);

-- In InnoDB, the entire table is organized around the primary key
-- All data is stored in the primary key's B-Tree structure

Secondary Index (Non-Clustered Index)

-- Single column index
CREATE INDEX idx_email ON users(email);

-- Alternative syntax
ALTER TABLE users ADD INDEX idx_email (email);

-- Named index with multiple columns
CREATE INDEX idx_name_email ON users(name, email);

Unique Index

-- Ensures uniqueness and creates an index
CREATE UNIQUE INDEX idx_unique_email ON users(email);

-- Multiple columns
CREATE UNIQUE INDEX idx_unique_sku_warehouse 
ON products(sku, warehouse_id);

Composite Index (Multi-Column Index)

-- Index on multiple columns (order matters!)
CREATE INDEX idx_user_date ON orders(user_id, order_date);

-- This index helps queries like:
-- ✅ WHERE user_id = 1
-- ✅ WHERE user_id = 1 AND order_date > '2024-01-01'
-- ✅ WHERE user_id = 1 ORDER BY order_date
-- ❌ WHERE order_date > '2024-01-01' (doesn't use the index efficiently)

-- Leftmost prefix rule
CREATE INDEX idx_name_email_status ON users(name, email, status);
-- Can use index for:
-- name
-- name, email
-- name, email, status
-- But NOT for: email, status (without name)

Full-Text Index

-- Full-text index for text search
CREATE TABLE articles (
    id INT PRIMARY KEY,
    title VARCHAR(200),
    content TEXT,
    FULLTEXT INDEX idx_fulltext (title, content)
);

-- Full-text search queries
SELECT * FROM articles
WHERE MATCH(title, content) AGAINST('database optimization');

-- Boolean mode search
SELECT * FROM articles
WHERE MATCH(title, content) 
AGAINST('+mysql -postgresql' IN BOOLEAN MODE);

Covering Index

-- Covering index includes all columns needed by a query
-- Allows index-only scan without accessing the table
CREATE INDEX idx_covering ON orders(user_id, order_date, total, status);

-- This query can be satisfied entirely from the index
SELECT order_date, total, status
FROM orders
WHERE user_id = 123
ORDER BY order_date DESC;

Index Management

-- Show indexes on a table
SHOW INDEX FROM users;

-- Drop an index
DROP INDEX idx_email ON users;
ALTER TABLE users DROP INDEX idx_email;

-- Add index to existing table
ALTER TABLE users ADD INDEX idx_status (status);
⚠️ Index Trade-offs: While indexes speed up reads, they slow down writes (INSERT, UPDATE, DELETE) and consume disk space. Don't over-index!

6. Index Optimization

When to Create Indexes

  • WHERE clauses: Columns frequently used in WHERE conditions
  • JOIN conditions: Foreign key columns used in joins
  • ORDER BY: Columns used for sorting
  • GROUP BY: Columns used for grouping
  • DISTINCT: Columns used with DISTINCT

Composite Index Optimization

-- Order columns by selectivity (most selective first)
-- Rule: Equality > Range > Sort

-- ✅ GOOD: High selectivity column first
CREATE INDEX idx_optimized ON orders(
    user_id,        -- Equality (most selective)
    status,         -- Equality
    order_date      -- Range/Sort
);

-- Query that benefits:
SELECT * FROM orders
WHERE user_id = 123
    AND status = 'completed'
    AND order_date >= '2024-01-01'
ORDER BY order_date DESC;

Index Selectivity Analysis

-- Calculate index selectivity (higher is better)
-- Perfect selectivity = 1.0 (all unique values)
-- Poor selectivity = 0.01 (few unique values)

SELECT 
    COUNT(DISTINCT email) / COUNT(*) AS email_selectivity,
    COUNT(DISTINCT status) / COUNT(*) AS status_selectivity,
    COUNT(DISTINCT country) / COUNT(*) AS country_selectivity
FROM users;
✅ Index Optimization Best Practices:
  • Composite index order: Equality → Range → Sort
  • Cardinality: Index high-cardinality columns
  • Covering indexes: Include all query columns when possible
  • Monitor: Use EXPLAIN to verify index usage
  • Maintain: Run ANALYZE TABLE periodically

7. Triggers

Triggers are database operations that automatically execute in response to specific events on a table.

Trigger Syntax

CREATE TRIGGER trigger_name
{BEFORE | AFTER} {INSERT | UPDATE | DELETE}
ON table_name
FOR EACH ROW
BEGIN
    -- Trigger logic
END;

BEFORE INSERT Trigger

-- Auto-generate UUID before insert
DELIMITER //
CREATE TRIGGER before_user_insert
BEFORE INSERT ON users
FOR EACH ROW
BEGIN
    IF NEW.uuid IS NULL THEN
        SET NEW.uuid = UUID();
    END IF;
    
    -- Convert email to lowercase
    SET NEW.email = LOWER(NEW.email);
    
    -- Set default values
    IF NEW.created_at IS NULL THEN
        SET NEW.created_at = NOW();
    END IF;
END;//
DELIMITER ;

AFTER INSERT Trigger

-- Update user statistics after order is placed
DELIMITER //
CREATE TRIGGER after_order_insert
AFTER INSERT ON orders
FOR EACH ROW
BEGIN
    UPDATE user_statistics
    SET 
        total_orders = total_orders + 1,
        total_spent = total_spent + NEW.total,
        last_order_date = NEW.order_date
    WHERE user_id = NEW.user_id;
END;//
DELIMITER ;

BEFORE UPDATE Trigger

-- Track changes and update modified timestamp
DELIMITER //
CREATE TRIGGER before_user_update
BEFORE UPDATE ON users
FOR EACH ROW
BEGIN
    -- Always update the updated_at timestamp
    SET NEW.updated_at = NOW();
    
    -- Prevent email from being changed
    IF OLD.email != NEW.email THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Email cannot be changed';
    END IF;
END;//
DELIMITER ;

Trigger Management

-- Show all triggers
SHOW TRIGGERS;

-- Show triggers for specific table
SHOW TRIGGERS WHERE `Table` = 'users';

-- Drop trigger
DROP TRIGGER IF EXISTS trigger_name;
⚠️ Trigger Limitations & Cautions:
  • Triggers can't be directly called - they're automatic
  • Excessive trigger logic can slow down operations
  • Triggers don't fire for TRUNCATE TABLE
  • Be careful with cascading triggers

8. Views

A view is a virtual table based on the result of a SQL query. It contains rows and columns just like a real table but doesn't store data physically.

Creating Views

-- Simple view
CREATE VIEW active_users AS
SELECT id, name, email, created_at
FROM users
WHERE status = 'active';

-- Use the view
SELECT * FROM active_users;

-- View with JOIN
CREATE VIEW order_summary AS
SELECT 
    o.id AS order_id,
    o.order_date,
    u.name AS customer_name,
    u.email,
    COUNT(oi.id) AS item_count,
    SUM(oi.quantity * oi.price) AS total
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN order_items oi ON o.id = oi.order_id
GROUP BY o.id, o.order_date, u.name, u.email;

View Management

-- Show all views
SHOW FULL TABLES WHERE Table_type = 'VIEW';

-- Show view definition
SHOW CREATE VIEW view_name;

-- Replace existing view
CREATE OR REPLACE VIEW active_users AS
SELECT id, name, email, status, created_at
FROM users
WHERE status = 'active';

-- Drop view
DROP VIEW IF EXISTS view_name;
💡 View vs Table: Views don't store data. They execute the underlying query each time they're accessed. Use views for simplification and security, not performance optimization.

9. Stored Procedures

Stored procedures are reusable SQL code blocks stored in the database. They can accept parameters, contain logic, and return results.

Basic Stored Procedure

-- Simple procedure without parameters
DELIMITER //
CREATE PROCEDURE get_all_users()
BEGIN
    SELECT id, name, email FROM users;
END;//
DELIMITER ;

-- Call procedure
CALL get_all_users();

Procedures with IN Parameters

-- Procedure with input parameters
DELIMITER //
CREATE PROCEDURE get_user_orders(
    IN p_user_id INT,
    IN p_limit INT
)
BEGIN
    SELECT 
        id,
        order_date,
        total,
        status
    FROM orders
    WHERE user_id = p_user_id
    ORDER BY order_date DESC
    LIMIT p_limit;
END;//
DELIMITER ;

-- Call with parameters
CALL get_user_orders(123, 10);

Procedures with OUT Parameters

-- Procedure with output parameters
DELIMITER //
CREATE PROCEDURE get_user_stats(
    IN p_user_id INT,
    OUT p_order_count INT,
    OUT p_total_spent DECIMAL(10,2)
)
BEGIN
    SELECT 
        COUNT(*),
        COALESCE(SUM(total), 0)
    INTO p_order_count, p_total_spent
    FROM orders
    WHERE user_id = p_user_id;
END;//
DELIMITER ;

-- Call and get output
CALL get_user_stats(123, @count, @total);
SELECT @count AS order_count, @total AS total_spent;

Procedure Management

-- Show all procedures
SHOW PROCEDURE STATUS WHERE Db = 'your_database';

-- Show procedure definition
SHOW CREATE PROCEDURE procedure_name;

-- Drop procedure
DROP PROCEDURE IF EXISTS procedure_name;
✅ Stored Procedure Benefits:
  • Reusable business logic
  • Reduced network traffic
  • Centralized security control
  • Performance improvements

10. Transactions & ACID

ACID Properties

Property Description Example
Atomicity All operations succeed or all fail (all-or-nothing) Bank transfer: debit and credit both succeed or both fail
Consistency Database remains in valid state before and after transaction Account balances can't be negative
Isolation Concurrent transactions don't interfere with each other Two users booking the same seat see consistent data
Durability Committed changes persist even after system failure Completed payment survives server crash

Basic Transaction Operations

-- Start transaction
START TRANSACTION;
-- or
BEGIN;

-- Perform operations
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

-- Commit transaction
COMMIT;

-- Or rollback on error
ROLLBACK;

Transaction with Error Handling

DELIMITER //
CREATE PROCEDURE transfer_money(
    IN from_account INT,
    IN to_account INT,
    IN amount DECIMAL(10,2)
)
BEGIN
    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        ROLLBACK;
        SELECT 'Transaction failed' AS message;
    END;
    
    START TRANSACTION;
    
    -- Deduct from sender
    UPDATE accounts 
    SET balance = balance - amount 
    WHERE id = from_account AND balance >= amount;
    
    -- Check if deduction succeeded
    IF ROW_COUNT() = 0 THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Insufficient funds';
    END IF;
    
    -- Add to receiver
    UPDATE accounts 
    SET balance = balance + amount 
    WHERE id = to_account;
    
    COMMIT;
    SELECT 'Transfer successful' AS message;
END;//
DELIMITER ;

Isolation Levels

Level Dirty Read Non-Repeatable Read Phantom Read
READ UNCOMMITTED Yes Yes Yes
READ COMMITTED No Yes Yes
REPEATABLE READ (default) No No Yes (InnoDB prevents)
SERIALIZABLE No No No 
-- Set isolation level
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- Or for specific transaction
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
START TRANSACTION;
-- ... operations ...
COMMIT;

Locking

-- Shared lock (read lock)
SELECT * FROM products WHERE id = 1 LOCK IN SHARE MODE;

-- Exclusive lock (write lock)
SELECT * FROM products WHERE id = 1 FOR UPDATE;

-- Transaction with locking
START TRANSACTION;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
-- Other transactions can't modify this row until commit
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
💡 Best Practice: Keep transactions short and commit as soon as possible. Long-running transactions can cause locking issues and degrade performance.

11. Query Optimization

Query Optimization Techniques

1. Use Indexes Effectively

-- ❌ BAD: No index on WHERE clause
SELECT * FROM users WHERE email = 'john@example.com';

-- ✅ GOOD: Index on email column
CREATE INDEX idx_email ON users(email);
SELECT * FROM users WHERE email = 'john@example.com';

2. Avoid SELECT *

-- ❌ BAD: Fetches all columns
SELECT * FROM users WHERE id = 1;

-- ✅ GOOD: Only fetch needed columns
SELECT id, name, email FROM users WHERE id = 1;

3. Use LIMIT for Large Result Sets

-- ❌ BAD: Fetches all rows
SELECT * FROM orders ORDER BY created_at DESC;

-- ✅ GOOD: Limit results with pagination
SELECT * FROM orders 
ORDER BY created_at DESC 
LIMIT 20 OFFSET 0;

4. Optimize JOIN Operations

-- ❌ BAD: Join without indexes
SELECT u.name, o.total
FROM users u
JOIN orders o ON u.id = o.user_id;

-- ✅ GOOD: Index foreign keys
CREATE INDEX idx_user_id ON orders(user_id);
SELECT u.name, o.total
FROM users u
JOIN orders o ON u.id = o.user_id;

5. Avoid Functions on Indexed Columns

-- ❌ BAD: Function prevents index usage
SELECT * FROM users WHERE YEAR(created_at) = 2024;

-- ✅ GOOD: Use range comparison
SELECT * FROM users 
WHERE created_at >= '2024-01-01' 
    AND created_at < '2025-01-01';

6. Use EXISTS Instead of IN for Subqueries

-- ❌ SLOWER: IN with subquery
SELECT * FROM users
WHERE id IN (SELECT user_id FROM orders WHERE total > 1000);

-- ✅ FASTER: EXISTS
SELECT * FROM users u
WHERE EXISTS (
    SELECT 1 FROM orders o 
    WHERE o.user_id = u.id AND o.total > 1000
);

7. Avoid OR Conditions When Possible

-- ❌ BAD: OR can prevent index usage
SELECT * FROM users WHERE name = 'John' OR email = 'john@example.com';

-- ✅ GOOD: Use UNION if appropriate
SELECT * FROM users WHERE name = 'John'
UNION
SELECT * FROM users WHERE email = 'john@example.com';

8. Use Covering Indexes

-- Create covering index
CREATE INDEX idx_covering ON orders(user_id, status, total);

-- Query uses only index (no table lookup needed)
SELECT status, total 
FROM orders 
WHERE user_id = 123;
✅ Query Optimization Checklist:
  • Index columns used in WHERE, JOIN, ORDER BY
  • Select only needed columns
  • Use LIMIT for pagination
  • Avoid functions on indexed columns
  • Use EXISTS instead of IN for large subqueries
  • Analyze queries with EXPLAIN
  • Keep statistics up to date (ANALYZE TABLE)

12. EXPLAIN & Query Analysis

Using EXPLAIN

-- Basic EXPLAIN
EXPLAIN SELECT * FROM users WHERE email = 'john@example.com';

-- More detailed format
EXPLAIN FORMAT=JSON SELECT * FROM users WHERE id = 1;

-- Analyze actual execution
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 123;

EXPLAIN Output Columns

Column Description
id SELECT identifier (for subqueries)
select_type Type of SELECT (SIMPLE, PRIMARY, SUBQUERY, etc.)
table Table name
type Join type (system, const, eq_ref, ref, range, index, ALL)
possible_keys Indexes that could be used
key Index actually used
key_len Length of index used
rows Estimated rows examined
Extra Additional information

Join Types (Best to Worst)

Type Description Performance
system Table has only one row ⭐⭐⭐⭐⭐ Best

Example Analysis

-- Query to analyze
EXPLAIN SELECT u.name, o.total
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.status = 'completed'
ORDER BY o.created_at DESC
LIMIT 10;

-- Good EXPLAIN output should show:
-- type: ref or eq_ref (not ALL)
-- key: showing index name (not NULL)
-- rows: small number
-- Extra: Using index condition (good)
⚠️ Warning Signs in EXPLAIN:
  • type: ALL - Full table scan (add indexes)
  • key: NULL - No index used
  • rows: large number - Too many rows examined
  • Extra: Using filesort - Expensive sort operation
  • Extra: Using temporary - Creates temporary table

13. Storage Engines (InnoDB vs MyISAM)

InnoDB vs MyISAM Comparison

Feature InnoDB MyISAM
Transactions ✅ Yes (ACID compliant) ❌ No
Foreign Keys ✅ Yes ❌ No
Row-level Locking ✅ Yes (better concurrency) ❌ No (table-level only)
Crash Recovery ✅ Yes (automatic) ⚠️ Manual repair needed
Full-text Search ✅ Yes (5.6+) ✅ Yes
Clustered Index ✅ Yes (data stored in PK order) ❌ No
Memory Usage Higher (caches data & indexes) Lower (caches indexes only)
Best For Most applications, OLTP, high concurrency Read-heavy, no transactions needed

Choosing Storage Engine

-- Create table with InnoDB (default in MySQL 5.5+)
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(100)
) ENGINE=InnoDB;

-- Create table with MyISAM
CREATE TABLE logs (
    id INT PRIMARY KEY,
    message TEXT
) ENGINE=MyISAM;

-- Check table engine
SHOW TABLE STATUS WHERE Name = 'users';

-- Change engine
ALTER TABLE users ENGINE=InnoDB;

InnoDB Features

-- InnoDB specific features

-- Foreign key constraints
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    FOREIGN KEY (user_id) REFERENCES users(id)
        ON DELETE CASCADE
) ENGINE=InnoDB;

-- Transactions
START TRANSACTION;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;

-- Row-level locking
SELECT * FROM products WHERE id = 1 FOR UPDATE;

When to Use Each Engine

✅ Use InnoDB for:
  • Transactional applications (e-commerce, banking)
  • High concurrency (many simultaneous users)
  • Data integrity is critical
  • Foreign key constraints needed
  • Crash recovery important
  • Default choice for most applications
💡 Use MyISAM for:
  • Read-heavy workloads with rare writes
  • Logging and archival data
  • Full-text search (pre MySQL 5.6)
  • Simple data that doesn't need transactions
  • Note: InnoDB is recommended for most use cases

Performance Tuning

-- InnoDB buffer pool (most important setting)
-- Set to 70-80% of available RAM
SET GLOBAL innodb_buffer_pool_size = 8G;

-- InnoDB log file size
SET GLOBAL innodb_log_file_size = 512M;

-- Check engine status
SHOW ENGINE INNODB STATUS;

-- View engine variables
SHOW VARIABLES LIKE 'innodb%';
⚠️ Important: MySQL 5.5+ uses InnoDB as the default storage engine. MyISAM is considered legacy and should only be used for specific use cases. Always prefer InnoDB unless you have a specific reason to use another engine.
