1. AWS Lambda

What is AWS Lambda?

AWS Lambda is a serverless compute service that runs code in response to events without requiring server management. You pay only for the compute time consumed.

Key Features

  • Event-driven execution: Triggered by AWS services (S3, DynamoDB, API Gateway, etc.)
  • Auto-scaling: Automatically scales based on request volume
  • Pay-per-use: Charged only for execution time (100ms increments)
  • Multiple runtimes: Node.js, Python, Java, Go, Ruby, .NET, custom runtimes
  • Built-in fault tolerance: Automatic retries and dead-letter queues

Lambda Function Example (Node.js)

exports.handler = async (event, context) => {
    console.log('Event:', JSON.stringify(event));
    
    const response = {
        statusCode: 200,
        headers: {
            'Content-Type': 'application/json',
            'Access-Control-Allow-Origin': '*'
        },
        body: JSON.stringify({
            message: 'Hello from Lambda!',
            input: event
        })
    };
    
    return response;
};

// Processing S3 events
exports.s3Handler = async (event) => {
    const records = event.Records;
    
    for (const record of records) {
        const bucket = record.s3.bucket.name;
        const key = decodeURIComponent(record.s3.object.key.replace(/\+/g, ' '));
        
        console.log(`Processing file: ${key} from bucket: ${bucket}`);
        await processFile(bucket, key);
    }
    
    return { statusCode: 200 };
};

Lambda Limitations

Resource Limit
Max execution time 15 minutes
Memory allocation 128 MB to 10,240 MB
Deployment package size 50 MB (zipped), 250 MB (unzipped)
Concurrent executions 1,000 (soft limit, can be increased)
💡 Use Cases: API backends, data processing, scheduled tasks, real-time file processing, IoT backends, and event-driven workflows.

2. AWS S3

What is S3?

Amazon S3 is an object storage service offering scalability, data availability, security, and performance. It stores data as objects within buckets.

S3 Operations with AWS SDK

const AWS = require('aws-sdk');
const s3 = new AWS.S3();

async function uploadFile(bucket, key, body) {
    const params = {
        Bucket: bucket,
        Key: key,
        Body: body,
        ContentType: 'image/jpeg',
        ACL: 'public-read'
    };
    
    const result = await s3.upload(params).promise();
    return result.Location;
}

function getPresignedUrl(bucket, key, expiresIn = 3600) {
    return s3.getSignedUrl('getObject', {
        Bucket: bucket,
        Key: key,
        Expires: expiresIn
    });
}

S3 Storage Classes

Class Use Case
S3 Standard Frequently accessed data
S3 Intelligent-Tiering Unknown access patterns
S3 Glacier Long-term archive

4. AWS RDS

What is RDS?

Amazon RDS is a managed relational database service supporting MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, and Aurora.

RDS Features

  • Automated backups with point-in-time recovery
  • Multi-AZ deployment for high availability
  • Read replicas for read scaling
  • Encryption at rest with AWS KMS
  • Automated patching and upgrades

6. AWS SES

What is SES?

Amazon SES is a cloud-based email sending service for transactional and marketing emails with high deliverability.

Sending Email

const ses = new AWS.SES();

async function sendEmail(to, subject, body) {
    const params = {
        Source: 'noreply@example.com',
        Destination: { ToAddresses: [to] },
        Message: {
            Subject: { Data: subject },
            Body: { Text: { Data: body } }
        }
    };
    
    return await ses.sendEmail(params).promise();
}

7. AWS SNS

What is SNS?

Amazon SNS is a fully managed pub/sub messaging service for application-to-application and application-to-person communication.

Publishing Messages

const sns = new AWS.SNS();

async function publishMessage(topicArn, message) {
    const params = {
        TopicArn: topicArn,
        Message: JSON.stringify(message)
    };
    
    return await sns.publish(params).promise();
}

8. AWS Cognito

What is AWS Cognito?

Amazon Cognito provides authentication, authorization, and user management for web and mobile apps.

User Authentication

const cognito = new AWS.CognitoIdentityServiceProvider();

async function signIn(username, password) {
    const params = {
        AuthFlow: 'USER_PASSWORD_AUTH',
        ClientId: CLIENT_ID,
        AuthParameters: {
            USERNAME: username,
            PASSWORD: password
        }
    };
    
    const result = await cognito.initiateAuth(params).promise();

}

10. Kubernetes Concepts

What is Kubernetes?

Kubernetes is an open-source container orchestration platform for automating deployment, scaling, and management of containerized applications.

Core Components

  • Pod: Smallest deployable unit, one or more containers
  • Deployment: Manages replica sets and rolling updates
  • Service: Network endpoint to access pods
  • ConfigMap: Configuration data
  • Secret: Sensitive data like passwords
  • Ingress: HTTP/HTTPS routing

Deployment Example

apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myapp:latest
        ports:
        - containerPort: 3000
        resources:
          requests:
            memory: "128Mi"
            cpu: "100m"
          limits:
            memory: "256Mi"
            cpu: "200m"

Service Example

apiVersion: v1
kind: Service
metadata:
  name: myapp-service
spec:
  type: LoadBalancer
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 3000

11. Redis & Memcached Caching

Caching Strategies

  • Cache-Aside: Application manages cache, loads from DB on miss
  • Write-Through: Write to cache and DB simultaneously
  • Write-Behind: Write to cache first, DB asynchronously
  • Refresh-Ahead: Proactively refresh before expiration

Redis Cache Implementation

const redis = require('redis');
const client = redis.createClient();

async function getUser(userId) {
    const cacheKey = `user:${userId}`;
    
    // Try cache first
    let user = await client.get(cacheKey);
    if (user) {
        return JSON.parse(user);
    }
    
    // Fetch from database
    user = await db.users.findById(userId);
    
    // Store in cache
    await client.setEx(cacheKey, 3600, JSON.stringify(user));
    
    return user;
}

Cache Invalidation Patterns

  • TTL-based: Automatic expiration after time period
  • Event-based: Invalidate on data updates
  • Tag-based: Group related cache entries

12. Microservices Architecture

What are Microservices?

Microservices architecture structures an application as a collection of small, independent services that communicate over networks.

Key Principles

  • Single Responsibility: Each service focuses on one business capability
  • Autonomous: Independent deployment and scaling
  • Decentralized: Own database per service
  • Resilient: Failure isolation and graceful degradation
  • Observable: Logging, monitoring, tracing

Communication Patterns

Pattern Description Example
Synchronous (REST) HTTP request/response API Gateway to services
Asynchronous (Events) Message queues, pub/sub Order service publishes events
gRPC High-performance RPC Service-to-service communication

API Gateway Pattern

// API Gateway routes requests to microservices
const express = require('express');
const httpProxy = require('http-proxy');

const app = express();
const proxy = httpProxy.createProxyServer();

app.use('/users', (req, res) => {
    proxy.web(req, res, { target: 'http://user-service:3001' });
});

app.use('/orders', (req, res) => {
    proxy.web(req, res, { target: 'http://order-service:3002' });
});

app.use('/products', (req, res) => {
    proxy.web(req, res, { target: 'http://product-service:3003' });
});

Service Discovery

Services register themselves and discover other services dynamically using tools like Consul, Eureka, or Kubernetes DNS.

⚠️ Challenges: Distributed data management, network latency, service coordination, testing complexity, and operational overhead.

13. Serverless Architecture

What is Serverless?

Serverless computing allows building applications without managing servers. Cloud provider handles infrastructure, scaling, and maintenance.

Serverless Components

  • Functions as a Service (FaaS): AWS Lambda, Azure Functions, Google Cloud Functions
  • Backend as a Service (BaaS): Firebase, AWS Amplify, Supabase
  • API Gateway: HTTP endpoints for functions
  • Event Sources: Triggers from various services

Serverless REST API Example

# serverless.yml
service: my-api

provider:
  name: aws
  runtime: nodejs18.x
  region: us-east-1

functions:
  getUsers:
    handler: handlers/users.get
    events:
      - http:
          path: users
          method: get
          cors: true
  
  createUser:
    handler: handlers/users.create
    events:
      - http:
          path: users
          method: post
          cors: true
  
  getUser:
    handler: handlers/users.getById
    events:
      - http:
          path: users/{id}
          method: get
          cors: true

Benefits & Trade-offs

Benefits Trade-offs
No server management Cold start latency
Auto-scaling Vendor lock-in
Pay per execution Execution time limits
High availability Debugging complexity

14. Load Balancing

What is Load Balancing?

Load balancing distributes network traffic across multiple servers to ensure no single server becomes overwhelmed, improving performance and availability.

Load Balancing Algorithms

  • Round Robin: Distributes requests evenly in sequence
  • Least Connections: Routes to server with fewest active connections
  • IP Hash: Routes based on client IP address
  • Weighted Round Robin: Assigns weights based on server capacity
  • Least Response Time: Routes to fastest responding server

AWS Load Balancers

Type Layer Use Case
Application Load Balancer (ALB) Layer 7 (HTTP/HTTPS) Web applications, microservices
Network Load Balancer (NLB) Layer 4 (TCP/UDP) High performance, low latency
Gateway Load Balancer Layer 3 Third-party virtual appliances

Nginx Load Balancer Configuration

upstream backend {
    # Load balancing method
    least_conn;
    
    # Backend servers
    server app1.example.com:3000 weight=3;
    server app2.example.com:3000 weight=2;
    server app3.example.com:3000 weight=1;
    
    # Health check
    server app4.example.com:3000 backup;
}

server {
    listen 80;
    server_name example.com;
    
    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        # Timeouts
        proxy_connect_timeout 5s;
        proxy_send_timeout 10s;
        proxy_read_timeout 10s;
    }
}

Health Checks

Load balancers perform health checks to ensure traffic only routes to healthy instances. Failed instances are automatically removed from the pool.

15. Scalability Patterns

Types of Scaling

  • Vertical Scaling (Scale Up): Increase resources of single server (CPU, RAM)
  • Horizontal Scaling (Scale Out): Add more servers to distribute load

Horizontal Scaling Patterns

Pattern Description
Stateless Services No session data stored on servers, easy to add/remove instances
Database Replication Read replicas for read-heavy workloads
Caching Layer Redis/Memcached to reduce database load
CDN Content delivery network for static assets
Message Queues Async processing to handle traffic spikes
Database Sharding Partition data across multiple databases

Auto-Scaling Example (AWS)

{
  "AutoScalingGroupName": "my-app-asg",
  "MinSize": 2,
  "MaxSize": 10,
  "DesiredCapacity": 3,
  "TargetGroupARNs": ["arn:aws:elasticloadbalancing:..."],
  "HealthCheckType": "ELB",
  "HealthCheckGracePeriod": 300,
  "ScalingPolicies": [
    {
      "PolicyName": "scale-out",
      "AdjustmentType": "ChangeInCapacity",
      "ScalingAdjustment": 2,
      "TargetTrackingConfiguration": {
        "PredefinedMetricType": "ASGAverageCPUUtilization",
        "TargetValue": 70.0
      }
    }
  ]
}

Database Scaling Strategies

  • Read Replicas: Offload read traffic from primary database
  • Connection Pooling: Reuse database connections efficiently
  • Vertical Partitioning: Split tables by columns
  • Horizontal Partitioning (Sharding): Split data by rows across multiple databases
  • CQRS: Separate read and write models

CAP Theorem

In distributed systems, you can only guarantee 2 of 3 properties:

  • Consistency: All nodes see the same data at the same time
  • Availability: Every request receives a response
  • Partition Tolerance: System continues despite network failures
✅ Best Practices: Design for horizontal scaling, use caching aggressively, implement async processing, monitor metrics, use CDN for static content, and optimize database queries.
