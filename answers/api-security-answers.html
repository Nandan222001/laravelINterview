1. REST API Principles

What is REST?

REST (Representational State Transfer) is an architectural style for designing networked applications. It relies on stateless, client-server communication using HTTP protocols.

Core Principles of REST

  • Client-Server Architecture: Separation of concerns between client and server
  • Stateless: Each request contains all information needed to process it
  • Cacheable: Responses must define themselves as cacheable or not
  • Uniform Interface: Standardized way of communicating between components
  • Layered System: Client cannot tell if connected directly to server or intermediary
  • Code on Demand (Optional): Server can extend client functionality

RESTful API Design Best Practices

<?php
// Resource-based URLs (use nouns, not verbs)
// Good:
GET    /api/users          // Get all users
GET    /api/users/123      // Get specific user
POST   /api/users          // Create new user
PUT    /api/users/123      // Update user
DELETE /api/users/123      // Delete user

// Bad:
GET  /api/getUsers
POST /api/createUser
POST /api/deleteUser

// Use plural nouns for collections
GET /api/users
GET /api/products
GET /api/orders

// Nested resources for relationships
GET /api/users/123/orders          // Get user's orders
GET /api/users/123/orders/456      // Get specific order
POST /api/users/123/orders         // Create order for user

// Filtering, sorting, pagination
GET /api/users?status=active&role=admin
GET /api/products?sort=price&order=desc
GET /api/users?page=2&limit=20

// Versioning
GET /api/v1/users
GET /api/v2/users

// Laravel API Routes Example
Route::prefix('api/v1')->group(function () {
    // User resources
    Route::apiResource('users', UserController::class);
    
    // Nested resources
    Route::get('users/{user}/posts', [UserController::class, 'posts']);
    
    // Custom actions (use sparingly)
    Route::post('users/{user}/activate', [UserController::class, 'activate']);
    Route::post('users/{user}/deactivate', [UserController::class, 'deactivate']);
});
?>

HTTP Status Codes

Code Category Description Use Case
200 Success OK Successful GET, PUT, PATCH
201 Success Created Successful POST creating resource
204 Success No Content Successful DELETE or update with no response body
400 Client Error Bad Request Invalid request data
401 Client Error Unauthorized Missing or invalid authentication
403 Client Error Forbidden Authenticated but not authorized
404 Client Error Not Found Resource doesn't exist
422 Client Error Unprocessable Entity Validation errors
429 Client Error Too Many Requests Rate limit exceeded
500 Server Error Internal Server Error Unexpected server error

Laravel API Response Example

<?php
namespace App\Http\Controllers\Api;

use App\Http\Controllers\Controller;
use App\Models\User;
use Illuminate\Http\Request;
use Illuminate\Http\JsonResponse;

class UserController extends Controller
{
    public function index(): JsonResponse
    {
        $users = User::paginate(20);
        
        return response()->json([
            'success' => true,
            'data' => $users->items(),
            'meta' => [
                'current_page' => $users->currentPage(),
                'total' => $users->total(),
                'per_page' => $users->perPage(),
            ]
        ]);
    }
    
    public function store(Request $request): JsonResponse
    {
        $validated = $request->validate([
            'name' => 'required|string|max:255',
            'email' => 'required|email|unique:users',
            'password' => 'required|min:8',
        ]);
        
        $user = User::create($validated);
        
        return response()->json([
            'success' => true,
            'message' => 'User created successfully',
            'data' => $user
        ], 201);
    }
    
    public function show(User $user): JsonResponse
    {
        return response()->json([
            'success' => true,
            'data' => $user
        ]);
    }
    
    public function update(Request $request, User $user): JsonResponse
    {
        $validated = $request->validate([
            'name' => 'sometimes|string|max:255',
            'email' => 'sometimes|email|unique:users,email,' . $user->id,
        ]);
        
        $user->update($validated);
        
        return response()->json([
            'success' => true,
            'message' => 'User updated successfully',
            'data' => $user
        ]);
    }
    
    public function destroy(User $user): JsonResponse
    {
        $user->delete();
        
        return response()->json([
            'success' => true,
            'message' => 'User deleted successfully'
        ], 204);
    }
}
?>
💡 Best Practice: Use API Resources (Transformers) to format and standardize your API responses consistently across your application.

2. HTTP Methods

Overview of HTTP Methods

Method Purpose Idempotent Safe Cacheable
GET Retrieve resource ✓ ✓ ✓
POST Create resource ✗ ✗ Only if Cache-Control set
PUT Update/Replace resource ✓ ✗ ✗
PATCH Partial update ✗ ✗ ✗
DELETE Delete resource ✓ ✗ ✗
HEAD Get headers only ✓ ✓ ✓
OPTIONS Get allowed methods ✓ ✓ ✗

Detailed Method Examples

<?php
// GET - Retrieve resources (read-only)
Route::get('/api/users', function () {
    return User::all();
});

Route::get('/api/users/{id}', function ($id) {
    return User::findOrFail($id);
});

// POST - Create new resource
Route::post('/api/users', function (Request $request) {
    $user = User::create($request->validated());
    return response()->json($user, 201);
});

// PUT - Full replacement of resource
Route::put('/api/users/{id}', function (Request $request, $id) {
    $user = User::findOrFail($id);
    // PUT requires all fields
    $user->update([
        'name' => $request->name,
        'email' => $request->email,
        'role' => $request->role,
    ]);
    return response()->json($user);
});

// PATCH - Partial update of resource
Route::patch('/api/users/{id}', function (Request $request, $id) {
    $user = User::findOrFail($id);
    // PATCH allows partial updates
    $user->update($request->only(['name', 'email']));
    return response()->json($user);
});

// DELETE - Remove resource
Route::delete('/api/users/{id}', function ($id) {
    $user = User::findOrFail($id);
    $user->delete();
    return response()->json(null, 204);
});

// HEAD - Get metadata without body
Route::match(['head', 'get'], '/api/users', function () {
    return User::all();
});

// OPTIONS - Discover allowed methods (CORS)
Route::options('/api/users', function () {
    return response()->json(null, 200, [
        'Allow' => 'GET, POST, PUT, PATCH, DELETE, OPTIONS'
    ]);
});
?>

Idempotency Explained

<?php
// Idempotent: Multiple identical requests have same effect

// GET is idempotent - reading doesn't change state
GET /api/users/123  // Returns same data every time

// PUT is idempotent - replacing with same data multiple times
PUT /api/users/123
{
    "name": "John Doe",
    "email": "john@example.com"
}
// Running this 1 time or 100 times has same result

// DELETE is idempotent - deleting already deleted resource
DELETE /api/users/123
// First call deletes, subsequent calls get 404 but system state unchanged

// POST is NOT idempotent - creates new resource each time
POST /api/users
{
    "name": "Jane Doe",
    "email": "jane@example.com"
}
// Each call creates a new user

// Making POST idempotent with idempotency key
namespace App\Http\Middleware;

class IdempotencyMiddleware
{
    public function handle($request, Closure $next)
    {
        if ($request->isMethod('POST')) {
            $key = $request->header('Idempotency-Key');
            
            if ($key && Cache::has("idempotency:{$key}")) {
                return Cache::get("idempotency:{$key}");
            }
            
            $response = $next($request);
            
            if ($key && $response->isSuccessful()) {
                Cache::put("idempotency:{$key}", $response, 3600);
            }
            
            return $response;
        }
        
        return $next($request);
    }
}
?>
✅ Remember: Use GET for reading, POST for creating, PUT for full replacement, PATCH for partial updates, and DELETE for removal.

3. Authentication vs Authorization

Key Differences

Aspect Authentication Authorization
Question Who are you? What can you do?
Purpose Verify identity Grant permissions
Process Login with credentials Check permissions/roles
HTTP Code 401 Unauthorized 403 Forbidden
Example Username/Password, Token Roles, Policies, Gates

Authentication in Laravel

<?php
namespace App\Http\Controllers\Auth;

use App\Http\Controllers\Controller;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Auth;
use Illuminate\Validation\ValidationException;

class LoginController extends Controller
{
    public function login(Request $request)
    {
        $credentials = $request->validate([
            'email' => 'required|email',
            'password' => 'required',
        ]);
        
        // Authentication - Verify identity
        if (!Auth::attempt($credentials)) {
            throw ValidationException::withMessages([
                'email' => ['The provided credentials are incorrect.'],
            ]);
        }
        
        $user = Auth::user();
        $token = $user->createToken('api-token')->plainTextToken;
        
        return response()->json([
            'user' => $user,
            'token' => $token,
        ]);
    }
    
    public function logout(Request $request)
    {
        $request->user()->currentAccessToken()->delete();
        
        return response()->json([
            'message' => 'Logged out successfully'
        ]);
    }
}

// Authentication Middleware
Route::middleware('auth:sanctum')->group(function () {
    Route::get('/user', function (Request $request) {
        return $request->user();
    });
});
?>

Authorization in Laravel

<?php
// 1. Gates - Simple authorization logic
namespace App\Providers;

use Illuminate\Support\Facades\Gate;
use App\Models\User;
use App\Models\Post;

class AuthServiceProvider extends ServiceProvider
{
    public function boot()
    {
        // Define gates
        Gate::define('update-post', function (User $user, Post $post) {
            return $user->id === $post->user_id;
        });
        
        Gate::define('delete-post', function (User $user, Post $post) {
            return $user->id === $post->user_id || $user->isAdmin();
        });
        
        // Super admin bypass
        Gate::before(function (User $user, $ability) {
            if ($user->isSuperAdmin()) {
                return true;
            }
        });
    }
}

// 2. Policies - Organized authorization for models
namespace App\Policies;

use App\Models\User;
use App\Models\Post;

class PostPolicy
{
    public function view(User $user, Post $post): bool
    {
        return $post->published || $user->id === $post->user_id;
    }
    
    public function create(User $user): bool
    {
        return $user->can_create_posts;
    }
    
    public function update(User $user, Post $post): bool
    {
        return $user->id === $post->user_id;
    }
    
    public function delete(User $user, Post $post): bool
    {
        return $user->id === $post->user_id || $user->isAdmin();
    }
}

// 3. Using Authorization in Controllers
namespace App\Http\Controllers;

use App\Models\Post;
use Illuminate\Http\Request;

class PostController extends Controller
{
    public function update(Request $request, Post $post)
    {
        // Using gates
        if (Gate::denies('update-post', $post)) {
            abort(403, 'Unauthorized to update this post');
        }
        
        // Or using authorize helper (throws 403 automatically)
        $this->authorize('update', $post);
        
        $post->update($request->validated());
        
        return response()->json($post);
    }
    
    public function destroy(Post $post)
    {
        // Inline authorization check
        if (!auth()->user()->can('delete', $post)) {
            return response()->json([
                'error' => 'You do not have permission to delete this post'
            ], 403);
        }
        
        $post->delete();
        
        return response()->json(null, 204);
    }
}

// 4. Role-Based Access Control (RBAC)
namespace App\Models;

class User extends Authenticatable
{
    public function hasRole(string $role): bool
    {
        return $this->roles()->where('name', $role)->exists();
    }
    
    public function hasPermission(string $permission): bool
    {
        return $this->roles()
            ->whereHas('permissions', function ($query) use ($permission) {
                $query->where('name', $permission);
            })
            ->exists();
    }
    
    public function isAdmin(): bool
    {
        return $this->hasRole('admin');
    }
}

// Middleware for role checking
namespace App\Http\Middleware;

class CheckRole
{
    public function handle($request, Closure $next, ...$roles)
    {
        if (!$request->user() || !$request->user()->hasAnyRole($roles)) {
            abort(403, 'Unauthorized access');
        }
        
        return $next($request);
    }
}

// Usage in routes
Route::middleware(['auth', 'role:admin'])->group(function () {
    Route::get('/admin/dashboard', [AdminController::class, 'index']);
});
?>
⚠️ Important: Always perform authorization checks on the server side. Never rely solely on client-side permission checks.

4. JWT Tokens

What is JWT?

JWT (JSON Web Token) is a compact, URL-safe token format used for securely transmitting information between parties. It consists of three parts: Header, Payload, and Signature.

JWT Structure

// JWT Format: header.payload.signature
// Example:
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

// Decoded:
// Header
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload (Claims)
{
  "sub": "1234567890",     // Subject (user ID)
  "name": "John Doe",      // Custom claim
  "iat": 1516239022,        // Issued at
  "exp": 1516242622         // Expiration
}

// Signature
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret
)

JWT Implementation in Laravel

<?php
// Using tymon/jwt-auth package
// Install: composer require tymon/jwt-auth

namespace App\Http\Controllers\Auth;

use App\Http\Controllers\Controller;
use Illuminate\Http\Request;
use Tymon\JWTAuth\Facades\JWTAuth;
use Tymon\JWTAuth\Exceptions\JWTException;

class JWTAuthController extends Controller
{
    public function login(Request $request)
    {
        $credentials = $request->only('email', 'password');
        
        try {
            if (!$token = JWTAuth::attempt($credentials)) {
                return response()->json([
                    'error' => 'Invalid credentials'
                ], 401);
            }
        } catch (JWTException $e) {
            return response()->json([
                'error' => 'Could not create token'
            ], 500);
        }
        
        return response()->json([
            'token' => $token,
            'token_type' => 'bearer',
            'expires_in' => auth()->factory()->getTTL() * 60
        ]);
    }
    
    public function me(Request $request)
    {
        try {
            $user = JWTAuth::parseToken()->authenticate();
        } catch (JWTException $e) {
            return response()->json(['error' => 'Token invalid'], 401);
        }
        
        return response()->json($user);
    }
    
    public function refresh()
    {
        try {
            $token = JWTAuth::refresh(JWTAuth::getToken());
            return response()->json(['token' => $token]);
        } catch (JWTException $e) {
            return response()->json(['error' => 'Token cannot be refreshed'], 401);
        }
    }
    
    public function logout()
    {
        try {
            JWTAuth::invalidate(JWTAuth::getToken());
            return response()->json(['message' => 'Successfully logged out']);
        } catch (JWTException $e) {
            return response()->json(['error' => 'Failed to logout'], 500);
        }
    }
}

// Custom JWT Service
namespace App\Services;

use Firebase\JWT\JWT;
use Firebase\JWT\Key;

class JWTService
{
    private string $secret;
    private string $algorithm = 'HS256';
    
    public function __construct()
    {
        $this->secret = config('app.jwt_secret');
    }
    
    public function generateToken(array $payload, int $expiresIn = 3600): string
    {
        $payload['iat'] = time();
        $payload['exp'] = time() + $expiresIn;
        
        return JWT::encode($payload, $this->secret, $this->algorithm);
    }
    
    public function verifyToken(string $token): object
    {
        return JWT::decode($token, new Key($this->secret, $this->algorithm));
    }
    
    public function refreshToken(string $token): string
    {
        $decoded = $this->verifyToken($token);
        
        // Create new payload without exp
        $payload = (array) $decoded;
        unset($payload['iat'], $payload['exp']);
        
        return $this->generateToken($payload);
    }
}

// JWT Middleware
namespace App\Http\Middleware;

use Closure;
use App\Services\JWTService;

class JWTMiddleware
{
    private JWTService $jwtService;
    
    public function __construct(JWTService $jwtService)
    {
        $this->jwtService = $jwtService;
    }
    
    public function handle($request, Closure $next)
    {
        $token = $request->bearerToken();
        
        if (!$token) {
            return response()->json(['error' => 'Token not provided'], 401);
        }
        
        try {
            $decoded = $this->jwtService->verifyToken($token);
            $request->merge(['user_id' => $decoded->sub]);
        } catch (\Exception $e) {
            return response()->json(['error' => 'Invalid token'], 401);
        }
        
        return $next($request);
    }
}
?>

JWT Best Practices

  • Use HTTPS: Always transmit JWT over HTTPS to prevent interception
  • Set Expiration: Use short expiration times (15-60 minutes)
  • Implement Refresh Tokens: Use refresh tokens for long-lived sessions
  • Store Securely: Never store in localStorage, use httpOnly cookies
  • Validate Claims: Always validate iss, aud, exp, nbf claims
  • Use Strong Secrets: Use long, random secrets for signing
  • Implement Token Blacklist: Maintain blacklist for logged out tokens
🔴 Security Warning: JWTs are not encrypted by default, only signed. Don't store sensitive data in JWT payload as it can be decoded by anyone.
