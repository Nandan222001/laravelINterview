1. Git Basics

Git is a distributed version control system for tracking changes in source code during software development.

# Initialize repository
git init

# Clone repository
git clone https://github.com/user/repo.git

# Check status
git status

# Stage files
git add file.txt
git add .

# Commit changes
git commit -m "Commit message"

# View history
git log
git log --oneline --graph

# Push/Pull
git push origin main
git pull origin main

2. Fetch vs Pull

Git Fetch

Downloads changes from remote without merging.

# Fetch from remote
git fetch origin
git fetch --all
git fetch --prune

# View fetched changes
git log origin/main
git diff origin/main

Git Pull

Fetch + Merge in one command.

# Pull with merge
git pull origin main

# Pull with rebase
git pull --rebase origin main
AspectFetchPull
OperationDownload onlyDownload + merge
SafetySafeCan cause conflicts
Use CaseReview before mergeQuick update

3. Merge vs Rebase

Git Merge

# Merge branch
git checkout main
git merge feature-branch

# No fast-forward
git merge --no-ff feature-branch

# Squash merge
git merge --squash feature-branch

Git Rebase

# Rebase onto main
git checkout feature-branch
git rebase main

# Interactive rebase
git rebase -i HEAD~5

# Continue/abort
git rebase --continue
git rebase --abort
⚠️ Golden Rule: Never rebase public branches!

4. Cherry-Pick

Apply specific commits from one branch to another.

# Cherry-pick commit
git cherry-pick abc123

# Multiple commits
git cherry-pick abc123 def456

# Range of commits
git cherry-pick abc123..def456

# Continue/abort
git cherry-pick --continue
git cherry-pick --abort

5. Git Reset (Soft, Mixed, Hard)

Soft Reset

Move HEAD, keep changes staged.

git reset --soft HEAD~1

Mixed Reset (default)

Move HEAD, unstage changes.

git reset HEAD~1
git reset --mixed HEAD~1

Hard Reset

Move HEAD, discard all changes.

git reset --hard HEAD~1
git reset --hard origin/main
⚠️ DANGER: Hard reset permanently deletes changes!

7. Conflict Resolution

Identifying Conflicts

git merge feature-branch
# CONFLICT in file.txt

git status
# Shows unmerged paths

Conflict Markers

<<<<<<< HEAD
Your changes
=======
Their changes
>>>>>>> feature-branch

Resolving

# Edit file, remove markers
git add file.txt
git commit

# Or accept one side
git checkout --ours file.txt
git checkout --theirs file.txt

# Abort if needed
git merge --abort

8. Git Workflows

Pull Request Workflow

# Create feature branch
git checkout -b feature/new-feature

# Make changes
git add .
git commit -m "Add feature"

# Push and create PR
git push -u origin feature/new-feature

# After review, merge via GitHub UI

Forking Workflow

# Fork on GitHub, then clone
git clone https://github.com/you/repo.git

# Add upstream
git remote add upstream https://github.com/original/repo.git

# Keep fork updated
git fetch upstream
git merge upstream/main

9. CI/CD Pipelines

CI: Automatically build and test code changes.

CD: Automatically deploy to production.

Pipeline Stages

1. Source → 2. Build → 3. Test → 4. Deploy → 5. Monitor

GitLab CI Example

# .gitlab-ci.yml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm ci
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  script:
    - npm test

deploy:
  stage: deploy
  script:
    - npm run deploy
  only:
    - main

10. GitHub Actions

Basic Workflow

# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3
    - uses: actions/setup-node@v3
      with:
        node-version: '18'
    - run: npm ci
    - run: npm test
    - run: npm run build

Deploy Workflow

deploy:
  runs-on: ubuntu-latest
  if: github.ref == 'refs/heads/main'
  steps:
    - uses: actions/checkout@v3
    - name: Deploy
      run: npm run deploy
      env:
        API_KEY: ${{ secrets.API_KEY }}

11. Deployment Processes

Deployment Strategies

1. Blue-Green Deployment

Two identical environments. Deploy to inactive, then switch traffic.

2. Rolling Deployment

Gradually replace instances one at a time.

3. Canary Deployment

Deploy to small subset first, monitor, then roll out to all.

Docker Deployment

# Build and push
docker build -t myapp:1.0.0 .
docker push registry.com/myapp:1.0.0

# Deploy with compose
docker-compose pull
docker-compose up -d

Zero-Downtime with PM2

pm2 reload ecosystem.config.js
pm2 gracefulReload myapp

13. Project Lifecycle Management

Development Phases

  1. Planning: Requirements gathering, architecture design
  2. Development: Feature branches, code reviews, testing
  3. Testing: QA in staging environment
  4. Release: Version tagging, deployment to production
  5. Maintenance: Bug fixes, security patches

Release Process

# 1. Create release branch
git checkout -b release/1.0.0 develop

# 2. Finalize version
npm version 1.0.0

# 3. Merge to main
git checkout main
git merge --no-ff release/1.0.0

# 4. Tag release
git tag -a v1.0.0 -m "Release 1.0.0"

# 5. Merge back to develop
git checkout develop
git merge --no-ff release/1.0.0

# 6. Push everything
git push origin main develop --tags

14. Advanced Git Commands

Git Stash

# Stash changes
git stash
git stash save "Work in progress"

# List stashes
git stash list

# Apply stash
git stash apply
git stash pop

# Apply specific stash
git stash apply stash@{2}

# Delete stash
git stash drop
git stash clear

Git Reflog

# View reflog
git reflog

# Recover lost commit
git reset --hard HEAD@{2}

# Create branch from reflog
git branch recovered HEAD@{5}

Git Bisect

# Start bisect
git bisect start
git bisect bad                # Current commit is bad
git bisect good v1.0.0        # This commit was good

# Git will checkout middle commit
# Test and mark
git bisect good   # or git bisect bad

# Repeat until found
git bisect reset  # End bisect

Git Submodules

# Add submodule
git submodule add https://github.com/user/lib.git lib

# Clone with submodules
git clone --recurse-submodules repo-url

# Update submodules
git submodule update --init --recursive
git submodule update --remote

Git Worktree

# Add worktree
git worktree add ../hotfix hotfix/bug-123

# List worktrees
git worktree list

# Remove worktree
git worktree remove ../hotfix

15. DevOps Best Practices

Git Best Practices

  • Commit Often: Small, atomic commits are easier to review and revert
  • Write Good Commit Messages: Clear, descriptive messages
  • Use Branches: Never commit directly to main
  • Pull Before Push: Always sync before pushing
  • Review Code: Use pull requests for all changes
  • Tag Releases: Use semantic versioning
  • Keep History Clean: Use rebase for feature branches

CI/CD Best Practices

  • Automate Everything: Build, test, deploy
  • Fast Feedback: Keep pipeline under 10 minutes
  • Test Pyramid: More unit tests, fewer E2E tests
  • Deploy Often: Small, frequent releases
  • Monitor Everything: Logs, metrics, alerts
  • Rollback Ready: Always have a rollback plan

Deployment Best Practices

  • Infrastructure as Code: Version control infrastructure
  • Immutable Infrastructure: Never modify running servers
  • Database Migrations: Always backward compatible
  • Feature Flags: Control feature rollout
  • Health Checks: Automated monitoring
  • Blue-Green/Canary: Safe deployment strategies
  • Secrets Management: Never commit secrets

Code Review Checklist

✓ Code follows project conventions
✓ Tests are included and passing
✓ Documentation updated
✓ No secrets or credentials committed
✓ Error handling implemented
✓ Performance considered
✓ Security vulnerabilities checked
✓ Backward compatibility maintained
✅ Remember: DevOps is about culture, automation, measurement, and sharing. Focus on continuous improvement.
