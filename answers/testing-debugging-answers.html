1. Testing Approaches

Testing Pyramid

The testing pyramid represents the ideal distribution of different test types in your application.

Test Type Purpose Speed Coverage
Unit Tests Test individual components in isolation Fast (milliseconds) 70-80% of tests
Integration Tests Test interaction between components Medium (seconds) 15-20% of tests
Feature/E2E Tests Test complete user workflows Slow (seconds to minutes) 5-10% of tests

Test-Driven Development (TDD)

TDD Cycle (Red-Green-Refactor):

1. RED: Write a failing test
   - Define the expected behavior
   - Test should fail (no implementation yet)

2. GREEN: Write minimal code to pass
   - Implement just enough to make test pass
   - Don't worry about perfection

3. REFACTOR: Improve the code
   - Clean up implementation
   - Ensure tests still pass
   - Optimize and follow best practices

4. REPEAT: Continue the cycle

Testing Best Practices

  • AAA Pattern: Arrange (setup), Act (execute), Assert (verify)
  • Test Independence: Each test should run independently without relying on others
  • Single Responsibility: Each test should verify one specific behavior
  • Clear Naming: Test names should describe what they test and expected outcome
  • Fast Execution: Tests should run quickly to encourage frequent execution
  • Deterministic: Tests should produce consistent results (no flaky tests)
  • Maintainable: Tests should be easy to understand and update
💡 Coverage Goal: Aim for 80%+ code coverage, but focus on testing critical paths and business logic rather than achieving 100% coverage.

2. Unit Testing

Basic Unit Test Structure

<?php

namespace Tests\Unit;

use PHPUnit\Framework\TestCase;
use App\Services\PriceCalculator;

class PriceCalculatorTest extends TestCase
{
    /** @test */
    public function it_calculates_price_with_tax()
    {
        // Arrange
        $calculator = new PriceCalculator();
        $basePrice = 100;
        $taxRate = 0.10;
        
        // Act
        $finalPrice = $calculator->calculateWithTax($basePrice, $taxRate);
        
        // Assert
        $this->assertEquals(110, $finalPrice);
    }
    
    /** @test */
    public function it_applies_discount_correctly()
    {
        $calculator = new PriceCalculator();
        
        $result = $calculator->applyDiscount(100, 20);
        
        $this->assertEquals(80, $result);
    }
    
    /** @test */
    public function it_throws_exception_for_invalid_discount()
    {
        $calculator = new PriceCalculator();
        
        $this->expectException(\InvalidArgumentException::class);
        $this->expectExceptionMessage('Discount cannot exceed 100%');
        
        $calculator->applyDiscount(100, 150);
    }
}

Data Providers

<?php

namespace Tests\Unit;

use PHPUnit\Framework\TestCase;

class ValidationTest extends TestCase
{
    /**
     * @test
     * @dataProvider validEmailProvider
     */
    public function it_validates_email_addresses($email, $expected)
    {
        $validator = new EmailValidator();
        
        $result = $validator->isValid($email);
        
        $this->assertEquals($expected, $result);
    }
    
    public function validEmailProvider(): array
    {
        return [
            'valid email' => ['user@example.com', true],
            'valid with subdomain' => ['user@mail.example.com', true],
            'invalid no at symbol' => ['userexample.com', false],
            'invalid no domain' => ['user@', false],
            'invalid no username' => ['@example.com', false],
            'empty string' => ['', false],
        ];
    }
}
✅ Best Practice: Use data providers to test multiple scenarios without duplicating test code. This makes tests more maintainable and comprehensive.

3. Integration Testing

Database Integration Tests

<?php

namespace Tests\Integration;

use Tests\TestCase;
use Illuminate\Foundation\Testing\RefreshDatabase;
use App\Models\User;
use App\Models\Order;

class OrderRepositoryTest extends TestCase
{
    use RefreshDatabase;
    
    /** @test */
    public function it_creates_order_with_items()
    {
        // Arrange
        $user = User::factory()->create();
        $repository = app(OrderRepository::class);
        
        $orderData = [
            'user_id' => $user->id,
            'items' => [
                ['product_id' => 1, 'quantity' => 2, 'price' => 50],
                ['product_id' => 2, 'quantity' => 1, 'price' => 100],
            ],
        ];
        
        // Act
        $order = $repository->createWithItems($orderData);
        
        // Assert
        $this->assertDatabaseHas('orders', [
            'id' => $order->id,
            'user_id' => $user->id,
            'total' => 200,
        ]);
        
        $this->assertDatabaseCount('order_items', 2);
        $this->assertEquals(2, $order->items->count());
    }
}
💡 Integration Testing: Focus on testing how different components work together. Use real database interactions but mock external API calls.

5. PHPUnit Framework

PHPUnit Assertions

<?php

// Equality assertions
$this->assertEquals($expected, $actual);
$this->assertSame($expected, $actual); // Strict comparison
$this->assertNotEquals($expected, $actual);

// Boolean assertions
$this->assertTrue($condition);
$this->assertFalse($condition);
$this->assertNull($variable);
$this->assertNotNull($variable);

// String assertions
$this->assertStringContainsString('substring', $string);
$this->assertStringStartsWith('prefix', $string);
$this->assertStringEndsWith('suffix', $string);
$this->assertMatchesRegularExpression('/pattern/', $string);

// Array assertions
$this->assertContains($needle, $haystack);
$this->assertArrayHasKey('key', $array);
$this->assertCount(5, $array);
$this->assertEmpty($array);

// Object assertions
$this->assertInstanceOf(User::class, $object);

// Exception assertions
$this->expectException(InvalidArgumentException::class);
$this->expectExceptionMessage('Invalid argument');

// JSON assertions (Laravel)
$response->assertJson(['key' => 'value']);
$response->assertJsonCount(5, 'data');
$response->assertJsonStructure(['data' => ['id', 'name']]);

Test Lifecycle Methods

<?php

namespace Tests\Unit;

use PHPUnit\Framework\TestCase;

class ServiceTest extends TestCase
{
    private $service;
    
    // Runs once before any test in the class
    public static function setUpBeforeClass(): void
    {
        // Initialize expensive resources
    }
    
    // Runs before each test method
    protected function setUp(): void
    {
        parent::setUp();
        $this->service = new MyService();
    }
    
    // Runs after each test method
    protected function tearDown(): void
    {
        $this->service = null;
        parent::tearDown();
    }
    
    // Runs once after all tests in the class
    public static function tearDownAfterClass(): void
    {
        // Clean up shared resources
    }
}

6. Test Doubles & Mocking

Types of Test Doubles

Type Purpose Verification
Stub Returns predefined responses State verification
Mock Has expectations about calls Behavior verification
Spy Records information about calls Behavior verification
Fake Working implementation (simplified) State verification

Mocking with Mockery

<?php

namespace Tests\Unit;

use Tests\TestCase;
use Mockery;
use App\Services\PaymentService;
use App\Gateways\StripeGateway;

class PaymentServiceTest extends TestCase
{
    /** @test */
    public function it_processes_payment_through_gateway()
    {
        // Create mock
        $gateway = Mockery::mock(StripeGateway::class);
        
        // Set expectations
        $gateway->shouldReceive('charge')
            ->once()
            ->with(100, 'usd', 'tok_123')
            ->andReturn([
                'success' => true,
                'charge_id' => 'ch_123',
            ]);
        
        $service = new PaymentService($gateway);
        $result = $service->processPayment(100, 'tok_123');
        
        $this->assertTrue($result['success']);
        $this->assertEquals('ch_123', $result['charge_id']);
    }
}

Laravel Mocking Facades

<?php

namespace Tests\Feature;

⚠️ Mocking Warning: Over-mocking can make tests brittle. Mock external dependencies, but test your own code with real implementations when possible.

9. Code Review Practices

Code Review Checklist

FUNCTIONALITY:
☐ Does the code do what it's supposed to do?
☐ Are there any edge cases not handled?
☐ Is error handling appropriate?
☐ Are there any security vulnerabilities?

CODE QUALITY:
☐ Is the code readable and maintainable?
☐ Are variables and functions well-named?
☐ Is there unnecessary code duplication?
☐ Are functions/methods doing one thing?

TESTING:
☐ Are there appropriate tests?
☐ Do tests cover edge cases?
☐ Do all tests pass?

PERFORMANCE:
☐ Are there any performance issues?
☐ Are database queries optimized?
☐ Are there N+1 query issues?

SECURITY:
☐ Is user input properly validated?
☐ Are SQL injection risks mitigated?
☐ Are XSS vulnerabilities prevented?
☐ Is authentication/authorization correct?

Common Code Smells

  • Long Methods: Methods over 20-30 lines should be split
  • Large Classes: Classes with too many responsibilities
  • Duplicate Code: Copy-pasted code blocks
  • Magic Numbers: Unexplained numeric constants
  • Deep Nesting: More than 3-4 levels of indentation
💡 Code Review Goals: The goal is to improve code quality and share knowledge, not to criticize. Be respectful, specific, and constructive.

10. Error Handling

Exception Handling Basics

<?php

try {
    $result = $service->processPayment($amount);
} catch (PaymentFailedException $e) {
    \Log::error('Payment failed', [
        'amount' => $amount,
        'error' => $e->getMessage(),
    ]);
    
    return response()->json([
        'error' => 'Payment processing failed',
    ], 402);
} catch (\Exception $e) {
    \Log::error('Unexpected error', ['error' => $e->getMessage()]);
    
    return response()->json([
        'error' => 'An unexpected error occurred',
    ], 500);
} finally {
    $this->cleanup();
}

Custom Exceptions

<?php

namespace App\Exceptions;

use Exception;

class InsufficientFundsException extends Exception
{
    protected $code = 402;
    
    public function __construct(
        public readonly float $required,
        public readonly float $available
    ) {
        parent::__construct(
            "Insufficient funds. Required: {$required}, Available: {$available}"
        );
    }
    
    public function render($request)
    {
        return response()->json([
            'error' => 'Insufficient funds',
            'required' => $this->required,
            'available' => $this->available,
        ], $this->code);
    }
}
⚠️ Security Note: Never expose detailed error messages or stack traces in production. Log detailed errors server-side and return generic messages to users.

Health Checks

<?php

namespace App\Http\Controllers;

use Illuminate\Support\Facades\DB;
use Illuminate\Support\Facades\Cache;
use Illuminate\Support\Facades\Redis;

class HealthCheckController extends Controller
{
    public function index()
    {
        $checks = [
            'database' => $this->checkDatabase(),
            'cache' => $this->checkCache(),
            'redis' => $this->checkRedis(),
        ];
        
        $healthy = !in_array(false, $checks);
        
        return response()->json([
            'status' => $healthy ? 'healthy' : 'unhealthy',
            'checks' => $checks,
        ], $healthy ? 200 : 503);
    }
    
    private function checkDatabase(): bool
    {
        try {
            DB::connection()->getPdo();
            return true;
        } catch (\Exception $e) {
            return false;
        }
    }
    
    private function checkCache(): bool
    {
        try {
            Cache::put('health_check', true, 10);
            return Cache::get('health_check') === true;
        } catch (\Exception $e) {
            return false;
        }
    }
    
    private function checkRedis(): bool
    {
        try {
            Redis::ping();
            return true;
        } catch (\Exception $e) {
            return false;
        }
    }
}

Response Time Monitoring

<?php

namespace App\Http\Middleware;

use Closure;
use Illuminate\Http\Request;

class PerformanceMonitor
{
    public function handle(Request $request, Closure $next)
    {
        $start = microtime(true);
        
        $response = $next($request);
        
        $duration = (microtime(true) - $start) * 1000;
        
        if ($duration > 1000) { // Log slow requests
            \Log::warning('Slow request detected', [
                'url' => $request->fullUrl(),
                'method' => $request->method(),
                'duration_ms' => $duration,
            ]);
        }
        
        $response->headers->set('X-Response-Time', $duration . 'ms');
        
        return $response;
    }
}

14. Code Optimization Strategies

Database Query Optimization

<?php

// ❌ Bad: N+1 Query Problem
$posts = Post::all();
foreach ($posts as $post) {
    echo $post->user->name; // Triggers separate query for each post
}

// ✅ Good: Eager Loading
$posts = Post::with('user')->get();
foreach ($posts as $post) {
    echo $post->user->name; // No additional queries
}

// ❌ Bad: Loading unnecessary data
$users = User::all();

// ✅ Good: Select only needed columns
$users = User::select('id', 'name', 'email')->get();

// ❌ Bad: Not using indexes
User::where('email', 'user@example.com')->first();
// Make sure 'email' column has index

// ✅ Good: Chunk large datasets
User::chunk(100, function ($users) {
    foreach ($users as $user) {
        // Process user
    }
});

Caching Strategies

<?php

// Cache database queries
$users = Cache::remember('users.active', 3600, function () {
    return User::where('active', true)->get();
});

// Cache API responses
public function getProducts()
{
    return Cache::tags(['products'])->remember('products.all', 3600, function () {
        return $this->apiClient->fetchProducts();
    });
}

// Invalidate cache when data changes
public function updateProduct(Product $product, array $data)
{
    $product->update($data);
    Cache::tags(['products'])->flush();
}

// Fragment caching in views
@cache('widget.popular-posts', 3600)
    @include('widgets.popular-posts')
@endcache

Optimization Best Practices

  • Use Eager Loading: Prevent N+1 queries with with() and load()
  • Cache Expensive Operations: Cache database queries, API calls, and computed values
  • Optimize Images: Compress and resize images, use CDN
  • Use Queue for Heavy Tasks: Move time-consuming operations to background jobs
  • Database Indexing: Add indexes to frequently queried columns
  • Lazy Loading: Use lazy() for large collections to reduce memory usage
  • Response Compression: Enable GZIP compression for responses
  • Asset Optimization: Minify CSS/JS, use asset bundling

Code-Level Optimization

<?php

// ❌ Bad: Inefficient loop
$total = 0;
foreach ($orders as $order) {
    $total += $order->total;
}

// ✅ Good: Use collection methods
$total = $orders->sum('total');

// ❌ Bad: Multiple queries in loop
foreach ($userIds as $userId) {
    $user = User::find($userId);
    // Process user
}

// ✅ Good: Single query
$users = User::whereIn('id', $userIds)->get();
foreach ($users as $user) {
    // Process user
}

// ❌ Bad: Loading entire collection
if (Post::all()->count() > 100) {
    // ...
}

// ✅ Good: Use count query
if (Post::count() > 100) {
    // ...
}
✅ Optimization Principle: Measure first, optimize second. Use profiling tools to identify actual bottlenecks before optimizing.
