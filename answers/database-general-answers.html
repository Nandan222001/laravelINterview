1. Keys and Constraints

Primary Key

Definition: A primary key uniquely identifies each record in a table. It must contain unique values and cannot contain NULL values.

Characteristics:

  • Uniqueness: No two rows can have the same primary key value
  • Non-null: Primary key columns cannot contain NULL values
  • Single per table: Only one primary key per table
  • Immutable: Should not change after assignment
  • Minimal: Should use the fewest columns necessary
-- Single column primary key
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    email VARCHAR(100),
    name VARCHAR(100)
);

-- Composite primary key (multiple columns)
CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT,
    price DECIMAL(10,2),
    PRIMARY KEY (order_id, product_id)
);

Unique Key

Difference from Primary Key:

  • Can have multiple unique keys per table
  • Can contain NULL values (one NULL per column in most databases)
  • Not used as the default clustering key
-- Unique constraint examples
CREATE TABLE users (
    id INT PRIMARY KEY,
    email VARCHAR(100) UNIQUE,          -- Single unique constraint
    username VARCHAR(50) UNIQUE,
    ssn VARCHAR(11),
    UNIQUE (ssn)                        -- Alternative syntax
);

-- Composite unique constraint
CREATE TABLE products (
    id INT PRIMARY KEY,
    sku VARCHAR(50),
    warehouse_id INT,
    CONSTRAINT uk_sku_warehouse UNIQUE (sku, warehouse_id)
);

Foreign Key

Purpose: Enforces referential integrity between two tables by ensuring values in one table match values in another table's primary key or unique key.

Referential Actions:

Action ON DELETE ON UPDATE Use Case
CASCADE Delete child rows Update child rows Order items when order is deleted
SET NULL Set FK to NULL Set FK to NULL Optional relationships
RESTRICT Prevent deletion Prevent update Enforce data integrity
NO ACTION Similar to RESTRICT Similar to RESTRICT Default behavior
SET DEFAULT Set to default value Set to default value Rarely used in MySQL 
-- Foreign key with various actions
CREATE TABLE orders (
    id INT PRIMARY KEY,
    user_id INT,
    total DECIMAL(10,2),
    CONSTRAINT fk_orders_user FOREIGN KEY (user_id) 
        REFERENCES users(id)
        ON DELETE RESTRICT
        ON UPDATE CASCADE
);

CREATE TABLE comments (
    id INT PRIMARY KEY,
    post_id INT,
    user_id INT,
    content TEXT,
    FOREIGN KEY (post_id) REFERENCES posts(id)
        ON DELETE CASCADE,
    FOREIGN KEY (user_id) REFERENCES users(id)
        ON DELETE SET NULL
);

Composite Keys

When to use:

  • Natural composite uniqueness (e.g., student + course + semester)
  • Junction/bridge tables (many-to-many relationships)
  • Time-series data (entity + timestamp)

Advantages vs Disadvantages:

Advantages Disadvantages
Natural business meaning More complex joins
Enforces business rules Harder to reference in foreign keys
No surrogate key needed Potential performance impact
Prevents duplicate combinations Less portable code
💡 Best Practice: For composite keys, order columns by selectivity (most selective first) to optimize index performance. Consider surrogate keys for complex applications with many relationships.

2. Index Types

B-Tree Index (Default)

How it works: B-Tree (Balanced Tree) indexes organize data in a tree structure where each node can have multiple children. Data is sorted and balanced.

Characteristics:

  • Structure: Root → Branch nodes → Leaf nodes (containing actual data or pointers)
  • Performance: O(log n) for search, insert, update, delete
  • Use cases: Range queries, equality searches, ORDER BY, GROUP BY
  • Optimization: Supports leftmost prefix for composite indexes
B-Tree Index Structure: [50] / \ [20,35] [70,90] / | \ / | \ [10] [30] [40] [60] [80] [100] Leaf nodes contain: actual data (clustered) or pointers to data (non-clustered)
-- B-Tree index (default type)
CREATE INDEX idx_email ON users(email);

-- B-Tree supports these queries efficiently:
SELECT * FROM users WHERE email = 'john@example.com';      -- Equality
SELECT * FROM users WHERE email > 'a@example.com';         -- Range
SELECT * FROM users WHERE email LIKE 'john%';              -- Prefix match
SELECT * FROM users ORDER BY email;                        -- Sorting
SELECT email FROM users WHERE email BETWEEN 'a' AND 'z';   -- Range

-- Composite B-Tree index (leftmost prefix rule)
CREATE INDEX idx_name_email ON users(last_name, first_name, email);

-- Can use index for:
-- ✅ WHERE last_name = 'Smith'
-- ✅ WHERE last_name = 'Smith' AND first_name = 'John'
-- ✅ WHERE last_name = 'Smith' AND first_name = 'John' AND email = 'john@example.com'
-- ❌ WHERE first_name = 'John' (doesn't start with leftmost column)
-- ❌ WHERE email = 'john@example.com' (doesn't start with leftmost column)

Hash Index

How it works: Uses a hash function to compute the location of data. Provides constant-time O(1) lookups for equality comparisons.

Performance Analysis:

Operation B-Tree Hash
Equality (=) O(log n) O(1) ⭐
Range (<, >, BETWEEN) O(log n) ⭐ Not supported ❌
Prefix (LIKE 'abc%') O(log n) ⭐ Not supported ❌
ORDER BY Supported ⭐ Not supported ❌ 
-- Hash index (mainly for MEMORY tables)
CREATE TABLE cache (
    id INT PRIMARY KEY,
    key_name VARCHAR(100),
    value TEXT,
    INDEX USING HASH (key_name)
) ENGINE=MEMORY;

-- Hash index is optimal for:
SELECT * FROM cache WHERE key_name = 'user_123_session';  -- ✅ Equality

-- Hash index CANNOT help with:
SELECT * FROM cache WHERE key_name > 'user_100';          -- ❌ Range
SELECT * FROM cache WHERE key_name LIKE 'user_%';         -- ❌ Prefix
SELECT * FROM cache ORDER BY key_name;                    -- ❌ Sorting
⚠️ Note: InnoDB doesn't support explicit hash indexes. It uses adaptive hash indexing internally and automatically for frequently accessed data.

Full-Text Index

Purpose: Optimized for natural language text search. Supports word-based searching with relevance ranking.

Features:

  • Natural language search mode (default)
  • Boolean search mode (AND, OR, NOT operators)
  • Query expansion mode
  • Stopword filtering (common words like "the", "is", "and")
  • Minimum word length (default: 4 characters for InnoDB)
-- Create full-text index
CREATE TABLE articles (
    id INT PRIMARY KEY,
    title VARCHAR(200),
    content TEXT,
    FULLTEXT INDEX idx_fulltext_content (title, content)
);

-- Natural language search (default)
SELECT *, MATCH(title, content) AGAINST('database optimization') AS relevance
FROM articles
WHERE MATCH(title, content) AGAINST('database optimization')
ORDER BY relevance DESC;

-- Boolean mode search
SELECT * FROM articles
WHERE MATCH(title, content) AGAINST('+mysql -postgresql +optimization' IN BOOLEAN MODE);

-- Boolean operators:
-- + : Must contain this word
-- - : Must NOT contain this word
-- * : Wildcard (word*)
-- " : Exact phrase ("database optimization")
-- () : Grouping
-- > < : Increase/decrease relevance

-- Query expansion (finds related terms)
SELECT * FROM articles
WHERE MATCH(title, content) 
AGAINST('database' WITH QUERY EXPANSION);

Clustered Index

Definition: The clustered index determines the physical order of data in the table. The table data is stored in the leaf nodes of the clustered index.

Key Points:

  • One per table: Only one clustered index allowed (because data can only be physically sorted one way)
  • In InnoDB: The primary key IS the clustered index
  • Data storage: Table rows are stored in primary key order
  • Performance: Excellent for range queries on primary key
  • Sequential keys: AUTO_INCREMENT performs better than UUIDs
Clustered Index (InnoDB): Primary Key = Clustered Index Table data stored in PK order: [PK: 100] → [Data: John, john@example.com, ...] [PK: 101] → [Data: Jane, jane@example.com, ...] [PK: 102] → [Data: Bob, bob@example.com, ...] Range queries are fast: SELECT * FROM users WHERE id BETWEEN 100 AND 105; (Data is physically sequential)
-- InnoDB automatically creates clustered index on primary key
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,  -- This becomes the clustered index
    email VARCHAR(100),
    name VARCHAR(100)
) ENGINE=InnoDB;

-- If no primary key, InnoDB uses:
-- 1. First UNIQUE NOT NULL index
-- 2. Hidden 6-byte row ID (auto-generated)

-- ✅ GOOD: Sequential primary key
CREATE TABLE orders (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,  -- Sequential, good for clustered index
    user_id INT,
    order_date DATETIME
);

-- ⚠️ PROBLEMATIC: UUID as clustered index
CREATE TABLE sessions (
    session_id CHAR(36) PRIMARY KEY,  -- UUID: random, causes page splits
    user_id INT,
    created_at DATETIME
);
-- Better: Use AUTO_INCREMENT PK and UUID as unique key

Non-Clustered Index (Secondary Index)

Definition: A separate structure from the table data. Contains indexed columns plus a pointer to the actual row.

In InnoDB:

  • Leaf nodes contain: indexed column values + primary key value
  • To access full row: Index lookup → Primary key lookup → Data row
  • Multiple non-clustered indexes allowed per table
Non-Clustered Index Structure (InnoDB): Secondary Index on email: [email: alice@ex.com] → [PK: 105] [email: bob@ex.com] → [PK: 102] [email: carol@ex.com] → [PK: 130] ↓ [Use PK to find data in clustered index] Two lookups required unless it's a covering index!
-- Non-clustered (secondary) indexes
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,     -- Clustered index
    email VARCHAR(100),
    name VARCHAR(100),
    status VARCHAR(20),
    created_at DATETIME,
    INDEX idx_email (email),               -- Non-clustered index
    INDEX idx_status (status),             -- Non-clustered index
    INDEX idx_created (created_at)         -- Non-clustered index
);

-- Regular query (two lookups):
SELECT * FROM users WHERE email = 'john@example.com';
-- 1. Search idx_email → find PK value
-- 2. Search clustered index using PK → get full row

-- Covering index (one lookup only):
CREATE INDEX idx_covering ON users(status, email, name);

SELECT status, email, name FROM users WHERE status = 'active';
-- All needed columns in index → no table lookup needed!

Performance Comparison:

Aspect Clustered Index Non-Clustered Index
Lookups needed 1 (data is in index) 2 (index + table lookup)
Range queries Very fast (sequential) Slower (random access)
Insert performance Can cause page splits Faster inserts
Number per table 1 Multiple (16-64 typical limit)
Storage Contains actual data Contains indexed cols + PK
✅ Index Selection Strategy:
  • B-Tree: Default choice for most queries (equality, range, sorting)
  • Hash: Only for exact equality lookups on MEMORY tables
  • Full-Text: For natural language text search
  • Clustered: Automatic on primary key in InnoDB
  • Non-Clustered: For additional access paths to data

3. Join Types with Visual Examples

INNER JOIN

Returns: Only rows that have matching values in both tables.

Table A (users) Table B (orders) id | name user_id | total 1 | Alice 1 | 100 2 | Bob 1 | 200 3 | Carol 2 | 150 4 | David NULL | 50 INNER JOIN Result: id | name | total 1 | Alice | 100 1 | Alice | 200 2 | Bob | 150 Only rows with matching user_id in both tables
-- Basic INNER JOIN
SELECT u.id, u.name, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- Multiple INNER JOINs
SELECT 
    u.name AS customer,
    o.id AS order_id,
    p.name AS product,
    oi.quantity,
    oi.price
FROM users u
INNER JOIN orders o ON u.id = o.user_id
INNER JOIN order_items oi ON o.id = oi.order_id
INNER JOIN products p ON oi.product_id = p.id
WHERE o.status = 'completed';

LEFT JOIN (LEFT OUTER JOIN)

Returns: All rows from the left table, and matching rows from the right table. NULL for non-matching rows from the right table.

Table A (users) Table B (orders) id | name user_id | total 1 | Alice 1 | 100 2 | Bob 1 | 200 3 | Carol 2 | 150 4 | David NULL | 50 LEFT JOIN Result: id | name | total 1 | Alice | 100 1 | Alice | 200 2 | Bob | 150 3 | Carol | NULL ← No orders, but Carol is included 4 | David | NULL ← No orders, but David is included
-- LEFT JOIN - all users with their orders (if any)
SELECT u.id, u.name, COUNT(o.id) AS order_count, COALESCE(SUM(o.total), 0) AS total_spent
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name;

-- Find users with NO orders
SELECT u.id, u.name
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;

RIGHT JOIN (RIGHT OUTER JOIN)

Returns: All rows from the right table, and matching rows from the left table. NULL for non-matching rows from the left table.

Table A (users) Table B (orders) id | name user_id | total 1 | Alice 1 | 100 2 | Bob 2 | 150 3 | Carol 5 | 300 ← user_id 5 doesn't exist RIGHT JOIN Result: id | name | total 1 | Alice | 100 2 | Bob | 150 NULL | NULL | 300 ← Order exists but no matching user
-- RIGHT JOIN - all orders with user info (if available)
SELECT u.name, o.id AS order_id, o.total
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id;

-- Find orphaned orders (orders without valid users)
SELECT o.id, o.total
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id
WHERE u.id IS NULL;

FULL OUTER JOIN

Returns: All rows from both tables. NULL where there's no match.

MySQL doesn't support FULL OUTER JOIN directly. Simulate with UNION of LEFT JOIN and RIGHT JOIN: Table A (users) Table B (orders) 1 | Alice 1 | 100 2 | Bob 5 | 300 3 | Carol FULL OUTER JOIN Result: id | name | total 1 | Alice | 100 2 | Bob | NULL 3 | Carol | NULL NULL | NULL | 300
-- Simulate FULL OUTER JOIN in MySQL
SELECT u.id, u.name, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id

UNION

SELECT u.id, u.name, o.total
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id
WHERE u.id IS NULL;

CROSS JOIN

Returns: Cartesian product - every row from the first table combined with every row from the second table.

Table A (colors) Table B (sizes) color size Red Small Blue Large CROSS JOIN Result (2 × 2 = 4 rows): color | size Red | Small Red | Large Blue | Small Blue | Large
-- CROSS JOIN
SELECT c.color, s.size
FROM colors c
CROSS JOIN sizes s;

-- Alternative syntax (implicit cross join)
SELECT c.color, s.size
FROM colors c, sizes s;

-- Practical use case: Generate all combinations
SELECT 
    dates.date,
    products.name
FROM 
    (SELECT DATE_ADD('2024-01-01', INTERVAL n DAY) AS date 
     FROM numbers WHERE n < 365) dates
CROSS JOIN products;

SELF JOIN

Purpose: Join a table with itself to compare rows within the same table.

Table: employees id | name | manager_id 1 | Alice | NULL (CEO) 2 | Bob | 1 (reports to Alice) 3 | Carol | 1 (reports to Alice) 4 | David | 2 (reports to Bob) SELF JOIN Result (employee and their manager): employee | manager Bob | Alice Carol | Alice David | Bob
-- SELF JOIN - employees and their managers
SELECT 
    e.name AS employee,
    m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

-- Find employees in the same department
SELECT 
    e1.name AS employee1,
    e2.name AS employee2,
    e1.department
FROM employees e1
INNER JOIN employees e2 ON e1.department = e2.department
WHERE e1.id < e2.id;  -- Prevent duplicate pairs

-- Find employees earning more than their manager
SELECT 
    e.name AS employee,
    e.salary AS emp_salary,
    m.name AS manager,
    m.salary AS mgr_salary
FROM employees e
INNER JOIN employees m ON e.manager_id = m.id
WHERE e.salary > m.salary;

Join Performance Tips:

✅ Optimization Strategies:
  • Index foreign keys: Always index columns used in JOIN conditions
  • Filter early: Apply WHERE conditions before joining when possible
  • Join order: Join smaller result sets first
  • Use INNER JOIN when possible: Faster than OUTER joins
  • Avoid SELECT *: Only select needed columns
  • Covering indexes: Include all joined and selected columns in index
  • Use EXPLAIN: Analyze and verify join strategy

4. Triggers - Use Cases and Examples

Definition: A trigger is a stored procedure that automatically executes when a specific event (INSERT, UPDATE, DELETE) occurs on a table.

Common Use Cases

1. Automatic Timestamp Management

-- Auto-update updated_at timestamp
DELIMITER //
CREATE TRIGGER before_user_update
BEFORE UPDATE ON users
FOR EACH ROW
BEGIN
    SET NEW.updated_at = NOW();
END;//
DELIMITER ;

-- Set created_at automatically
DELIMITER //
CREATE TRIGGER before_user_insert
BEFORE INSERT ON users
FOR EACH ROW
BEGIN
    IF NEW.created_at IS NULL THEN
        SET NEW.created_at = NOW();
    END IF;
    IF NEW.updated_at IS NULL THEN
        SET NEW.updated_at = NOW();
    END IF;
END;//
DELIMITER ;

2. Data Validation and Business Rules

-- Validate email format
DELIMITER //
CREATE TRIGGER before_user_insert_validate
BEFORE INSERT ON users
FOR EACH ROW
BEGIN
    -- Ensure email is lowercase
    SET NEW.email = LOWER(NEW.email);
    
    -- Validate email contains @
    IF NEW.email NOT LIKE '%@%' THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Invalid email format';
    END IF;
    
    -- Validate age
    IF NEW.age IS NOT NULL AND NEW.age < 0 THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Age cannot be negative';
    END IF;
END;//
DELIMITER ;

-- Prevent price from being set to zero or negative
DELIMITER //
CREATE TRIGGER before_product_update
BEFORE UPDATE ON products
FOR EACH ROW
BEGIN
    IF NEW.price <= 0 THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Price must be greater than zero';
    END IF;
END;//
DELIMITER ;

3. Audit Trail and Change Tracking

-- Audit log table
CREATE TABLE user_audit_log (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT,
    action VARCHAR(10),
    old_email VARCHAR(100),
    new_email VARCHAR(100),
    old_name VARCHAR(100),
    new_name VARCHAR(100),
    changed_by VARCHAR(100),
    changed_at DATETIME
);

-- Track all changes to users table
DELIMITER //
CREATE TRIGGER after_user_update_audit
AFTER UPDATE ON users
FOR EACH ROW
BEGIN
    INSERT INTO user_audit_log (
        user_id, action, 
        old_email, new_email,
        old_name, new_name,
        changed_by, changed_at
    ) VALUES (
        NEW.id, 'UPDATE',
        OLD.email, NEW.email,
        OLD.name, NEW.name,
        USER(), NOW()
    );
END;//
DELIMITER ;

-- Track deletions
DELIMITER //
CREATE TRIGGER after_user_delete_audit
AFTER DELETE ON users
FOR EACH ROW
BEGIN
    INSERT INTO user_audit_log (
        user_id, action, 
        old_email, old_name,
        changed_by, changed_at
    ) VALUES (
        OLD.id, 'DELETE',
        OLD.email, OLD.name,
        USER(), NOW()
    );
END;//
DELIMITER ;

4. Maintaining Denormalized Data / Statistics

-- Update user statistics when order is placed
DELIMITER //
CREATE TRIGGER after_order_insert
AFTER INSERT ON orders
FOR EACH ROW
BEGIN
    UPDATE user_statistics
    SET 
        total_orders = total_orders + 1,
        total_spent = total_spent + NEW.total,
        last_order_date = NEW.order_date
    WHERE user_id = NEW.user_id;
    
    -- Insert if doesn't exist
    INSERT IGNORE INTO user_statistics (
        user_id, total_orders, total_spent, last_order_date
    ) VALUES (
        NEW.user_id, 1, NEW.total, NEW.order_date
    );
END;//
DELIMITER ;

-- Update product inventory on order
DELIMITER //
CREATE TRIGGER after_order_item_insert
AFTER INSERT ON order_items
FOR EACH ROW
BEGIN
    UPDATE products
    SET stock_quantity = stock_quantity - NEW.quantity
    WHERE id = NEW.product_id;
    
    -- Alert if low stock
    IF (SELECT stock_quantity FROM products WHERE id = NEW.product_id) < 10 THEN
        INSERT INTO alerts (message, created_at)
        VALUES (CONCAT('Low stock for product_id: ', NEW.product_id), NOW());
    END IF;
END;//
DELIMITER ;

5. Enforcing Complex Constraints

-- Prevent deleting users with active orders
DELIMITER //
CREATE TRIGGER before_user_delete
BEFORE DELETE ON users
FOR EACH ROW
BEGIN
    DECLARE active_orders INT;
    
    SELECT COUNT(*) INTO active_orders
    FROM orders
    WHERE user_id = OLD.id AND status = 'active';
    
    IF active_orders > 0 THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Cannot delete user with active orders';
    END IF;
END;//
DELIMITER ;

-- Prevent modifying certain critical fields
DELIMITER //
CREATE TRIGGER before_order_update
BEFORE UPDATE ON orders
FOR EACH ROW
BEGIN
    -- Prevent changing order_id
    IF OLD.id != NEW.id THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Order ID cannot be changed';
    END IF;
    
    -- Prevent changing total after order is completed
    IF OLD.status = 'completed' AND OLD.total != NEW.total THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Cannot modify completed order total';
    END IF;
END;//
DELIMITER ;

6. Cascade Operations (Alternative to FK CASCADE)

-- Custom cascade delete logic
DELIMITER //
CREATE TRIGGER after_user_delete_cascade
AFTER DELETE ON users
FOR EACH ROW
BEGIN
    -- Archive orders before deleting
    INSERT INTO archived_orders 
    SELECT *, NOW() AS archived_at 
    FROM orders 
    WHERE user_id = OLD.id;
    
    -- Delete related records
    DELETE FROM orders WHERE user_id = OLD.id;
    DELETE FROM user_preferences WHERE user_id = OLD.id;
    DELETE FROM user_sessions WHERE user_id = OLD.id;
END;//
DELIMITER ;

Trigger Management

-- Show all triggers
SHOW TRIGGERS;

-- Show triggers for specific table
SHOW TRIGGERS WHERE `Table` = 'users';

-- Show trigger definition
SHOW CREATE TRIGGER trigger_name;

-- Drop trigger
DROP TRIGGER IF EXISTS trigger_name;

Trigger Performance Considerations

⚠️ Performance Impact:
  • Automatic execution: Triggers fire on every row operation, increasing processing time
  • Hidden logic: Difficult to debug and trace performance issues
  • Lock escalation: Triggers can hold locks longer, causing contention
  • Cascade effects: One trigger can activate another, creating chains
  • Bulk operations: Triggers fire for each row in bulk inserts/updates
💡 Best Practices:
  • Keep trigger logic simple and fast
  • Avoid complex queries or joins in triggers
  • Don't call triggers from triggers (avoid recursion)
  • Use triggers for data integrity, not business logic
  • Document all triggers thoroughly
  • Consider application-level logic for complex operations

5. Views - Benefits and Use Cases

Definition: A view is a virtual table based on a SQL query. It doesn't store data itself but presents data from one or more tables.

Key Benefits

1. Simplification - Hide Complex Queries

-- Complex query that joins multiple tables
CREATE VIEW order_summary AS
SELECT 
    o.id AS order_id,
    o.order_date,
    u.name AS customer_name,
    u.email AS customer_email,
    COUNT(oi.id) AS total_items,
    SUM(oi.quantity) AS total_quantity,
    SUM(oi.quantity * oi.price) AS total_amount,
    o.status
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN order_items oi ON o.id = oi.order_id
GROUP BY o.id, o.order_date, u.name, u.email, o.status;

-- Now users can simply query:
SELECT * FROM order_summary WHERE status = 'completed';

-- Instead of writing the complex join every time

2. Security - Restrict Access to Sensitive Data

-- View exposes only safe columns, hiding sensitive data
CREATE VIEW public_user_info AS
SELECT 
    id,
    username,
    first_name,
    last_name,
    avatar_url,
    created_at
FROM users;
-- Columns NOT exposed: password_hash, email, ssn, credit_card, etc.

-- Grant access to view, not the underlying table
GRANT SELECT ON database.public_user_info TO 'app_user'@'localhost';

-- Department-specific view
CREATE VIEW sales_department_orders AS
SELECT 
    o.id,
    o.order_date,
    o.total,
    o.status
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE u.department = 'Sales';

GRANT SELECT ON database.sales_department_orders TO 'sales_team'@'localhost';

3. Data Abstraction - Logical Data Independence

-- Original table structure
CREATE TABLE customers (
    id INT PRIMARY KEY,
    full_name VARCHAR(200),
    contact_email VARCHAR(100)
);

-- Create view with different column names for compatibility
CREATE VIEW users AS
SELECT 
    id,
    full_name AS name,
    contact_email AS email
FROM customers;

-- Applications using "users" view don't break when table changes
-- You can later refactor the underlying table without affecting applications

4. Consistent Calculated Fields

-- View with calculated fields
CREATE VIEW product_details AS
SELECT 
    p.id,
    p.name,
    p.price,
    p.cost,
    (p.price - p.cost) AS profit,
    ROUND(((p.price - p.cost) / p.cost) * 100, 2) AS profit_margin_pct,
    p.stock_quantity,
    (p.stock_quantity * p.cost) AS inventory_value,
    CASE 
        WHEN p.stock_quantity = 0 THEN 'Out of Stock'
        WHEN p.stock_quantity < 10 THEN 'Low Stock'
        ELSE 'In Stock'
    END AS stock_status
FROM products p;

-- Everyone gets consistent calculations
SELECT * FROM product_details WHERE stock_status = 'Low Stock';

5. Data Formatting and Presentation

-- Format dates and combine fields
CREATE VIEW customer_list AS
SELECT 
    id,
    CONCAT(first_name, ' ', last_name) AS full_name,
    email,
    phone,
    CONCAT(address, ', ', city, ', ', state, ' ', zip_code) AS full_address,
    DATE_FORMAT(created_at, '%M %d, %Y') AS member_since,
    DATEDIFF(NOW(), created_at) AS days_active,
    CASE 
        WHEN status = 1 THEN 'Active'
        WHEN status = 0 THEN 'Inactive'
        ELSE 'Suspended'
    END AS account_status
FROM customers;

6. Aggregate and Summary Views

-- Daily sales summary
CREATE VIEW daily_sales_summary AS
SELECT 
    DATE(order_date) AS sale_date,
    COUNT(DISTINCT id) AS total_orders,
    COUNT(DISTINCT user_id) AS unique_customers,
    SUM(total) AS total_revenue,
    AVG(total) AS average_order_value,
    MIN(total) AS min_order,
    MAX(total) AS max_order
FROM orders
WHERE status = 'completed'
GROUP BY DATE(order_date);

-- User activity summary
CREATE VIEW user_activity_summary AS
SELECT 
    u.id,
    u.name,
    u.email,
    COUNT(o.id) AS total_orders,
    COALESCE(SUM(o.total), 0) AS lifetime_value,
    MAX(o.order_date) AS last_order_date,
    DATEDIFF(NOW(), MAX(o.order_date)) AS days_since_last_order,
    CASE 
        WHEN MAX(o.order_date) > DATE_SUB(NOW(), INTERVAL 30 DAY) THEN 'Active'
        WHEN MAX(o.order_date) > DATE_SUB(NOW(), INTERVAL 90 DAY) THEN 'At Risk'
        ELSE 'Churned'
    END AS customer_status
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name, u.email;

Updatable Views

Conditions for updatable views:

  • No aggregate functions (SUM, AVG, COUNT, etc.)
  • No DISTINCT clause
  • No GROUP BY or HAVING
  • No UNION
  • No subqueries in SELECT list
  • Only one table in FROM clause (simple views)
-- Updatable view (simple, single table)
CREATE VIEW active_users AS
SELECT id, name, email, status
FROM users
WHERE status = 'active';

-- Can UPDATE through view
UPDATE active_users SET name = 'John Smith' WHERE id = 1;

-- Can INSERT through view
INSERT INTO active_users (name, email, status) 
VALUES ('Jane Doe', 'jane@example.com', 'active');

-- WITH CHECK OPTION prevents updates that violate WHERE clause
CREATE VIEW active_users_checked AS
SELECT id, name, email, status
FROM users
WHERE status = 'active'
WITH CHECK OPTION;

-- This will fail (violates WHERE status = 'active')
UPDATE active_users_checked SET status = 'inactive' WHERE id = 1;

View Management

-- Show all views
SHOW FULL TABLES WHERE Table_type = 'VIEW';

-- Show view definition
SHOW CREATE VIEW view_name;

-- Modify existing view
CREATE OR REPLACE VIEW order_summary AS
SELECT 
    o.id,
    o.order_date,
    u.name,
    o.total,
    o.status
FROM orders o
JOIN users u ON o.user_id = u.id;

-- Drop view
DROP VIEW IF EXISTS view_name;

-- Drop multiple views
DROP VIEW IF EXISTS view1, view2, view3;

Performance Considerations

⚠️ Important Notes:
  • Not materialized: Views execute the underlying query every time (no caching)
  • Performance: Same as running the underlying query directly
  • Indexes: Cannot create indexes on views (use indexes on base tables)
  • Nested views: Views referencing views can impact performance
  • Complex views: Multiple joins/aggregations can be slow
💡 Best Practices:
  • Use views for simplification and security, not performance
  • Index the underlying tables appropriately
  • Avoid deep nesting of views (view → view → view)
  • Consider materialized views for expensive queries (if supported)
  • Document view purposes and dependencies
  • Use naming conventions (e.g., vw_order_summary)

6. Database Normalization (1NF to 3NF)

Purpose: Normalization is the process of organizing database tables to reduce redundancy and improve data integrity.

First Normal Form (1NF)

Rules:

  • Each column contains atomic (indivisible) values
  • Each column contains values of a single type
  • Each column has a unique name
  • The order of rows/columns doesn't matter

❌ Violates 1NF:

-- BAD: Multiple values in single column
CREATE TABLE students_bad (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    phone_numbers VARCHAR(200),  -- '555-1234, 555-5678, 555-9012'
    courses VARCHAR(500)          -- 'Math, Science, History, English'
);

-- Example data:
id | name  | phone_numbers           | courses
1  | John  | 555-1234, 555-5678     | Math, Science
2  | Jane  | 555-9999               | History, English, Art

✅ Follows 1NF:

-- GOOD: Atomic values only
CREATE TABLE students (
    id INT PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE student_phones (
    id INT AUTO_INCREMENT PRIMARY KEY,
    student_id INT,
    phone_number VARCHAR(20),
    phone_type ENUM('mobile', 'home', 'work'),
    FOREIGN KEY (student_id) REFERENCES students(id)
);

CREATE TABLE courses (
    id INT PRIMARY KEY,
    course_name VARCHAR(100)
);

CREATE TABLE student_courses (
    student_id INT,
    course_id INT,
    enrollment_date DATE,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES students(id),
    FOREIGN KEY (course_id) REFERENCES courses(id)
);

Second Normal Form (2NF)

Rules:

  • Must be in 1NF
  • All non-key columns must depend on the entire primary key (no partial dependencies)
  • Only applies to tables with composite primary keys

❌ Violates 2NF (Partial Dependency):

-- BAD: Partial dependency on composite key
CREATE TABLE order_items_bad (
    order_id INT,
    product_id INT,
    product_name VARCHAR(100),      -- Depends ONLY on product_id
    product_category VARCHAR(50),   -- Depends ONLY on product_id
    product_price DECIMAL(10,2),    -- Depends ONLY on product_id
    quantity INT,                   -- Depends on BOTH order_id AND product_id
    PRIMARY KEY (order_id, product_id)
);

-- Problem: product_name, product_category, product_price depend only on 
-- part of the primary key (product_id), not the full key (order_id, product_id)

Example data:
order_id | product_id | product_name | product_category | product_price | quantity
1        | 101        | Laptop       | Electronics      | 999.99        | 2
1        | 102        | Mouse        | Electronics      | 29.99         | 1
2        | 101        | Laptop       | Electronics      | 999.99        | 1
                       ↑ REDUNDANT DATA (repeated for each order)

✅ Follows 2NF (No Partial Dependencies):

-- GOOD: Separate product information
CREATE TABLE products (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    category VARCHAR(50),
    price DECIMAL(10,2)
);

CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT,
    price_at_purchase DECIMAL(10,2),  -- Snapshot of price when ordered
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);

-- Now all non-key columns in order_items depend on the FULL composite key

Third Normal Form (3NF)

Rules:

  • Must be in 2NF
  • No transitive dependencies (non-key columns must not depend on other non-key columns)
  • All non-key columns depend directly on the primary key

❌ Violates 3NF (Transitive Dependency):

-- BAD: Transitive dependency
CREATE TABLE employees_bad (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    department_id INT,
    department_name VARCHAR(100),    -- Depends on department_id (transitive)
    department_location VARCHAR(100), -- Depends on department_id (transitive)
    department_manager VARCHAR(100)   -- Depends on department_id (transitive)
);

-- Problem: department_name, department_location, department_manager 
-- depend on department_id, which is NOT the primary key
-- This is a transitive dependency: id → department_id → department_name

Example data:
id | name  | dept_id | dept_name | dept_location | dept_manager
1  | Alice | 10      | Sales     | New York      | Bob Smith
2  | Carol | 10      | Sales     | New York      | Bob Smith
3  | David | 20      | IT        | Austin        | Jane Doe
        ↑ REDUNDANT: Department info repeated for each employee

✅ Follows 3NF (No Transitive Dependencies):

-- GOOD: Remove transitive dependencies
CREATE TABLE departments (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    location VARCHAR(100),
    manager_name VARCHAR(100)
);

CREATE TABLE employees (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    department_id INT,
    FOREIGN KEY (department_id) REFERENCES departments(id)
);

-- Now all non-key columns depend DIRECTLY on the primary key only
-- No transitive dependencies

Complete Normalization Example

-- ❌ UNNORMALIZED TABLE
CREATE TABLE orders_unnormalized (
    order_id INT PRIMARY KEY,
    customer_name VARCHAR(100),
    customer_email VARCHAR(100),
    customer_phone VARCHAR(20),
    customer_address VARCHAR(200),
    customer_city VARCHAR(50),
    customer_state VARCHAR(2),
    customer_zip VARCHAR(10),
    product_names VARCHAR(500),      -- 'Laptop, Mouse, Keyboard'
    product_prices VARCHAR(200),     -- '999.99, 29.99, 79.99'
    product_quantities VARCHAR(100)  -- '1, 2, 1'
);

-- ✅ NORMALIZED TO 3NF
-- Customers table (1NF, 2NF, 3NF)
CREATE TABLE customers (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100) UNIQUE,
    phone VARCHAR(20),
    address_line VARCHAR(200),
    city VARCHAR(50),
    state VARCHAR(2),
    zip_code VARCHAR(10)
);

-- Products table (1NF, 2NF, 3NF)
CREATE TABLE products (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    price DECIMAL(10,2),
    category VARCHAR(50)
);

-- Orders table (1NF, 2NF, 3NF)
CREATE TABLE orders (
    id INT PRIMARY KEY,
    customer_id INT,
    order_date DATETIME,
    status VARCHAR(20),
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);

-- Order items table (1NF, 2NF, 3NF)
CREATE TABLE order_items (
    order_id INT,
    product_id INT,
    quantity INT,
    price_at_order DECIMAL(10,2),
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);

Benefits of Normalization

✅ Advantages:
  • Eliminates redundancy: Data stored once, reducing storage
  • Improves data integrity: Updates in one place affect all references
  • Easier maintenance: Changes to data structure are localized
  • Prevents anomalies: Avoids insert, update, delete anomalies
  • Better organization: Logical separation of concerns

When to Denormalize

⚠️ Strategic Denormalization:
  • Read-heavy applications: Reduce joins for frequently accessed data
  • Performance critical queries: Materialized computed columns
  • Reporting/Analytics: Denormalized data warehouses (OLAP)
  • Historical snapshots: Store point-in-time data (e.g., price_at_order)

Trade-off: Better read performance vs data redundancy and maintenance complexity

7. Query Optimization Strategies

1. Index Optimization

Use Appropriate Indexes

-- ❌ BAD: No index on WHERE clause
SELECT * FROM users WHERE email = 'john@example.com';
-- Full table scan required

-- ✅ GOOD: Index on frequently queried column
CREATE INDEX idx_email ON users(email);
SELECT * FROM users WHERE email = 'john@example.com';
-- Index seek - much faster

-- ✅ COMPOSITE INDEX for multiple columns
CREATE INDEX idx_status_date ON orders(status, order_date);
SELECT * FROM orders 
WHERE status = 'completed' 
  AND order_date >= '2024-01-01';

Covering Indexes

-- ❌ REQUIRES TABLE LOOKUP
CREATE INDEX idx_user_id ON orders(user_id);
SELECT user_id, order_date, total FROM orders WHERE user_id = 123;
-- Index seek + table lookup to get order_date and total

-- ✅ COVERING INDEX (includes all needed columns)
CREATE INDEX idx_covering ON orders(user_id, order_date, total);
SELECT user_id, order_date, total FROM orders WHERE user_id = 123;
-- Index-only scan - no table lookup needed!

2. Query Writing Best Practices

Avoid SELECT *

-- ❌ BAD: Fetches all columns (even unused ones)
SELECT * FROM users WHERE id = 1;

-- ✅ GOOD: Only fetch needed columns
SELECT id, name, email FROM users WHERE id = 1;

-- Benefits:
-- 1. Less data transfer
-- 2. Can use covering indexes
-- 3. Better cache utilization

Optimize WHERE Clauses

-- ❌ BAD: Function on indexed column prevents index usage
SELECT * FROM users WHERE YEAR(created_at) = 2024;

-- ✅ GOOD: Rewrite to use index
SELECT * FROM users 
WHERE created_at >= '2024-01-01' 
  AND created_at < '2025-01-01';

-- ❌ BAD: Leading wildcard prevents index usage
SELECT * FROM users WHERE email LIKE '%@gmail.com';

-- ✅ GOOD: Prefix matching uses index
SELECT * FROM users WHERE email LIKE 'john%';

-- ❌ BAD: Implicit conversion
SELECT * FROM users WHERE id = '123';  -- id is INT, '123' is string

-- ✅ GOOD: Correct type
SELECT * FROM users WHERE id = 123;

Use EXISTS Instead of IN for Subqueries

-- ❌ SLOWER: IN with subquery
SELECT * FROM users
WHERE id IN (SELECT user_id FROM orders WHERE total > 1000);

-- ✅ FASTER: EXISTS (stops at first match)
SELECT * FROM users u
WHERE EXISTS (
    SELECT 1 FROM orders o 
    WHERE o.user_id = u.id AND o.total > 1000
);

-- EXISTS advantages:
-- 1. Stops searching after first match
-- 2. Better for correlated subqueries
-- 3. Can use indexes more efficiently

3. Join Optimization

Index Foreign Keys

-- Always index columns used in JOINs
CREATE INDEX idx_user_id ON orders(user_id);
CREATE INDEX idx_order_id ON order_items(order_id);
CREATE INDEX idx_product_id ON order_items(product_id);

-- Now joins are fast
SELECT u.name, o.total, p.name
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id;

Filter Early

-- ❌ BAD: Filter after joining (joins more rows)
SELECT u.name, o.total
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.status = 'completed' AND o.order_date >= '2024-01-01';

-- ✅ BETTER: Filter in WHERE or use subquery
SELECT u.name, o.total
FROM users u
JOIN (
    SELECT user_id, total 
    FROM orders 
    WHERE status = 'completed' AND order_date >= '2024-01-01'
) o ON u.id = o.user_id;

4. Pagination Optimization

Avoid Large OFFSET

-- ❌ BAD: Large offset (still scans all skipped rows)
SELECT * FROM orders 
ORDER BY id 
LIMIT 20 OFFSET 100000;  -- Scans 100,000 rows then returns 20

-- ✅ GOOD: Keyset pagination (cursor-based)
SELECT * FROM orders 
WHERE id > 100000  -- Use last seen ID
ORDER BY id 
LIMIT 20;

-- ✅ ALSO GOOD: Deferred join
SELECT o.* FROM orders o
JOIN (
    SELECT id FROM orders ORDER BY id LIMIT 20 OFFSET 100000
) AS t ON o.id = t.id;

5. Aggregation Optimization

Optimize COUNT Queries

-- ❌ SLOW: COUNT(*) on large tables
SELECT COUNT(*) FROM orders;

-- ✅ FASTER: Use approximate counts for large tables
SELECT table_rows FROM information_schema.tables
WHERE table_name = 'orders' AND table_schema = DATABASE();

-- ✅ MAINTAIN COUNTER TABLES
CREATE TABLE order_counts (
    count BIGINT,
    updated_at DATETIME
);
-- Update via triggers or scheduled jobs

-- ❌ SLOW: COUNT with GROUP BY on unindexed column
SELECT status, COUNT(*) FROM orders GROUP BY status;

-- ✅ FASTER: Index the GROUP BY column
CREATE INDEX idx_status ON orders(status);
SELECT status, COUNT(*) FROM orders GROUP BY status;

6. Subquery Optimization

-- ❌ BAD: Correlated subquery (executes for each row)
SELECT 
    u.name,
    (SELECT COUNT(*) FROM orders WHERE user_id = u.id) AS order_count
FROM users u;

-- ✅ BETTER: JOIN with GROUP BY
SELECT 
    u.name,
    COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name;

7. Use EXPLAIN to Analyze Queries

-- Analyze query execution plan
EXPLAIN SELECT u.name, o.total
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.status = 'completed';

-- Key things to look for in EXPLAIN:
-- ✅ type: const, eq_ref, ref (good)
-- ❌ type: ALL (full table scan - bad)
-- ✅ key: shows index name (good)
-- ❌ key: NULL (no index used - bad)
-- ✅ rows: low number (good)
-- ❌ rows: high number (bad)
-- ❌ Extra: Using filesort, Using temporary (potential issues)
-- ✅ Extra: Using index (good - covering index)

Query Optimization Checklist

✅ Optimization Checklist:
  1. Index Strategy:
    • Index WHERE clause columns
    • Index JOIN columns (foreign keys)
    • Index ORDER BY columns
    • Create covering indexes for frequently run queries
    • Use composite indexes (order: Equality → Range → Sort)
  2. Query Writing:
    • Select only needed columns (avoid SELECT *)
    • Avoid functions on indexed columns
    • Use correct data types (avoid implicit conversion)
    • Use EXISTS instead of IN for subqueries
    • Use UNION ALL instead of UNION when possible
  3. Join Optimization:
    • Always index foreign keys
    • Filter before joining when possible
    • Use INNER JOIN when possible (faster than OUTER)
    • Consider join order (smaller tables first)
  4. Analysis:
    • Use EXPLAIN to analyze queries
    • Monitor slow query log
    • Run ANALYZE TABLE to update statistics
    • Benchmark before and after optimization
💡 Pro Tips:
  • Optimize for common queries first (80/20 rule)
  • Test with production-like data volumes
  • Monitor actual query performance in production
  • Balance read vs write performance based on workload
  • Consider caching for frequently accessed data
